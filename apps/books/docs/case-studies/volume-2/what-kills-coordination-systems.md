# Chapter 9: What Kills Coordination Systems

## Overview

Coordination network failures follow a predictable pattern. Analysis of failed networks identifies five primary failure modes: founder dependency, free riding, scope creep, imbalanced exchange, and premature formalization. Each failure mode is preventable through specific structural interventions. This chapter catalogs each failure mode, its mechanism, its warning signs, and its fix. It also addresses conflict, which is commonly misidentified as a failure mode but is not.

---

## 9.1 Failure Mode 1: Founder Dependency

**Prevalence:** Most common cause of network death.

**Mechanism:** A single person performs all coordination functions: board management, member recruitment, dispute resolution, exchange tracking. The network becomes operationally dependent on one individual. When that individual departs (job change, burnout, relocation, life event), the network collapses because no system exists to continue operations without them.

**Warning signs:**
- One person manages the board, recruits members, resolves disputes, and tracks exchanges
- Other members describe the network using the founder's name ("that's Sarah's thing")
- No written documentation of processes; everything lives in one person's memory

**Structural fixes:**

| Fix | Implementation |
|---|---|
| Distribute coordination | Split coordination across minimum 3 people from Day 1: board manager, recruiter, exchange tracker |
| Rotate roles | Rotate each coordination role every 3 months |
| Document all processes | Skills inventory, exchange rules, points ledger all stored in shared document (not in anyone's head) |

**Test:** If the founder disappeared tomorrow and a new person read the shared documentation, could they keep the network running? If not, the documentation is insufficient.

## 9.2 Failure Mode 2: Free Riding

**Prevalence:** Occurs in every network exceeding 8 members.

**Mechanism:** One or more members consistently consume network labor without contributing equivalent hours. In a 10-person network, one non-contributing member creates a 10% drag that the remaining 9 members absorb. The highest-contributing members leave first because they have the most external options. The network hollows out.

**Warning signs:**
- One member always needs help "this month" but will contribute "next month"
- Contributing members reduce their own output due to resentment
- The most skilled members disengage or leave

**Structural fixes:**

| Fix | Implementation |
|---|---|
| Visible tracking | Track all exchanges: who gave, who received, hours contributed, hours consumed. Free riding becomes mathematically visible. |
| Contribution floor | Set a minimum contribution: 5 hours/month. Two consecutive months below floor triggers a check-in conversation (not a penalty). |
| Graceful exits | Allow members to leave without guilt. A 10-member network with 10 engaged members outperforms a 15-member network with 5 non-contributors. |

## 9.3 Failure Mode 3: Scope Creep

**Prevalence:** Common in networks that survive past 6 months.

**Mechanism:** The network begins as a skill exchange (task board). Over time, members propose additional functions: collective purchasing, community gardens, shared childcare, mutual health insurance, vision-and-values discussions. Each addition increases complexity, governance requirements, and potential for disagreement. Energy shifts from the core function (connecting needs to skills) to managing side initiatives. The task board goes quiet while committee meetings multiply.

**Warning signs:**
- Board activity declining while meeting frequency increasing
- Members debating bylaws for side projects
- Multiple parallel initiatives competing for the same members' time
- Someone proposes a "planning session" about the network's "future direction"

**Structural fixes:**

| Fix | Implementation |
|---|---|
| Protect the core | The task board is the network. Health metric: DONE column growing. If the board goes quiet, the network is dying regardless of how many committees are meeting. |
| Spin off, don't absorb | New initiatives (purchasing group, community garden, childcare rotation) get their own structure, leadership, and communication channel. They are connected to the network but not part of the task board. |
| Refuse meta-meetings | The network does not have a "direction." It has a board. The board reflects current needs. If the board is active, the network is working. |

## 9.4 Failure Mode 4: Imbalanced Exchange

**Prevalence:** Structural; emerges in any network with uneven skill demand.

**Mechanism:** Some skills (plumbing, electrical) are in constant demand. Other skills (graphic design, tutoring) are needed rarely. High-demand members contribute 20 hours/month and get 20 hours back easily. Low-demand members contribute 5 hours/month and cannot spend their points because no one needs their services. Low-demand members feel undervalued and leave. The network narrows to trades-only mutual aid.

**Warning signs:**
- Some members' CAN DO columns are full but DONE columns are empty
- Membership attrition concentrated among non-trades members
- Board posts skew toward one skill category

**Structural fixes:**

| Fix | Implementation |
|---|---|
| Broaden need recognition | Actively surface needs beyond trades: menu redesign, flyer creation, branding, event organization, tutoring, administrative support |
| Match underutilized members | When a member's CAN DO is full but DONE is empty, proactively identify needs they can fill. Normalize asking for help. |
| Create internal projects | The network itself has needs: website management, recruitment flyer design, monthly gathering organization. Assign these to underutilized members. |

## 9.5 Failure Mode 5: Premature Formalization

**Prevalence:** Occurs when networks reach moderate success and members propose "professionalizing."

**Mechanism:** A functioning network attracts proposals to incorporate, obtain liability insurance, build a custom app, apply for grants, or register as a 501(c)(3). Each formalization step reintroduces the extraction systems the network was built to avoid.

**Formalization costs:**

| Proposal | Cost Introduced |
|---|---|
| Incorporation | Lawyers, filing fees, annual reporting |
| Liability insurance | Monthly premiums flowing to external insurer |
| Custom app | Tech dependency, maintenance costs, acquisition/degradation risk |
| Grant funding | Proposal writing, compliance tracking, funder accountability (replaces member accountability) |
| 501(c)(3) | Board of directors, annual IRS filings, structural constraints |

**Warning signs:**
- Proposals framed around "looking more legitimate" rather than solving a specific member problem
- Urgency to formalize driven by perceived external threat rather than internal need

**Structural fixes:**

| Fix | Implementation |
|---|---|
| Apply the benefit test | For every formalization proposal, ask: does this benefit the members directly, or does it serve appearances? |
| Delay formalization | Formalization is a one-way door. Maintain informal structure until the network is large enough that formalization solves a concrete, documented problem. |
| Recognize the trade-off | Every formal structure adds overhead and invites extraction. A group of people trading skills does not require legal infrastructure. |

## 9.6 Conflict (Not a Failure Mode)

Interpersonal conflict is commonly misidentified as a network-killing problem. It is not. Conflict is a normal operational event in any human group.

Conflict in small networks resolves effectively because social stakes are personal. Members maintain ongoing relationships outside the network (shared community, shared commerce, shared geography). This social pressure is more effective than HR departments, legal contracts, or algorithmic rating systems.

**Conflict resolution protocol (4 steps):**

1. The two parties talk directly.
2. If unresolved, they select a mutually agreed-upon third party.
3. The third party facilitates (does not decide).
4. If resolution is not possible, the departing member leaves without penalty or drama.

**Diagnostic:** If a network cannot resolve interpersonal conflict without formal rules, either the network is too large (members do not know each other well enough) or the network lacks sufficient social density. Both are correctable.

---

## Implementation Steps

1. Audit current coordination load: count how many people perform coordination tasks. If fewer than 3, redistribute immediately.
2. Review all exchange logs for contribution balance. Flag any member below 5 hours/month for two consecutive months.
3. List all active initiatives beyond the task board. For each, determine: does this strengthen the board or compete with it?
4. Spin off any initiative competing with the board into its own structure.
5. Review member CAN DO vs. DONE columns. Identify underutilized members and proactively match them to network needs.
6. Create a list of internal network needs (website, recruitment, event planning) and assign to underutilized members.
7. Reject any formalization proposal that does not solve a specific, documented member problem.
8. Document all operating procedures in the shared sheet so any member could run the network.

## Checklist

- [ ] Coordination tasks distributed across minimum 3 people
- [ ] Role rotation schedule established (3-month cycle)
- [ ] All processes documented in shared, accessible location
- [ ] Exchange tracking active (all exchanges logged with hours)
- [ ] Contribution floor defined (5 hours/month recommended)
- [ ] Check-in conversation protocol defined for below-floor members
- [ ] Side initiatives spun off from the task board into separate structures
- [ ] Underutilized members identified and proactively matched
- [ ] Internal network projects assigned to underutilized members
- [ ] Formalization proposals evaluated against member-benefit test
- [ ] Conflict resolution protocol communicated to all members (4 steps)
- [ ] Quarterly audit scheduled: coordination load, exchange balance, scope, formalization pressure
