# MBT Security Model Documentation

This document outlines the security model for the MBT application, detailing our current practices, architectural decisions, and an honest assessment of known gaps and our roadmap for continuous improvement. Our approach prioritizes a "security-by-design" philosophy, integrating security considerations from the ground up, rather than as an afterthought.

## Introduction

At MBT, safeguarding our users' data and maintaining the integrity and availability of our services are paramount. This document serves as a living record of our security posture, covering key areas such as authentication, authorization, data protection, and threat mitigation. We strive for transparency and clarity, acknowledging areas where further hardening or development is required.

## 1. Authentication

Authentication is the process of verifying the identity of a user or service. For MBT, we've adopted a modern, passwordless approach leveraging established third-party identity providers.

### Current State

**NextAuth v5 with Google OAuth:**
Our primary authentication mechanism for end-users and administrators is NextAuth.js v5, configured exclusively with Google OAuth. This strategic choice offers several significant advantages:

*   **Delegated Security:** By relying on Google as the identity provider, we offload the complex and high-stakes responsibility of password management, multi-factor authentication (MFA), and brute-force protection to a world-class security team. Google's infrastructure is inherently robust against common authentication attacks.
*   **Passwordless Experience:** Users do not create or manage passwords directly within our system. This eliminates the risk of password breaches, weak passwords, and the need for us to securely store or hash credentials.
*   **Simplified User Experience:** Most users already have a Google account, streamlining the sign-up and login process.
*   **Open Standard:** NextAuth.js implements OAuth 2.0 and OpenID Connect, ensuring adherence to widely accepted security protocols. NextAuth.js handles session management securely, including token rotation, invalidation, and secure cookie handling (e.g., `HttpOnly`, `Secure`, `SameSite`).

**Whitelisted Emails for Admin Access:**
Administrator access to sensitive parts of the application is strictly controlled. During the NextAuth.js sign-in callback, after a user successfully authenticates with Google, their email address is checked against a predefined whitelist. This whitelist is maintained as an environment variable (e.g., `ADMIN_EMAIL_WHITELIST`) or, for larger deployments, could reside in a secure configuration database.

*   **Implementation Detail:** Upon successful Google authentication, NextAuth.js provides the user's email. Our backend logic intercepts this, compares it to the `ADMIN_EMAIL_WHITELIST`. If a match is found, the user's session object is enriched with an `isAdmin: true` flag or a specific `role: 'admin'` attribute. If not, the user is authenticated as a regular user without admin privileges.
*   **Security Benefit:** This layered approach ensures that even if a regular user's Google account were compromised, they would not automatically gain administrative access to MBT without their email being explicitly added to our whitelist. It provides a clear separation of concerns between authentication (who you are) and authorization (what you can do).

### Gaps and Considerations

*   **Single Provider Dependency:** Our sole reliance on Google OAuth means that any outage or major security incident affecting Google's authentication services could impact our users' ability to log in. While Google's reliability is exceptionally high, this represents a single point of failure.
*   **Lack of Alternative Providers:** For users who do not have or prefer not to use a Google account, there is currently no alternative authentication method. This could be a barrier to adoption for some.
*   **Admin Whitelist Management:** The current whitelist mechanism is manual (updating environment variables). While simple for a small team, it lacks an audit trail or a self-service management interface for larger organizations.
*   **No MFA Management:** While Google provides robust MFA for its accounts, we do not directly manage or enforce MFA for MBT accounts. We rely entirely on users configuring MFA within their Google accounts.

### Roadmap

*   **Evaluate Additional OAuth Providers:** Consider integrating other reputable OAuth providers (e.g., Microsoft, Apple, GitHub) to offer users more choice and reduce single-provider dependency.
*   **Automated Admin Whitelist Management:** Implement a secure, auditable interface for managing administrator email whitelists, potentially backed by a database and requiring multi-person approval for changes.
*   **Enforce Google MFA (if possible):** Investigate if NextAuth.js or Google's APIs provide mechanisms to programmatically check if a user has MFA enabled on their Google account and, if not, prompt them to enable it for MBT access.

## 2. Authorization

Authorization dictates what an authenticated user or service is permitted to do within the system. Our current authorization model is role-based, with a primary distinction between regular users and administrators.

### Current State

**`requireAdmin()` Helper:**
Our primary authorization mechanism for administrative functions is a helper function located at `apps/web/lib/admin-auth.ts`. This function is designed to gate access to routes, API endpoints, or specific UI components that require administrative privileges.

*   **Implementation:** The `requireAdmin()` helper operates by inspecting the user's session object, which is populated by NextAuth.js after successful authentication. Specifically, it checks for the `isAdmin: true` flag or `role: 'admin'` attribute that was set during the authentication callback (as described in Section 1). If the session indicates the user is an administrator, the request proceeds. Otherwise, it typically throws an authorization error (e.g., HTTP 403 Forbidden) or redirects the user to an unauthorized access page.
*   **Placement:** This helper is strategically used in server-side components, API routes (`/api/admin/*`), and potentially client-side components that render admin-specific UI elements. For server-side rendering or API routes, it ensures that unauthorized users cannot even fetch the data or perform the action.
*   **Scope:** Currently, this helper provides a coarse-grained "all-or-nothing" administrative access. An administrator has full access to all administrative functions across all tenants.

### Future: Per-Tenant Role-Based Access

Our roadmap includes a significant enhancement to our authorization model to support more granular, tenant-specific role-based access control (RBAC). This is crucial for scaling our application to serve multiple tenants with varying organizational structures and access requirements.

*   **Concept:** Instead of a global `isAdmin` flag, users will have roles assigned *per tenant*. A user might be an 'admin' for Tenant A, a 'viewer' for Tenant B, and have no access to Tenant C.
*   **Example Scenarios:**
    *   **Tracy has Inn-only access:** Tracy, an employee of "The Cozy Inn" (Tenant A), might have administrative privileges *only* for "The Cozy Inn's" data and settings. She would not be able to see or modify data for "Mountain View Lodge" (Tenant B).
    *   **Vicki has her-tenant-only access:** Vicki, an owner of "Mountain View Lodge" (Tenant B), would have administrative access restricted to her own tenant's data. She would not be able to access "The Cozy Inn's" information.
*   **Implementation Considerations:**
    *   **User-Tenant-Role Mapping:** This requires a new data model to store the relationships between users, tenants, and their assigned roles (e.g., `UserTenantRole` table with `userId`, `tenantId`, `roleId`).
    *   **Role Definitions:** Define a set of roles (e.g., `Admin`, `Editor`, `Viewer`, `Manager`) and their associated permissions (e.g., `canCreateUser`, `canEditSettings`, `canViewReports`).
    *   **Permission Checks:** Update authorization helpers (e.g., `requireRole(tenantId, requiredRole)`) to check the user's roles within the *context of the current tenant*. This means every authorized action must explicitly pass the `tenantId` to the authorization helper.
    *   **UI for Role Management:** Develop an administrative interface for tenant owners or designated administrators to manage user roles within their specific tenant.
    *   **Data Scoping Enforcement:** Ensure that all data access queries are scoped by the current tenant, preventing users from inadvertently or maliciously accessing data from other tenants, even if they have high-level roles.

### Gaps

*   **Coarse-Grained Authorization:** The current system only distinguishes between "admin" and "non-admin" globally. There is no concept of roles or permissions within a tenant.
*   **Manual Admin Whitelist:** As mentioned in Authentication, the admin whitelist is static and manually managed, lacking flexibility for dynamic role assignment.
*   **No User Interface for Role Management:** There is no self-service portal for administrators to manage user roles or permissions.
*   **Potential for Over-Privilege:** A global administrator currently has access to *all* tenant data, which might violate the principle of least privilege in certain scenarios.

## 3. Secrets Management

Securely handling sensitive information like API keys, database credentials, and other configuration variables is critical. Our strategy focuses on minimizing exposure and ensuring proper lifecycle management.

### Current State

**Bitwarden as Source of Truth:**
Bitwarden serves as our primary, encrypted repository for all secrets, credentials, and sensitive notes.

*   **Benefits:**
    *   **Strong Encryption:** Bitwarden encrypts data at rest and in transit, using industry-standard algorithms (AES-256 bit, PBKDF2 SHA-256).
    *   **Secure Sharing:** It allows for secure sharing of credentials among team members with appropriate access controls, eliminating the need to transmit secrets via insecure channels (e.g., email, chat).
    *   **Access Control & Auditing:** Bitwarden provides robust access control features and audit logs, showing who accessed which secret and when.
    *   **Password Generation:** Facilitates the creation of strong, unique passwords for all services.
*   **Usage:** All developers and operations personnel are required to use Bitwarden for storing and retrieving any non-public credentials.

**Vercel Environment Variables as Deployment Surface:**
For deploying our applications, particularly `apps/web`, Vercel's environment variable management system is utilized.

*   **Vercel's Security:** Vercel encrypts environment variables at rest and provides secure injection into build and runtime environments. Variables can be scoped to specific environments (e.g., `development`, `preview`, `production`) and projects.
*   **Principle of Least Privilege:** Only the necessary secrets are exposed to the specific Vercel project and environment that requires them.
*   **Separation of Concerns:** This approach separates the storage of secrets (Bitwarden) from their deployment and consumption (Vercel), reducing the attack surface.

**Never Commit Secrets:**
A fundamental and strictly enforced policy is that no secrets, API keys, or sensitive configuration values are ever committed directly into our source code repositories (e.g., Git).

*   **Mechanism:** We use `.env` files for local development, which are explicitly excluded from version control via `.gitignore`. For production deployments, secrets are injected via Vercel's environment variable system, not hardcoded.
*   **CI/CD Implications:** Our continuous integration/continuous deployment (CI/CD) pipelines are configured to access secrets from secure environment variables provided by the build environment (e.g., Vercel) rather than attempting to retrieve them from source control.
*   **Security Benefit:** This prevents secrets from being exposed in public or private repositories, even if the repository itself were compromised.

**Rotation Policy:**
We maintain a policy for rotating secrets to minimize the window of exposure should a secret ever be compromised.

*   **Current Policy:**
    *   **Critical Secrets (e.g., database credentials, payment gateway API keys):** Quarterly rotation or immediately upon suspicion of compromise.
    *   **Non-critical Secrets (e.g., third-party API keys for analytics):** Annually or upon suspicion of compromise.
    *   **Manual Process:** Currently, secret rotation is a manual process involving:
        1.  Generating a new secret in the respective service.
        2.  Updating the secret in Bitwarden.
        3.  Updating the corresponding environment variable in Vercel.
        4.  Triggering a redeployment of affected services.

### Gaps

*   **Lack of Automated Rotation:** The current rotation process is manual, which is prone to human error and can be cumbersome for a large number of secrets.
*   **No Centralized Secret Management Service (beyond Bitwarden):** We don't use a dedicated cloud-native secret management service (e.g., AWS Secrets Manager, Azure Key Vault, HashiCorp Vault) for programmatic access and fine-grained control, especially for service-to-service communication.
*   **Limited Audit Trail for Consumption:** While Bitwarden tracks access to the secret itself, Vercel's environment variable system offers less granular auditing on *when* and *by whom* a specific environment variable was consumed by a running application instance.
*   **No Secret Scanning in CI/CD:** We do not currently integrate automated secret scanning tools into our CI/CD pipeline to detect accidental secret commits.

### Roadmap

*   **Implement Automated Secret Rotation:** Investigate and integrate cloud-native secret management services (e.g., AWS Secrets Manager if we migrate to AWS, or a Vercel-native solution if one emerges) to automate the generation and rotation of critical secrets.
*   **Centralized Secrets Management Integration:** Explore using a dedicated secrets management solution that can integrate directly with our application runtime for more dynamic and secure secret retrieval, especially for internal service communication.
*   **Integrate Secret Scanning:** Add secret scanning tools (e.g., GitGuardian, TruffleHog) to our CI/CD pipeline to prevent accidental secret commits.
*   **Detailed Audit Logging:** Enhance logging around secret consumption at runtime where possible.

## 4. Customer Data

Protecting customer data is a top priority, especially personally identifiable information (PII) and sensitive financial details. Our strategy focuses on data minimization, leveraging specialized third-party services, and adherence to privacy regulations.

### Current State

**Stripe Handles All Card Data:**
For all payment processing, we exclusively use Stripe.

*   **PCI DSS Compliance:** Stripe is a PCI DSS (Payment Card Industry Data Security Standard) Level 1 compliant service provider, the highest level of certification. This means Stripe handles all aspects of cardholder data storage, processing, and transmission according to stringent security standards.
*   **No PAN (Primary Account Number) on Our Servers:** Critically, we never directly see, store, or transmit full credit card numbers (PANs) or other sensitive cardholder data on our servers. All card data is collected via Stripe's secure client-side elements (e.g., Stripe.js, Checkout) and tokenized before it ever reaches our backend. We only store a customer ID and a payment method ID provided by Stripe, which are non-sensitive references.
*   **Reduced Scope:** This significantly reduces our PCI compliance burden and attack surface related to financial data.

**Customer Email + Name in Our DB:**
We store a minimal set of customer PII directly in our database:

*   **Email Address:** Essential for user authentication (via Google OAuth), account management, password resets (if we ever implement local accounts, though currently passwordless), and communication (e.g., transactional emails, account notifications).
*   **Name:** Used for personalization (e.g., "Hello, [Name]!"), billing information, and general user identification.
*   **Data Minimization:** We adhere to the principle of data minimization, collecting only the information strictly necessary for providing our services. We do not collect or store sensitive PII such as date of birth, physical addresses (unless explicitly required for a specific service feature and then with explicit consent), or government identifiers unless absolutely critical and with robust justification and security measures.

**PII Handling per Regulations (GDPR-flexible, CCPA applies):**
While primarily based in the US, we acknowledge the global nature of the internet and the importance of privacy regulations.

*   **GDPR-Flexible (US-based):** We adopt principles inspired by the General Data Protection Regulation (GDPR) of the European Union, even though we are US-based and not directly subject to all GDPR requirements for all users. This includes:
    *   **Data Minimization:** Only collecting necessary data.
    *   **Transparency:** Clear privacy policy outlining what data is collected, why, and how it's used.
    *   **User Rights:** Procedures for handling Data Subject Access Requests (DSARs), such as requests for data access, rectification, erasure ("right to be forgotten"), and portability. These are currently handled manually upon request.
    *   **Consent:** Obtaining consent where required for non-essential data processing (e.g., marketing communications).
*   **California CCPA Applies:** As a US-based company, we are subject to the California Consumer Privacy Act (CCPA) if we meet its thresholds. We ensure compliance by:
    *   Providing a "Do Not Sell My Personal Information" link (though we do not sell PII).
    *   Informing consumers about their rights to know what PII is collected, to delete it, and to opt-out of its sale.
    *   Having a privacy policy that addresses CCPA requirements.

### Gaps

*   **Formal DPO/Privacy Officer:** We do not currently have a dedicated Data Protection Officer (DPO) or privacy officer to oversee compliance and handle privacy requests.
*   **Automated DSR Fulfillment:** Data Subject Request (DSR) fulfillment (e.g., data export, deletion) is currently a manual process, which can be time-consuming and error-prone for a large volume of requests.
*   **Formal GDPR/CCPA Compliance Audit:** While we aim for compliance, we have not undergone a formal third-party audit specifically for GDPR or CCPA.
*   **Data Retention Policy Enforcement:** While we have an informal policy, automated enforcement of data retention schedules (e.g., automatically deleting old, non-essential data) is not yet implemented.
*   **Data Classification:** A formal system for classifying all data (e.g., PII, sensitive, public) and applying appropriate security controls based on classification is not fully mature.

### Roadmap

*   **Automate DSR Fulfillment:** Develop tools and processes to streamline and automate the handling of Data Subject Access Requests (DSARs) for data access, rectification, and erasure.
*   **Formalize Data Retention Policies:** Implement and automate enforcement of data retention policies for all types of customer data.
*   **Conduct Formal Privacy Audit:** Engage with legal and privacy experts to conduct a formal audit of our data handling practices against GDPR and CCPA requirements.
*   **Enhance Privacy Policy:** Regularly review and update our privacy policy to reflect current practices and regulatory changes.

## 5. Multi-tenant Isolation

Multi-tenancy is a core architectural principle for MBT. Ensuring strict isolation between tenants is paramount to prevent data breaches and maintain customer trust.

### Current State

**Every Domain Model Row Scoped by `tenantId`:**
The foundational principle of our multi-tenant architecture is that every single row of data in our database that belongs to a specific tenant is explicitly associated with a `tenantId` foreign key.

*   **Application-Layer Enforcement:** This `tenantId` is rigorously enforced at the application layer. Every database query, every API request, and every data manipulation operation includes a `WHERE tenantId = :currentTenantId` clause.
*   **Mechanism:** When a user authenticates, their session is associated with a `tenantId`. All subsequent operations performed by that user automatically inherit and filter by this `tenantId`. This is typically handled by an application-level middleware or context management system that injects the current tenant's ID into all relevant data access calls.
*   **Security Benefit:** This design makes it extremely difficult for a user (or a developer's accidental query) to access data belonging to another tenant, as the `tenantId` filter is almost universally applied.

**The Coordination Event Bus Enforces `tenantScope`:**
Our internal event-driven architecture, powered by a "Coordination event bus," extends this `tenantId` scoping to asynchronous operations and inter-service communication.

*   **Event Tagging:** Every event published to the bus that pertains to tenant-specific data is tagged with the relevant `tenantId`.
*   **Consumer Filtering:** Event consumers (e.g., microservices, background workers) are designed to only process events that match their designated `tenantScope` or the `tenantId` associated with their current execution context. This prevents a service processing data for Tenant A from inadvertently receiving or acting upon an event meant for Tenant B.
*   **Security Benefit:** This ensures that even in an asynchronous, distributed environment, data remains strictly isolated by tenant, preventing cross-tenant data processing errors or leaks.

**Daily Audit Query (TBD) Ensures No Cross-Tenant Leaks:**
While the application-layer enforcement is robust, we plan to implement an additional, proactive security measure: a daily audit query.

*   **Purpose:** This query will systematically scan our database for any anomalies that might indicate a cross-tenant data leak or misconfiguration. Examples include:
    *   Rows in tenant-scoped tables that inexplicably have a `NULL` or invalid `tenantId`.
    *   Rows that might have been inadvertently duplicated across `tenantId`s (though our application logic should prevent this).
    *   Counts of records per tenant to detect unexpected spikes or drops that could signal an issue.
*   **Automation:** This query will be automated to run daily, and its results will be reviewed by the operations team. Any detected anomalies will trigger immediate alerts and investigation.
*   **Proactive Detection:** This serves as a critical safety net, helping to identify potential issues that might slip past application-level controls or arise from unforeseen edge cases or bugs.

### Gaps

*   **Audit Query is TBD:** The daily audit query is currently a planned feature and not yet implemented. Until it's active, detection of certain types of cross-tenant issues relies solely on application-level logging and monitoring.
*   **Reliance on Application-Layer Enforcement:** While robust, this model relies heavily on developers consistently applying the `tenantId` filter in every query. A single missed `WHERE` clause in a complex query could lead to a breach. We do not currently use database-level Row-Level Security (RLS) policies.
*   **No Automated Cross-Tenant Data Integrity Checks:** Beyond the planned audit query, there are no automated, continuous checks specifically designed to validate tenant data integrity across the entire database.
*   **Potential for Developer Error:** The human element in consistently applying `tenantId` filters across a growing codebase introduces a potential for error.

### Roadmap

*   **Implement Daily Audit Query:** Prioritize the development and deployment of the daily audit query, integrating it into our monitoring and alerting systems.
*   **Explore Database-Level Row-Level Security (RLS):** Investigate implementing PostgreSQL's Row-Level Security policies. RLS would enforce `tenantId` filtering directly at the database level, providing a stronger, more resilient isolation guarantee that is independent of application code logic. This would serve as a robust second layer of defense.
*   **Automated Code Review for TenantId:** Integrate static analysis tools into our CI/CD pipeline to automatically check for common patterns of missing `tenantId` filters in database access code.
*   **Dedicated Testing for Tenant Isolation:** Develop a comprehensive suite of automated tests specifically designed to validate multi-tenant isolation, simulating various access patterns and attempting to violate tenant boundaries.

## 6. Webhook Security

Webhooks are essential for real-time communication with external services and internal agents. Securing them against unauthorized calls and tampering is crucial.

### Current State

**Stripe Webhook Signature Verification:**
For webhooks received from Stripe (e.g., payment success, subscription changes), we rigorously verify the authenticity of each request.

*   **Mechanism:** Stripe sends a unique signature in the `Stripe-Signature` header of each webhook request. This signature is an HMAC (Hash-based Message Authentication Code) generated using a shared secret key (unique to our webhook endpoint) and the webhook's payload. Our application recalculates this signature using the same method and compares it to the one provided by Stripe.
*   **Security Benefit:** This prevents spoofing (ensuring the request truly came from Stripe) and tampering (ensuring the payload hasn't been altered in transit). Requests with invalid signatures are immediately rejected.
*   **Replay Attack Protection:** Stripe's signature also includes a timestamp,
