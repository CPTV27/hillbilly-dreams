# MBT Database Migration Runbook (Prisma & PostgreSQL)

## 1. The Expand-Contract Pattern for Breaking Changes

The expand-contract pattern is a critical strategy for implementing breaking schema changes (renames, type changes, drops) in a zero-downtime, production-safe manner. It involves three distinct phases, each typically corresponding to a separate application deployment and database migration. This pattern mitigates risks associated with direct, destructive schema modifications.

**Why it's necessary:**
Directly renaming or dropping columns, or changing data types in a live production environment, can lead to:
*   **Downtime:** Exclusive locks on tables, blocking read/write operations.
*   **Data Loss:** Accidental deletion of data.
*   **Application Errors:** Application code expecting the old schema will fail against the new schema.

The expand-contract pattern decouples schema changes from application logic changes, allowing for gradual transitions.

**The Three-Deploy Approach:**

### Phase 1: Expand (Add New, Duplicate Data, Migrate Reads)

In this phase, you introduce the *new* schema element alongside the *old* one. Your application code is updated to *read* from the new element while still *writing* to the old.

1.  **Schema Modification (Prisma):**
    *   **Renames:** Add a *new* column with the desired name and type. Keep the *old* column.
    *   **Type Changes:** Add a *new* column with the desired type. Keep the *old* column with its original type.
    *   **Drops:** This phase is not typically needed for drops, as they are handled in Phase 3.
    *   Make the new column `Optional` (e.g., `newColumn String?`) in `schema.prisma` if it's not immediately populated.
    *   Run `prisma migrate dev --name expand_phase_1_add_new_column`. Review the generated SQL, which should only contain `ALTER TABLE ADD COLUMN`.

2.  **Application Code Deployment 1:**
    *   **Write Dual:** Update your application logic to *write* data to *both* the old column and the new column. This ensures new data is present in both locations.
    *   **Read Old/New (Prefer New):** Modify your application to *prefer reading* from the new column. If the new column is `null` (because old data hasn't been backfilled yet), fall back to reading from the old column.
    *   **Backfill Logic (Initial):** Implement a background job or script to start backfilling existing data from the old column to the new column. This can be done incrementally or in batches to avoid overwhelming the database. Ensure idempotency for backfill.

3.  **Database Migration 1 (Production):**
    *   Deploy the migration generated in step 1 to production. This adds the new column.
    *   Deploy Application Code Deployment 1.
    *   Monitor application logs and database performance.
    *   Allow the backfill process to complete. Verify data integrity between old and new columns.

### Phase 2: Contract (Migrate Writes, Validate, Prepare for Cleanup)

Once the backfill is complete and verified, and your application is successfully reading from the new column, you can fully switch your application's write operations to the new schema element.

1.  **Schema Modification (Prisma - Optional):**
    *   If the new column was initially `Optional`, you can now update `schema.prisma` to make it `Required` (`newColumn String!`) if all existing data has been backfilled and future writes will always populate it.
    *   Run `prisma migrate dev --name contract_phase_2_make_new_required` (if applicable). This will generate an `ALTER TABLE ALTER COLUMN SET NOT NULL` statement. Review carefully.

2.  **Application Code Deployment 2:**
    *   **Write New Only:** Update your application logic to *only write* data to the new column. Stop writing to the old column.
    *   **Read New Only:** Update your application logic to *only read* data from the new column. Remove any fallbacks to the old column.
    *   **Disable Backfill:** Turn off or remove the backfill job/script.
    *   Perform thorough testing to ensure the application functions correctly with the new schema element exclusively.

3.  **Database Migration 2 (Production):**
    *   Deploy Migration 2 (if applicable, e.g., making the column `NOT NULL`).
    *   Deploy Application Code Deployment 2.
    *   Monitor closely. At this point, the old column is completely unused by the application.

### Phase 3: Contract (Cleanup)

With the application fully transitioned to the new schema, the old schema element can be safely removed.

1.  **Schema Modification (Prisma):**
    *   Remove the *old* column (or table) entirely from `schema.prisma`.
    *   Run `prisma migrate dev --name contract_phase_3_remove_old_column`. Review the generated SQL, which should contain `ALTER TABLE DROP COLUMN`.

2.  **Application Code Deployment 3 (Optional):**
    *   This deployment is usually minor, simply removing any residual references to the old column from the codebase. It might not be strictly necessary if all references were removed in Phase 2, but it's good practice for code hygiene.

3.  **Database Migration 3 (Production):**
    *   Deploy Migration 3. This permanently removes the old column/table.
    *   Deploy Application Code Deployment 3.

This phased approach ensures that at no point is the application operating on an inconsistent schema, and data is preserved throughout the transition.

## 2. Local Migration Development

Developing migrations locally is the first step in the deployment pipeline. This process involves modifying your Prisma schema, generating migration files, and applying them to your local development database.

1.  **Modify `schema.prisma`:**
    *   Open your `prisma/schema.prisma` file.
    *   Make the desired schema changes (e.g., add a new model, add a field to an existing model, change a field type, add an index).

    ```prisma
    // Example: Add a new field
    model User {
      id        String   @id @default(cuid())
      email     String   @unique
      name      String?
      createdAt DateTime @default(now())
      updatedAt DateTime @updatedAt
      status    String   @default("ACTIVE") // New field
    }
    ```

2.  **Generate and Apply Local Migration:**
    *   Open your terminal in the project root.
    *   Run the Prisma Migrate dev command:
        ```bash
        prisma migrate dev --name <descriptive_migration_name>
        ```
        Replace `<descriptive_migration_name>` with a short, meaningful name that reflects the changes (e.g., `add_user_status`, `rename_product_price`).
    *   Prisma will:
        *   Compare your `schema.prisma` to your local database schema.
        *   Generate a new migration file (e.g., `prisma/migrations/20231027100000_add_user_status/migration.sql`) based on the diff.
        *   Apply this migration to your local database.
        *   Update the `_prisma_migrations` table in your local database to record the applied migration.

3.  **Review Generated SQL:**
    *   **Crucial Step:** Navigate to the newly created migration directory (`prisma/migrations/<timestamp>_<name>/`) and open `migration.sql`.
    *   **Inspect the SQL:**
        *   Does it accurately reflect your intended changes?
        *   Are there any unexpected `DROP` statements? (Prisma tries to be safe, but review is key).
        *   Are `ALTER TABLE` statements reasonable?
        *   Consider the impact on existing data for type changes or `NOT NULL` constraints.
        *   Are indexes created `CONCURRENTLY` if they are for large tables and zero-downtime is critical? (Prisma doesn't do this by default; you might need to manually edit the SQL for production).

4.  **Test Locally:**
    *   Run your application locally.
    *   Perform actions that interact with the changed parts of the schema to ensure everything works as expected.
    *   Run your unit and integration tests against your local database.

5.  **Commit Changes:**
    *   Add `schema.prisma` and the entire newly generated migration directory (`prisma/migrations/<timestamp>_<name>/`) to version control.
    *   Commit these changes.

    ```bash
    git add prisma/schema.prisma prisma/migrations/<timestamp>_<name>/
    git commit -m "feat: Add user status field to User model"
    ```

    **Never manually modify the `_prisma_migrations` table directly.** Prisma manages this table.

## 3. Preview Migration (Vercel Integration)

Before merging to your main branch, it's essential to validate your migrations against a production-like environment. For Vercel deployments, this typically means applying migrations to Vercel-managed branch databases (Preview Deployments).

1.  **Vercel Branch Deployment Workflow:**
    *   When you push a new branch or commit to an existing branch (not `main`), Vercel automatically triggers a preview deployment.
    *   As part of your Vercel build configuration (e.g., in `vercel.json` or your build command), you should include a step to apply pending Prisma migrations to the preview database.

2.  **Vercel Build Command Example:**
    *   Your `package.json` `build` script or Vercel's build command should look something like this:

    ```json
    // package.json
    {
      "scripts": {
        "build": "prisma generate && prisma migrate deploy && next build"
      }
    }
    ```
    *   **Explanation:**
        *   `prisma generate`: Generates the Prisma client based on `schema.prisma`.
        *   `prisma migrate deploy`: This is the key command. It applies all pending migrations from your `prisma/migrations` directory to the target database. For Vercel preview deployments, this target database will be the Vercel-provisioned branch database.
        *   `next build`: Your application's build command.

3.  **Validation and Clean Application:**
    *   **Crucial Requirement:** The `prisma migrate deploy` command *must* execute successfully during the preview build.
    *   If `prisma migrate deploy` fails (e.g., due to SQL errors, data type mismatches, or constraint violations), the Vercel preview deployment will fail.
    *   **Action on Failure:** If a preview deployment fails due to a migration error, you *must* fix the migration (create a new migration to correct the issue, or amend the existing one if it's still in development and not yet merged), push the fix, and ensure the preview build passes cleanly.
    *   This step ensures that your migration is syntactically correct and compatible with the target PostgreSQL version and existing schema, catching potential issues before they reach production.

4.  **Review Preview Environment:**
    *   Once the preview deployment is successful, access the preview URL.
    *   Test the application's functionality, especially any features related to the schema changes.
    *   If possible, inspect the preview database schema directly (e.g., using a database client) to confirm the migrations were applied as expected.

By rigorously testing migrations in preview environments, you significantly reduce the risk of production issues.

## 4. Production Migration

After a migration has been successfully developed locally and validated in a preview environment, it's ready for deployment to production. This process is typically automated but can sometimes require manual intervention.

### Automatic Production Migration (Recommended)

The most robust approach is to integrate `prisma migrate deploy` into your main branch's CI/CD pipeline, specifically as part of your production build/deployment process.

1.  **Merge to `main`:**
    *   Once your feature branch (containing the `schema.prisma` changes and migration files) has passed all reviews and preview tests, merge it into your `main` branch.

2.  **Vercel Build Hook / CI/CD Trigger:**
    *   Merging to `main` should automatically trigger a production build and deployment on Vercel (or your chosen CI/CD platform).
    *   Your build command for the `main` branch should be configured identically to your preview builds, including `prisma migrate deploy`.

    ```json
    // package.json
    {
      "scripts": {
        "build": "prisma generate && prisma migrate deploy && next build"
      }
    }
    ```

3.  **Zero-Downtime Deployment:**
    *   Vercel's deployment strategy typically involves building a new instance of your application, applying migrations, and then atomically switching traffic to the new instance. This allows for zero-downtime deployments.
    *   `prisma migrate deploy` is designed to be non-interactive and idempotent, making it suitable for automated production environments. It will only apply migrations that haven't been applied yet.

4.  **Monitoring:**
    *   Closely monitor your application and database logs during and immediately after the production deployment.
    *   Look for any errors, performance degradation, or unexpected behavior.

### Manual Production Migration (When Required)

In certain scenarios, such as very long-running migrations, large backfills, or specific maintenance windows, you might opt for a manual migration step.

1.  **Preparation:**
    *   Ensure the `main` branch is fully deployed with the *new application code* that expects the *new schema*. The database schema itself will still be on the old version at this point. This is crucial for the expand-contract pattern where the application can handle both states.
    *   Communicate planned downtime if the migration is expected to be blocking or long-running.

2.  **SSH into Production Environment:**
    *   Access your production server or a designated deployment machine.
    *   Ensure you have the necessary environment variables configured for your production database connection (e.g., `DATABASE_URL`).

3.  **Execute `prisma migrate deploy`:**
    *   Navigate to your project directory.
    *   Run the command:
        ```bash
        prisma migrate deploy
        ```
    *   This command will apply all pending migrations from your `prisma/migrations` directory to your production database.

4.  **Monitor Closely:**
    *   Observe the output of the command for any errors.
    *   Simultaneously monitor your database metrics (connections, CPU, I/O, active queries) and application logs.
    *   If the migration is long-running, consider using a `screen` or `tmux` session to prevent the process from terminating if your SSH connection drops.

5.  **Post-Migration Validation:**
    *   After the migration completes, perform smoke tests on your application to ensure full functionality.
    *   Validate data integrity if data transformations were part of the migration.

**Important Note:** The decision between automatic and manual deployment hinges on the nature of the migration. For routine, additive changes, automated `prisma migrate deploy` is preferred. For complex, potentially blocking, or very long-running operations, a carefully planned manual deployment during a low-traffic window might be necessary.

## 5. Migration Safety Checklist

Before approving and merging any database migration, especially for production, review it against the following checklist to ensure safety, stability, and data integrity.

*   **Additive Only?**
    *   **Check:** Does the migration only add new tables, columns, or indexes?
    *   **Impact:** Additive changes are generally the safest as they don't modify or delete existing data. They are ideal for zero-downtime deployments.
    *   **Action:** If not additive, proceed with extreme caution and consider the expand-contract pattern.

*   **Backfill Needed?**
    *   **Check:** Is a new non-nullable column being added to an existing table? Are data types changing? Is a column being renamed?
    *   **Impact:** These changes require populating new columns with existing data or transforming data. Failing to backfill can lead to `NOT NULL` constraint violations or data loss.
    *   **Action:** Implement and verify a backfill strategy (application-level script, one-off job) *before* making the new column `NOT NULL` or removing the old column.

*   **Lock Acquisition?**
    *   **Check:** Does the generated SQL include `ALTER TABLE` statements that might acquire exclusive locks (e.g., `ADD COLUMN NOT NULL`, `ALTER COLUMN TYPE`, `DROP COLUMN`)? Does it create non-concurrent indexes?
    *   **Impact:** Exclusive locks can block read/write operations on the affected table for the duration of the migration, leading to application downtime.
    *   **Action:**
        *   Prefer additive changes (`ADD COLUMN` without `NOT NULL`).
        *   Use `CREATE INDEX CONCURRENTLY` (manual edit to migration file).
        *   For unavoidable blocking operations, schedule during low-traffic windows.

*   **New Index?**
    *   **Check:** Is a new index being added to a large table?
    *   **Impact:** Creating an index on a large table can be a long-running operation and, if not done concurrently, can acquire an exclusive lock, blocking DML operations.
    *   **Action:** If the table is large or critical, **manually edit the generated `migration.sql` to use `CREATE INDEX CONCURRENTLY`**.
        ```sql
        -- Before (Prisma default):
        -- CREATE INDEX "User_email_idx" ON "User"("email");

        -- After (Manual edit for safety):
        CREATE INDEX CONCURRENTLY "User_email_idx" ON "User"("email");
        ```
        (Note: `DROP INDEX CONCURRENTLY` is not supported; `DROP INDEX` is typically fast.)

*   **Foreign Key with Cascade?**
    *   **Check:** Are you adding or modifying a foreign key constraint with `ON DELETE CASCADE` or `ON UPDATE CASCADE`?
    *   **Impact:** Cascade actions can lead to unintended mass deletions or updates of related data if the parent record is modified or deleted.
    *   **Action:**
        *   Understand the full implications of cascade actions.
        *   Ensure data integrity and referential integrity are strictly maintained.
        *   Consider `RESTRICT` or `SET NULL` (if the column allows nulls) as safer alternatives if `CASCADE` is not absolutely required.

*   **Data Type Changes:**
    *   **Check:** Are you changing the data type of an existing column (e.g., `Int` to `String`, `String` to `Text`, `Decimal` to `Float`)?
    *   **Impact:**
        *   **Implicit Casts:** Postgres might perform implicit casts, which can fail if data is incompatible (e.g., `text` to `int` on non-numeric strings).
        *   **Data Loss/Truncation:** Changing `TEXT` to `VARCHAR(N)` can truncate long strings. `FLOAT` loses precision compared to `DECIMAL`.
        *   **Performance:** Type changes often require a full table rewrite, acquiring an exclusive lock.
    *   **Action:**
        *   Use the expand-contract pattern for complex type changes.
        *   If direct change is unavoidable, specify an explicit `USING` clause in the `ALTER COLUMN TYPE` statement to guide the cast (e.g., `ALTER COLUMN foo TYPE TEXT USING foo::TEXT`).
        *   Verify data compatibility before deployment.

*   **Default Values on Existing Columns:**
    *   **Check:** Are you adding a `DEFAULT` value to an existing column that is also `NOT NULL`?
    *   **Impact:** Postgres will update every row in the table to apply the default value, which can be a long-running, blocking operation on large tables.
    *   **Action:**
        *   Prefer adding the column as `NULL`
