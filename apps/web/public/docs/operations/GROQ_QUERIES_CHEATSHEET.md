## GROQ Cheatsheet for MBT Sanity Project

GROQ (Graph-Relational Object Queries) is Sanity's powerful query language for fetching data from your content lake. It's designed to be intuitive, flexible, and highly performant, allowing you to retrieve exactly the data you need, no more, no less.

This cheatsheet will guide you through common query patterns for your MBT project's schemas, along with fundamental GROQ concepts like handling drafts, resolving references, and working with rich text and images.

---

## General GROQ Concepts

Before diving into schema-specific queries, let's cover some foundational GROQ concepts applicable across your entire Sanity project.

### 1. Drafts vs. Published Documents

Sanity automatically creates a `drafts.` prefixed ID for any document being edited. This allows you to differentiate between published content and work-in-progress.

*   **Published Document ID:** `document-id-123`
*   **Draft Document ID:** `drafts.document-id-123`

**Querying only published documents:**
The most common scenario is to only fetch published content. You can achieve this by filtering out documents with `drafts.` prefixes in their `_id`. For schemas with a `publishedAt` field, it's also good practice to ensure it's defined and in the past.

```groq
// For documents without a specific 'publishedAt' field
*[_type == "article" && !(_id in path('drafts.**'))]

// For documents with a 'publishedAt' field, ensuring it's in the past
*[_type == "article" && defined(publishedAt) && publishedAt < now()]
```

**Querying only draft documents:**
Useful for preview environments or internal tools.

```groq
*[_type == "article" && _id in path('drafts.**')]
```

**Querying both published and draft documents (e.g., for previews):**
This will return both if a draft exists, with the draft superseding the published version due to Sanity's default behavior when `_id` and `drafts._id` are present.

```groq
*[_type == "article"] // This query inherently includes both.
```

**Important Note on `publishedAt`:**
For schemas like `article`, `podcastEpisode`, `magazineIssue`, and `pressRelease`, it's best practice to filter by `defined(publishedAt) && publishedAt < now()` to ensure the content is not only published but also intended to be public. For other schemas without a `publishedAt` field (e.g., `location`, `staff`), simply filtering `!(_id in path('drafts.**'))` is sufficient.

### 2. References Resolution (`->`)

Documents in Sanity often link to each other using references. The `->` operator allows you to "dereference" these links and fetch fields from the referenced document directly within your query. This avoids multiple API calls from your frontend.

**Basic Reference Resolution:**

```groq
*[_type == "event"][0]{
  title,
  slug,
  // Resolve the 'location' reference to fetch its name and slug
  location->{
    name,
    "locationSlug": slug.current // Alias to avoid naming collision with event.slug
  }
}
```
In this example, `location` is a reference field. `location->` tells GROQ to fetch the document pointed to by the `location` field's `_ref`, and then the `{ name, "locationSlug": slug.current }` projection specifies which fields to retrieve from that referenced `location` document.

**Deep Referencing:**
You can chain `->` operators to resolve references multiple levels deep.

```groq
*[_type == "showEvent"][0]{
  title,
  showDate,
  // Resolve location, then resolve a manager reference within location (if it existed)
  location->{
    name,
    address,
    // Assuming 'manager' is a reference on the location schema
    // manager->{
    //   name,
    //   email
    // }
  },
  // Resolve performers, and for each performer, resolve their image
  performers[]->{
    name,
    genre,
    image->{
      asset->url // Get the direct URL of the image asset
    }
  }
}
```

### 3. Filtering by Reference in an Array (`in`)

Sometimes you need to find documents that refer to a specific document, especially when the reference is part of an array. The `in` operator combined with `_ref` is useful here.

**Example: Find all articles written by a specific staff member:**

```groq
// Given a staff member's ID: $staffId = "staff-id-123"
*[_type == "article" && $staffId in staff[]._ref]{
  title,
  slug.current,
  publishedAt
}
```
This query checks if the `_ref` of the `$staffId` is present in any of the `staff` references within the `staff` array of an `article` document.

### 4. Image Asset URLs + Transformations

Sanity stores image metadata and a reference to the actual image asset. To get a usable URL for your frontend, you need to query for the `_ref` of the asset and then use a client-side Sanity Image URL Builder.

**What GROQ returns for an image field:**

```groq
{
  "mainImage": {
    "_type": "image",
    "asset": {
      "_ref": "image-f0f8f8f8-f8f8-4f8f-8f8f-8f8f8f8f8f8f-1920x1080-jpg",
      "_type": "reference"
    },
    "hotspot": { /* ... */ },
    "crop": { /* ... */ }
  }
}
```

**Querying for image asset data (e.g., for `mainImage` field):**

```groq
*[_type == "article"][0]{
  title,
  mainImage{
    asset->{
      _id,
      url,
      metadata{
        dimensions{width, height},
        lqip // Low Quality Image Placeholder
      }
    },
    hotspot,
    crop
  }
}
```
This query fetches the `_id`, `url`, and `metadata` from the actual image asset document, along with `hotspot` and `crop` data defined on the image field itself.

**Client-side Usage (Sanity Image URL Builder):**
On your frontend, you would typically use a library like `@sanity/image-url` (for React/Vue) or similar.

```javascript
import imageUrlBuilder from '@sanity/image-url'
import client from './sanityClient' // Your configured Sanity client

const builder = imageUrlBuilder(client)

function urlFor(source) {
  return builder.image(source)
}

// Example usage with data fetched by GROQ:
// const imageData = {
//   _type: "image",
//   asset: { _ref: "image-..." },
//   hotspot: { /* ... */ },
//   crop: { /* ... */ }
// }
// const imageUrl = urlFor(imageData).width(800).height(450).url()
```
The image builder allows transformations like `width()`, `height()`, `crop()`, `fit()`, `format()`, and more, all handled by Sanity's CDN.

### 5. Portable Text Resolution

Portable Text is Sanity's way of handling rich text content. It's stored as an array of block objects, each representing a paragraph, heading, list item, or custom block.

**Querying Portable Text:**
You typically fetch the entire Portable Text array as-is.

```groq
*[_type == "article"][0]{
  title,
  "bodyContent": body // Fetch the 'body' field, which is Portable Text
}
```

**What GROQ returns for a Portable Text field:**

```json
{
  "bodyContent": [
    {
      "_key": "abc1",
      "_type": "block",
      "children": [
        {
          "_key": "def2",
          "_type": "span",
          "marks": [],
          "text": "This is a paragraph of rich text."
        }
      ],
      "markDefs": [],
      "style": "normal"
    },
    {
      "_key": "ghi3",
      "_type": "block",
      "children": [
        {
          "_key": "jkl4",
          "_type": "span",
          "marks": ["strong"],
          "text": "This text is bold."
        }
      ],
      "markDefs": [],
      "style": "normal"
    },
    {
      "_key": "mno5",
      "_type": "image",
      "asset": {
        "_ref": "image-..."
      },
      "alt": "An embedded image"
    }
  ]
}
```

**Client-side Usage:**
You'll need a library to render Portable Text into HTML or React components. Popular choices include:
*   `@portabletext/react` (for React)
*   `@sanity/block-content-to-html` (for server-side rendering or plain HTML)

These libraries take the Portable Text array and render it according to your defined components for blocks, marks, and custom types.

## Schema-Specific Queries for MBT Project

Here are common GROQ queries tailored to your MBT project's schemas.

### `article` Schema

Represents blog posts, news, or general articles.

#### 1. List All Published Articles, Ordered by Publication Date (Newest First)

```groq
*[_type == "article" && defined(publishedAt) && publishedAt < now()] | order(publishedAt desc) {
  _id,
  title,
  "slug": slug.current,
  publishedAt,
  // Fetch only the URL of the main image for listings
  mainImage{
    asset->url
  },
  // Get names of the first 3 staff members
  "staffNames": staff[]->name[0...3],
  "excerpt": array::join(body[0].children[0].text, "") // Basic excerpt from first block
}
```
**Description:** Fetches published articles, ordered from newest to oldest. It includes essential fields for a listing page, resolves the main image URL, and gets names of associated staff. It also attempts to create a simple excerpt from the first Portable Text block.

#### 2. Get a Single Article by Slug

```groq
*[_type == "article" && slug.current == $slug && defined(publishedAt) && publishedAt < now()][0]{
  _id,
  _createdAt,
  title,
  "slug": slug.current,
  publishedAt,
  mainImage{
    asset->{
      _id,
      url,
      metadata{dimensions{width, height}, lqip}
    },
    hotspot,
    crop
  },
  body, // Portable Text
  // Resolve all staff members with their name, slug, and image
  staff[]->{
    _id,
    name,
    "slug": slug.current,
    role,
    image{
      asset->url
    }
  },
  tags // Assuming tags are an array of strings
}
```
**Description:** Retrieves a single published article matching the provided `$slug` parameter. It includes a comprehensive projection, resolving all related `staff` details and full image asset data for `mainImage`.

#### 3. Related Article Queries

**a. Get Articles by a Specific Staff Member:**

```groq

*[_type == "article" && defined(publishedAt) && publishedAt < now() && $staffId in staff[]._ref] | order(publishedAt desc){

  title,
  "slug": slug.current,
  publishedAt,
  mainImage{
    asset->url
  }
}
```
**Description:** Finds all published articles where the `staff` array contains a reference to the staff member identified by `$staffId`.

**b. Get Recent Articles (e.g., for homepage feed):**

```groq

  _id,
  title,
  "slug": slug.current,
  publishedAt,

    asset->url
  }
}
```
**Description:** Fetches the 5 most recently published articles, useful for "Latest News" sections.
