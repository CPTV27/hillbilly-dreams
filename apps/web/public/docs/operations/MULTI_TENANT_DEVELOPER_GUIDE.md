# MBT Multi-Tenant Developer Guide

This guide outlines the principles and practices for building features within the MBT platform that respect tenant boundaries. Understanding and adhering to these guidelines is crucial for maintaining data integrity, security, and a consistent user experience across all tenants.

## 2. Hostname Routing

The Next.js application leverages hostname routing to serve different tenant experiences from the same codebase. This is managed by `apps/web/middleware.ts` in conjunction with `apps/web/config/domain-routes.ts`.

**How it works:**
1.  **Incoming Request:** A user's browser sends a request to a domain (e.g., `www.bigmuddy.com`).
2.  **Middleware Interception:** `apps/web/middleware.ts` intercepts this request.
3.  **Tenant Identification:** The middleware extracts the `Host` header and uses `getTenantByDomain` (from `tenants.ts`) to identify the tenant.
4.  **Route Rewriting:** Based on the identified tenant and `domain-routes.ts`, the middleware rewrites the request URL. This allows a single Next.js page component (e.g., `src/app/[tenantSlug]/events/page.tsx`) to render content specific to the tenant identified in the URL.
5.  **Header Injection:** Crucially, the middleware also injects an `x-mbt-tenant` header into the request with the `id` of the identified tenant. This header is then available downstream in route handlers and server components.

**`apps/web/config/domain-routes.ts` example:**

```typescript
// apps/web/config/domain-routes.ts
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';
import { getTenantByDomain, tenants } from './tenants'; // Assume tenants.ts is in the same dir

export function middleware(req: NextRequest) {
  const url = req.nextUrl;
  const hostname = req.headers.get('host');

  // Handle API routes or static assets that don't need tenant routing
  if (url.pathname.startsWith('/api') || url.pathname.startsWith('/_next/static')) {
    return NextResponse.next();
  }

  // Find tenant based on hostname
  const tenant = getTenantByDomain(hostname || '');

  if (tenant) {
    // Rewrite the URL to include the tenant slug in the path
    // e.g., bigmuddy.com/events -> /big-muddy/events
    const newUrl = new URL(`/${tenant.slug}${url.pathname}${url.search}`, req.url);

    // Clone the request headers and add the x-mbt-tenant header
    const requestHeaders = new Headers(req.headers);
    requestHeaders.set('x-mbt-tenant', tenant.id);
    requestHeaders.set('x-mbt-tenant-slug', tenant.slug); // Also useful for logging/analytics

    return NextResponse.rewrite(newUrl, {
      request: {
        headers: requestHeaders,
      },
    });
  }

  // Fallback for unknown domains or internal MBT routes
  // For MBT internal tools or admin dashboards, we might have a default tenant or no tenant
  if (hostname === 'admin.mbt.com' || hostname === 'localhost:3000') {
    // Potentially set a default tenant for local dev or admin context, or no tenant
    // For now, let's just pass through for localhost as a dev convenience
    if (hostname === 'localhost:3000') {
      const requestHeaders = new Headers(req.headers);
      // For local dev, we might set a default tenant if not specified,
      // or rely on explicit tenant selection in the UI/URL.
      // For simplicity here, we'll assume local dev might hit a specific tenant domain
      // or handle tenant selection in the app logic for localhost.
      // E.g., if you hit localhost:3000/big-muddy, the page component will read big-muddy from the URL.
      // But for API routes, if hitting localhost:3000/api/some-endpoint, x-mbt-tenant might be missing
      // unless middleware explicitly sets a dev default.
      // For this guide, we assume local dev uses specific hostnames via /etc/hosts for tenant context.
      return NextResponse.next({
        request: {
          headers: requestHeaders,
        },
      });
    }
  }

  // If no tenant is found, redirect to a default landing page or error page
  // return NextResponse.redirect(new URL('/unknown-tenant', req.url));
  return NextResponse.next(); // Or handle as an error/redirect for production
}

// Config for Next.js middleware matching
export const config = {
  matcher: [
    /*
     * Match all request paths except for the ones starting with:
     * - _next/static (static files)
     * - _next/image (image optimization files)
     * - favicon.ico (favicon file)
     * - assets (public assets)
     * - api (API routes, might be handled differently or after tenant identification)
     */
    '/((?!_next/static|_next/image|favicon.ico|assets|api).*)',
  ],
};
```

This setup ensures that despite serving `bigmuddy.com/events` and `bearsvilletheater.com/events` from the same `src/app/[tenantSlug]/events/page.tsx` component, the application inherently knows which tenant it is serving.

## 3. Tenant Context in Route Handlers

After the `middleware.ts` has identified the tenant and injected the `x-mbt-tenant` header, route handlers (API routes, server actions, server components) can access this context. This is the primary way to determine the active tenant for any server-side operation.

**How to read the `x-mbt-tenant` header:**

Within any server-side context (API route, Server Component, Server Action), you can retrieve the header from the incoming request.

```typescript
// Example: apps/web/app/api/events/route.ts (API Route)
import { NextRequest, NextResponse } from 'next/server';
import { db } from '@mbt/db'; // Our ORM instance

export async function GET(request: NextRequest) {
  const tenantId = request.headers.get('x-mbt-tenant');

  if (!tenantId) {
    return new NextResponse('Tenant not identified', { status: 400 });
  }

  try {
    // Use the tenantId to scope data retrieval
    const events = await db.event.findMany({
      where: {
        tenantId: tenantId,
        isActive: true,
      },
      orderBy: {
        date: 'asc',
      },
    });

    return NextResponse.json(events);
  } catch (error) {
    console.error('Failed to fetch events for tenant:', tenantId, error);
    return new NextResponse('Internal Server Error', { status: 500 });
  }
}
```

**Important:** Always validate the presence of `tenantId`. If it's missing, it indicates a misconfiguration in the middleware or an unauthenticated/malformed request.

## 4. Per-Tenant Data Scoping

**Every domain model row MUST have a `tenantId` column.** This is the foundational principle for data isolation. All data pertaining to a specific tenant must be explicitly linked to that tenant via its `tenantId`.

When performing any database operation (reads, writes, updates, deletes), you **must** include the `tenantId` in your query conditions.

**Common Mistake: Forgetting `tenantId` in `findMany`**

Failing to include `tenantId` in a `findMany` (or `findUnique`, `updateMany`, `deleteMany`, etc.) query is a critical security vulnerability, as it can lead to data leaks or modifications across tenants.

```typescript
// Example: Correct and Incorrect Prisma queries

// Assume `tenantId` is retrieved from the request header as shown above.
const tenantId = 'big-muddy'; // In real code, this comes from req.headers.get('x-mbt-tenant')

// ❌ INCORRECT: Data leak risk!
// This query would return ALL events from ALL tenants.
const allEvents = await db.event.findMany({
  where: {
    isActive: true,
  },
});
console.log("DANGER: Fetched events for all tenants!", allEvents);


// ✅ CORRECT: Data scoped to the current tenant.
// This query correctly filters events for 'big-muddy' only.
const tenantEvents = await db.event.findMany({
  where: {
    tenantId: tenantId, // ALWAYS include tenantId
    isActive: true,
  },
  select: {
    id: true,
    name: true,
    date: true,
    // NEVER expose tenantId in API responses unless explicitly required for admin purposes
  }
});
console.log(`Fetched events for tenant ${tenantId}`, tenantEvents);


// ✅ CORRECT: Creating a new record for the current tenant.
await db.event.create({
  data: {
    name: 'Tenant Specific Concert',
    date: new Date(),
    tenantId: tenantId, // ALWAYS include tenantId on creation
    isActive: true,
  },
});

// ✅ CORRECT: Updating records for the current tenant.
await db.event.updateMany({
  where: {
    tenantId: tenantId, // ALWAYS include tenantId in updateMany where clause
    id: 'some-event-id',
  },
  data: {
    isActive: false,
  },
});

// ✅ CORRECT: Deleting records for the current tenant.
await db.event.deleteMany({
  where: {
    tenantId: tenantId, // ALWAYS include tenantId in deleteMany where clause
    id: 'another-event-id',
  },
});

// For `findUnique` or `update` by a unique ID, ensure the ID *and* tenantId match.
// If the unique ID is globally unique (e.g., UUID), you might still want to verify tenantId
// to prevent a tenant from accessing another tenant's record even if they guess the UUID.
const specificEvent = await db.event.findUnique({
  where: {
    id: 'some-globally-unique-id',
    tenantId: tenantId, // Redundant if ID is tenant-scoped, but good practice for safety
  },
});
```

**Review Process:** All pull requests involving database interactions will be rigorously reviewed for `tenantId` inclusion in `where` clauses. Automated linters and pre-commit hooks will eventually be implemented to help enforce this.

## 5. Cross-Tenant Data — When It's Actually Needed

While tenant isolation is the default and preferred mode of operation, there are legitimate scenarios where accessing or aggregating data across multiple tenants is necessary. These are typically for platform-level operations, not tenant-facing features.

**Legitimate Use Cases:**

*   **Admin Dashboards (e.g., Treasury):** Our internal administrative tools, such as the Treasury dashboard, need to view financial transactions, user activity, or system health metrics across all tenants to manage the platform effectively.
*   **Platform-Wide Analytics:** Aggregated data for understanding overall platform usage, identifying trends, or generating internal reports (e.g., total number of events hosted, total ticket sales across all tenants).
*   **MBT-level Operations:** Tasks like system-wide backups, database maintenance, or specific data migrations that might affect multiple tenants.
*   **Super-Admin Features:** Features accessible only to MBT super-admins who have explicit permission to view or manage data across the entire platform.

**How to Handle Cross-Tenant Access:**

When you need to perform an operation that *intentionally* bypasses `tenantId` filtering, you must:

1.  **Explicitly Mark the Code:** Add comments, specific function names, or even dedicated service layers to clearly indicate that this code operates at a cross-tenant level. This helps prevent accidental misuse and aids in code reviews.
2.  **Require Elevated Permissions:** Ensure that any code path allowing cross-tenant access is protected by robust authentication and authorization checks, restricting it to MBT internal users or specific super-admin roles.
3.  **Use Dedicated Contexts:** If possible, use a separate API context or a distinct service that bypasses the standard `x-mbt-tenant` header validation, but only after strict permission checks.

```typescript
// Example: Admin function to get all events across all tenants
// This function should ONLY be called from an authorized internal tool/service.

import { db } from '@mbt/db';

/**
 * WARNING: This function retrieves events across ALL tenants.
 * It must only be used by authorized MBT internal services (e.g., admin dashboard, analytics jobs).
 * DO NOT expose this via tenant-facing APIs.
 * @param {string} callerContext - Describes where this function is being called from (e.g., "Treasury Dashboard", "Daily Analytics Job").
 */
export async function getAllEventsPlatformWide(callerContext: string) {
  if (!['Treasury Dashboard', 'Daily Analytics Job', 'Internal Audit'].includes(callerContext)) {
    console.warn(`Unauthorized attempt to access cross-tenant data from: ${callerContext}`);
    throw new Error('Unauthorized access to platform-wide data.');
  }

  console.log(`Accessing platform-wide events from: ${callerContext}`);
  const allEvents = await db.event.findMany({
    // Purposely omitting tenantId here. This is the exception, not the rule.
    select: {
      id: true,
      name: true,
      date: true,
      tenantId: true, // In this case, we DO want to see the tenantId
    },
    orderBy: {
      date: 'asc',
    },
  });

  return allEvents;
}

// Example usage in an internal admin API route:
// apps/web/app/api/admin/platform-events/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { getAllEventsPlatformWide } from '../../../../services/admin/eventService'; // Assuming a dedicated admin service layer
import { authGuard } from '../../../../utils/auth'; // Assuming an internal auth utility

export async function GET(request: NextRequest) {
  // 1. Authenticate and authorize internal MBT admin user
  if (!authGuard.isAdmin(request)) {
    return new NextResponse('Unauthorized', { status: 401 });
  }

  try {
    const events = await getAllEventsPlatformWide('Treasury Dashboard API');
    return NextResponse.json(events);
  } catch (error) {
    console.error('Failed to fetch platform-wide events:', error);
    return new NextResponse('Internal Server Error', { status: 500 });
  }
}
```

## 6. The Coordination Event Bus + Tenant Scope

The Coordination event bus is used for asynchronous communication between different parts of our system. When publishing or subscribing to events, it's critical to consider tenant scope to prevent cross-tenant data processing or unintended side effects.

The `EventHandler.tenantScope` property dictates how an event handler should process events relative to tenants.

### `EventHandler.tenantScope` Modes:

1.  **`'own'` (Default):**
    *   **Description:** The event handler will only process events that originate from its *own* tenant. This is the **default and safest mode** for tenant-specific business logic.
    *   **When to Use:** For almost all tenant-facing features. If an event is triggered within `big-muddy`'s context (e.g., `big-muddy.event.created`), an `'own'` scoped handler for `bearsville` will ignore it. An `'own'` scoped handler for `big-muddy` will process it.
    *   **Implementation:** The event payload *must* include a `tenantId` field. The event bus will automatically filter events based on the `tenantId` in the event payload matching the `tenantId` of the handler's context (which is typically derived from the `x-mbt-tenant` header at the time the handler is invoked, or from its explicit configuration).

    ```typescript
    // Example: apps/web/events/handlers/TicketPurchasedHandler.ts
    import { EventHandler } from '@mbt/coordination';
    import { db } from '@mbt/db';
    import { TicketPurchasedEvent } from '@mbt/events/types';

    export const TicketPurchasedHandler: EventHandler<TicketPurchasedEvent> = {
      eventName: 'ticket.purchased',
      tenantScope: 'own', // Only process events for the handler's tenant context
      async handle(event) {
        // Event payload MUST contain tenantId
        if (!event.tenantId) {
          console.error('TicketPurchasedEvent missing tenantId, skipping processing.');
          return;
        }

        console.log(`Processing ticket purchased event for tenant: ${event.tenantId}`);

        // Business logic specific to the tenant that owns this event
        await db.ticket.update({
          where: { id: event.ticketId, tenantId: event.tenantId }, // Crucial: scope by tenantId
          data: { status: 'CONFIRMED' },
        });

        // Send a confirmation email specific to this tenant's branding
        // ...
      },
    };
    ```

2.  **`'all'`:**
    *   **Description:** The event handler will process events from **all** tenants. This mode should be used with extreme caution and only for platform-level concerns.
    *   **When to Use:** For cross-tenant analytics, platform-wide audit logging, centralized billing/reporting services, or global synchronization tasks.
    *   **Implementation:** The handler will receive events regardless of their `tenantId`. The handler *must* then explicitly handle the `tenantId` from the event payload, often by logging it or routing it to a tenant-specific sub-process.
    *   **Security Risk:** If not handled carefully, an `'all'` scoped handler can inadvertently process or expose data across tenants.

    ```typescript
    // Example: apps/web/events/handlers/GlobalActivityLoggerHandler.ts
    import { EventHandler } from '@mbt/coordination';
    import { analyticsService } from '@mbt/analytics'; // Dedicated analytics service
    import { AuditEvent } from '@mbt/events/types';

    export const GlobalActivityLoggerHandler: EventHandler<AuditEvent> = {
      eventName: 'user.activity', // Generic event for any user activity
      tenantScope: 'all', // Process activity from all tenants
      async handle(event) {
        // Event payload should contain tenantId, userId, action, etc.
        if (!event.tenantId) {
          console.warn('AuditEvent missing tenantId, logging as platform-level event.');
        }

        console.log(`Global activity logged: Tenant=${event.tenantId || 'N/A'}, User=${event.userId}, Action=${event.action}`);

        // Send to a centralized analytics system that can aggregate across tenants
        await analyticsService.trackPlatformActivity({
          tenantId: event.tenantId,
          userId: event.userId,
          action: event.action,
          timestamp: new Date(),
        });

        // No tenant-specific database operations should occur here without explicit tenantId filtering.
      },
    };
    ```

3.  **`allowlist: string[]`:**
    *   **Description:** The event handler will only process events that originate from a specific, explicitly defined list of tenant IDs.
    *   **When to Use:** For cross-tenant integrations with partner systems, or specialized services that manage a subset of tenants. For example, a handler that synchronizes data only between `big-muddy` and `studio-c` for a joint venture.
    *   **Implementation:** The event bus will filter events based on whether their `tenantId` is present in the `allowlist` array.

    ```typescript
    // Example: apps/web/events/handlers/PartnerSyncHandler.ts
    import { EventHandler } from '@mbt/coordination';
    import { partnerIntegrationService } from '@mbt/integrations';
    import { PartnerSyncEvent } from '@mbt/events/types';

    export const PartnerSyncHandler: EventHandler<PartnerSyncEvent> = {
      eventName: 'partner.data.update',
      tenantScope: ['big-muddy', 'studio-c'], // Only process events from these specific tenants
      async handle(event) {
        if (!event.tenantId) {
          console.error('PartnerSyncEvent missing tenantId, skipping processing.');
          return;
        }

        if (!this.tenantScope.includes(event.tenantId)) {
          // This check is redundant as the event bus handles it, but good for clarity.
          console.warn(`PartnerSyncHandler received event from unallowed tenant: ${event.tenantId}`);
          return;
        }

        console.log(`Processing partner sync event for allowed tenant: ${event.tenantId}`);
        await partnerIntegrationService.syncData(event.tenantId, event.payload);
      },
    };
    ```

**Key Takeaway:** **Always default to `'own'` for `tenantScope`**. Only deviate to `'all'` or `allowlist` when there's a clear, justified, and audited reason for cross-tenant interaction.
