S2PX as Lyra's operating layer

The platform underneath, the audio layer on top — written for Margaret, Nick, and Mason ahead of today's meeting at EMPAC. What's already built, what we'd build for Lyra specifically, and how the EMPAC capture pipeline runs end-to-end.

Chase Pierson · Big Muddy / Measurably Better Things

01What S2PX is today

S2PX is a B2B operating system for 3D-scanning + BIM companies, in production at s2px.web.app with staging and sandbox tiers. 567+ projects are in the catalog as of this month. Multi-tenant from day one, AI-first, production-grade infrastructure, currently being prepped to absorb additional customers from the AEC vertical (Twinner is the anchor at $134K/yr).

Stack

client/src/ — React 19 + Vite 6 SPA · Tailwind CSS 4

server/ — Express API on Cloud Run (us-east4)

shared/ — Drizzle ORM (61 tables across db / pricing / strategy schemas), framework-agnostic engine

AI — 7 Gemini agents via the Google GenAI SDK, server-side only

Storage — Cloud SQL Postgres + GCS buckets per tenant

Auth — Firebase Auth (multi-environment shared)

What's relevant to Lyra: the platform exists, runs production traffic today, and was designed multi-tenant from the first commit. Adding Lyra as a tenant means provisioning a Cloud SQL schema, a GCS bucket, and a domain — not building infrastructure from scratch.

  • Multi-format export: OBJ, GLB, USDZ, PLY for Unity / Unreal / web viewers

03What S2PX does with spatial data — already

Once a scan lands, the customer portal provides:

  • 3D viewer in-browser — Three.js based, runs on phone or desktop, no plugin
  • Annotation overlay — pin notes to mesh coordinates, share with collaborators
  • Asset registry — every scan tied to a project, every project to a customer, every customer to a tenant
  • BIM annotation export — for AEC customers, scan deliverables route into Revit / Rhino workflows
  • API access — REST endpoints at /api/scans, /api/scans/:id/mesh, /api/scans/:id/textures, /api/projects/:id/scans. Authenticated via tenant API keys.

The API layer is the integration surface for Lyra. Lyra's services don't have to live inside S2PX — they can live anywhere, talk to S2PX over HTTPS, and read/write through the REST endpoints. Same way Lyra would talk to any other operating-platform vendor, except this one was built by people who've been on the customer side.

04What we'd build for Lyra — the audio layer

This is the honest part. S2PX has no acoustic processing today. Every gram of audio capability in the Lyra integration is either Lyra's existing IP (LMIR database, the Acoustic Architect agent, Mason's analysis stack) or new code we'd write to bridge between S2PX's existing pipeline and Lyra's existing services.

The build is small in scope, large in value:

  • Build Spatial-audio extraction service. Yesterday's iPhone walkthroughs at EMPAC contain 4-channel APAC (Apple Positional Audio Codec) tracks embedded with the video frames — full ambisonic-style room capture, no separate rig. We need a small service that demuxes the APAC track from the .MOV, normalizes it, hands it to Lyra's LMIR fingerprint service.
  • Connect LMIR fingerprint integration. S2PX job queue calls Lyra's LMIR API on every new scan that has audio attached. Result lands back in the Scan record as acoustic_profile.
  • Connect RT60 + frequency response analysis. Same pattern — call Lyra's analysis service, store result on the Scan record.
  • Build Customer portal: acoustic overlay. The 3D viewer learns to render Lyra's acoustic visualization (heat maps, propagation cones, whatever Mason's spec calls for) on top of the mesh.
  • Build Spatial-audio export. Customer can download the room's acoustic signature as an ambisonic file Unity / Wwise / FMOD can ingest directly.

Estimated build scope: 2–4 weeks of focused engineering, sequenced as: spatial-audio extraction first (so we have the raw input), then LMIR integration (so we have Lyra's analysis output), then portal overlay (so customers see it), then export (so customers can use it).

None of it is research. All of it is plumbing — well-understood plumbing — between two systems that already exist.

05The Lyra Walkthrough product (the new flow)

One phone capture = mesh + room acoustic profile, in one shot, no separate rig.

Locked yesterday once we realized the iPhone Pro's video already includes 4-channel positional audio. The customer flow becomes:

  1. Customer shoots a 1–3 minute phone-camera walkthrough of their space
  2. Uploads via the S2PX portal (or a Lyra-branded surface that hits the same API)
  3. Cloud pipeline extracts video frames → Polycam Web for mesh; demuxes APAC audio → Lyra LMIR service for acoustic profile
  4. Customer portal shows: "Your scan is processing. ETA 2-4 hours" → email when done with mesh + acoustic-profile viewer link
  5. Both deliverables sit on the same Scan record. One bill, one thread, one source of truth.

This is what makes Lyra accessible to customers without specialized hardware — no LiDAR phone required, no dedicated acoustic mic rig. Any iPhone Pro user becomes a potential acoustic-twin client. The TAM expansion is roughly 10× compared to LiDAR-only.

The branding decision (Lyra Walkthrough? Lyra Capture? Lyra Snap?) is one of the things to discuss today.
