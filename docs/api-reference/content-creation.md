# `@bigmuddy/content-creation` API Reference

The `@bigmuddy/content-creation` module provides a comprehensive suite of tools for automating and streamlining the content generation process within BigMuddy applications. It orchestrates entity recognition, external data sourcing (like image fetching), AI-driven content generation, and integration with content management systems (CMS) like Sanity. This module is designed to facilitate the creation of various content types, from marketing copy and social media posts to detailed articles, by leveraging structured data and configurable templates.

## 1. Core Concepts & Data Flow

The content creation process within BigMuddy follows a structured, multi-stage pipeline, designed for flexibility, accuracy, and integration.

### 1.1. High-Level Overview

1.  **User Input & Context Setup**: A user (or automated process) initiates content creation, providing a general prompt, desired content type, and potentially a brand context.
2.  **Entity Resolution**: Key entities mentioned in or implied by the input are identified and validated against known data sources. This ensures factual accuracy and consistency.
3.  **External Data Sourcing**: Based on resolved entities, relevant external assets (e.g., images from Immich) are fetched.
4.  **Prompt Assembly**: A sophisticated system prompt is constructed, incorporating the user's intent, resolved entities, brand voice guidelines, content rules, and fetched assets.
5.  **AI Generation**: The assembled prompt is sent to a large language model (LLM) to generate the raw content.
6.  **Draft Saving**: The generated content, along with associated metadata and assets, is saved as a draft in the CMS (e.g., Sanity) for review and further editing.

### 1.2. Two-Stage Entity Resolution Flow

The module employs a robust two-stage entity resolution process to ensure that identified entities are not only recognized but also validated and disambiguated against authoritative BigMuddy data.

1.  **Stage 1: Initial Extraction (Candidate Generation)**:
    *   The `entityResolver.resolveEntities` function takes raw content (e.g., a user's prompt or a piece of existing text) and, using various `ExtractorFn` implementations (which might leverage LLMs, regex, or NLP libraries), identifies potential entities.
    *   These initial extractions are considered "candidates" because they haven't yet been verified against BigMuddy's internal data. They include a `name` and a potential `type` (e.g., `Business`, `Product`, `Service`).
    *   This stage is highly flexible and can be configured with different extraction strategies based on the `ContentType` and `Brand` context. For example, an article about "local businesses" might use a different extractor than a product description for "BigMuddy Coffee."

2.  **Stage 2: Validation and Disambiguation (Authoritative Resolution)**:
    *   Each `EntityCandidate` generated in Stage 1 is then cross-referenced against BigMuddy's authoritative data sources, such as the `DirectoryBusiness` for business entities, or internal product catalogs.
    *   This stage performs several critical functions:
        *   **Validation**: Confirms if a candidate entity actually exists within BigMuddy's ecosystem. For instance, if "BigMuddy Cafe" is extracted, this stage checks if an entity with that name exists and is officially recognized.
        *   **Disambiguation**: If multiple entities share a similar name (e.g., "BigMuddy Bakery" and "BigMuddy Bakeshop"), this stage uses contextual clues or predefined rules to select the most relevant one.
        *   **Enrichment**: Once an entity is validated, its full, authoritative data (e.g., official ID, precise name, associated services, location) can be retrieved and attached to the resolved entity. This enriched data is crucial for subsequent steps like image search or detailed content generation.

This two-stage approach balances the flexibility of broad entity recognition with the necessity of factual accuracy and consistency with BigMuddy's internal data.

### 1.3. Visibility-Gate on `DirectoryBusiness`

During the second stage of entity resolution, particularly when resolving entities of type `Business`, a "visibility-gate" mechanism is applied. This gate ensures that only businesses marked as "visible" or "public" within BigMuddy's `DirectoryBusiness` (or equivalent authoritative source) are considered valid and resolvable entities for content generation.

*   **Purpose**: To prevent the accidental generation of content about businesses that are internal, pending, deprecated, or otherwise not intended for public exposure. It acts as a safeguard against generating content for non-public entities.
*   **Mechanism**: When an `EntityCandidate` for a `Business` type is being validated, the system queries `DirectoryBusiness`. If the matching business record has a `visibility` or `status` field set to a non-public value (e.g., `internal`, `draft`, `archived`), that candidate is rejected or flagged as unresolvable. Only businesses explicitly marked as publicly `visible` are passed through the gate.
*   **Impact**: This means that even if a user explicitly prompts for "our internal project X," if "project X" isn't a publicly visible `DirectoryBusiness` entry, the entity resolution process will not resolve it as a valid business entity, guiding the content generation towards public-facing information or flagging the entity as unrecognized.

### 1.4. Sanity `drafts.{id}` Convention

When content is generated and saved via the `sanityWriter`, it adheres to a specific convention for draft management within Sanity Studio: `drafts.{id}`.

*   **Purpose**: This convention is fundamental to Sanity's real-time collaborative editing and publishing workflow. It allows for:
    *   **Simultaneous Editing**: Multiple users can edit the same document without conflicts.
    *   **Draft-first Workflow**: All changes are initially saved as drafts, separate from the published version.
    *   **Previewing**: Drafts can be previewed before publishing.
    *   **Versioning**: Sanity tracks changes to drafts, allowing for history and rollbacks.
*   **Mechanism**:
    *   When `sanityWriter.saveDraft` is called, it creates or updates a document in Sanity with an ID prefixed by `drafts.`. For example, if a document's canonical ID is `myArticle123`, its draft version will have the ID `drafts.myArticle123`.
    *   The `GeneratedDraft` object returned by `saveDraft` will include this `drafts.{id}` as its primary identifier within Sanity.
    *   When the draft is eventually published through Sanity Studio, Sanity automatically promotes the `drafts.{id}` content to the canonical `id` (e.g., `myArticle123`), effectively replacing the previously published version.
*   **Integration**: This convention ensures that the AI-generated content seamlessly integrates into existing Sanity workflows, allowing human editors to review, refine, and publish the AI output using familiar tools. It abstracts away the complexities of direct publishing and focuses on providing an editable starting point.

### 1.5. Brand-Voice Template Merge Order

The `templates.assembleSystemPrompt` function constructs a sophisticated prompt for the LLM by merging various configurable elements, with a specific hierarchy to ensure brand consistency and adherence to content rules. The merge order dictates which rules take precedence when conflicts arise or when combining different aspects of the brand's voice.

The merge order is as follows, from lowest precedence (overridden by subsequent layers) to highest precedence (overriding previous layers):

1.  **Base Voice Profile**:
    *   **Source**: A foundational, generic brand voice profile (e.g., "professional," "friendly," "authoritative"). This is the default tone and style for BigMuddy content when no specific brand is active or when higher-level rules don't override it.
    *   **Content**: General guidelines on tone, style, vocabulary, complexity, and overall persona.
    *   **Purpose**: Establishes a consistent baseline for all generated content.

2.  **Brand Profile (`Brand.voiceProfile`)**:
    *   **Source**: Specific voice guidelines defined for a particular `Brand` entity. This is retrieved from the `Brand` object passed into the context.
    *   **Content**: Overrides or refines the base voice profile with brand-specific nuances. For example, if the base is "professional," a specific brand might be "professional but approachable" or "professional and innovative."
    *   **Purpose**: Tailors the content to the unique identity and communication style of a specific BigMuddy brand.

3.  **Content Rules (`Brand.contentRules`)**:
    *   **Source**: Specific rules or directives associated with the `Brand` or the `ContentType` itself. These are often structured guidelines for specific content types (e.g., "all product descriptions must include a call to action," "articles must be SEO-optimized for keywords X, Y, Z").
    *   **Content**: Directives regarding structure, required elements, forbidden topics, specific phrasing patterns, or mandatory inclusions.
    *   **Purpose**: Ensures adherence to legal, marketing, or stylistic requirements for specific content categories or brands. These rules are more about *what* must be in the content, or *how* it must be structured, rather than just the voice.

4.  **Length Constraints**:
    *   **Source**: Explicit length requirements (e.g., "150-200 words," "short paragraph," "bullet points") provided in the `WizardContext`.
    *   **Content**: Instructions to the LLM regarding the desired output length or format.
    *   **Purpose**: Guides the LLM to produce content that fits specific platform requirements (e.g., Twitter character limits, blog post word counts).

5.  **Forbidden Phrases (`Brand.forbiddenPhrases`)**:
    *   **Source**: A list of words, phrases, or concepts that *must not* appear in the generated content for a specific `Brand`. This is typically part of the `Brand` configuration.
    *   **Content**: Negative constraints, explicitly instructing the LLM to avoid certain terminology due to legal, ethical, or brand-image considerations.
    *   **Purpose**: Acts as a final filter or explicit directive to prevent undesirable language from appearing in the output. This has the highest precedence as it's a hard constraint.

By following this precise merge order, the system ensures that highly specific, critical rules (like forbidden phrases or explicit length requirements) override more general stylistic guidelines, while still maintaining a consistent brand voice built upon a foundational profile.

### 3.2. `immichClient`

The `immichClient` module provides an interface to interact with the Immich photo management system, allowing for programmatic search and retrieval of visual assets. It requires the `IMMICH_API_KEY` environment variable.

#### `searchPhotos(terms: SearchTerms, numPhotos: number): Promise<PhotoCandidate[]>`

Searches the Immich system for photos matching the given search terms.

*   **Description**: This asynchronous function queries the Immich API using the provided `SearchTerms`. It aims to find the most relevant photos that can accompany the generated content. The number of desired photos can be specified.
*   **Parameters**:
    *   `terms` (`SearchTerms`): An array of strings representing the keywords or phrases to search for in Immich. These are typically derived from resolved entities or the content's main topics.
    *   `numPhotos` (`number`): The maximum number of photo candidates to retrieve.
*   **Returns**: `Promise<PhotoCandidate[]>` – A promise that resolves to an array of `PhotoCandidate` objects, each representing a potential image asset found in Immich.
*   **Throws**: `Error` if the `IMMICH_API_KEY` is missing, invalid, or if the Immich service is unreachable.
*   **Usage Example**:

    ```typescript
    import { immichClient, SearchTerms } from '@bigmuddy/content-creation';

    async function findContentImages(entityNames: string[]) {
        const searchTerms: SearchTerms = entityNames.map(name => `${name} store front`);
        try {
            const photos = await immichClient.searchPhotos(searchTerms, 3);
            console.log('Found photos:', photos);
            // Example output: [{ id: 'img001', url: 'https://immich.example.com/assets/img001.jpg', description: 'BigMuddy Coffee shop' }]
        } catch (error) {
            console.error('Error searching photos:', error);
        }
    }

    findContentImages(['BigMuddy Coffee']);
    ```
