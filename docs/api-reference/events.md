# @bigmuddy/events API Reference

The `@bigmuddy/events` module provides a robust, multi-tenant, and opinionated framework for building event-driven architectures in Node.js applications. It leverages a persistent message queue (typically [PgBoss](https://github.com/timgit/pg-boss)) to ensure at-least-once delivery of events, supports flexible event handling, and enforces strict multi-tenant isolation.

This document details the core concepts, types, and API functions provided by the module.

---

## Core Concepts

### At-Least-Once Delivery

The `@bigmuddy/events` module guarantees **at-least-once delivery** for all events to all registered handlers. This means that once an event is published, the system ensures it will be delivered to each intended handler at least one time.

**Implications:**

*   **Potential Duplicates**: Due to network failures, crashes, or retries, a handler might receive the same event multiple times. This is an inherent trade-off for ensuring an event is never lost.
*   **Idempotency Requirement**: Because of potential duplicates, all event handlers **must be idempotent**. This is a critical design principle for applications using this module.

### Idempotency Requirement on Handlers

An operation is **idempotent** if applying it multiple times has the same effect as applying it once. For `@bigmuddy/events` handlers, this means:

> Given the same `EventDelivery` input, executing a handler's `handle` method multiple times must result in the same final state of the system as executing it exactly once.

**Why it's Crucial:**

As discussed in "At-Least-Once Delivery," handlers may receive duplicate events. If a handler is not idempotent, processing a duplicate event could lead to incorrect application state (e.g., double-charging a customer, creating duplicate records).

**Strategies for Achieving Idempotency:**

1.  **Check for Existence Before Creation**: Before creating a new record, check if it already exists based on a unique identifier (often derived from the event data or the event's `id`).
    ```typescript
    async handle(delivery: EventDelivery): Promise<DeliveryStatus> {
        const { event } = delivery;
        const orderId = event.data.orderId;
        const idempotencyKey = `order-processed:${orderId}`; // Use event ID or data for key

        // Check if this operation has already been performed
        if (await this.processedOperationsRepo.exists(idempotencyKey)) {
            console.log(`Handler '${this.name}' already processed event ${event.id} for order ${orderId}. Skipping.`);
            return DeliveryStatus.SUCCESS; // Already processed, so success
        }

        // Perform the actual work
        await this.orderService.processOrder(orderId, event.data);

        // Mark operation as processed
        await this.processedOperationsRepo.record(idempotencyKey);
        return DeliveryStatus.SUCCESS;
    }
    ```
2.  **Conditional Updates**: Update records using conditional statements (e.g., `UPDATE ... WHERE version = X` or `UPDATE ... WHERE status = 'pending'`).
3.  **Unique Constraints**: Leverage database unique constraints. If an insert fails due to a unique constraint violation, it's often safe to assume the record already exists and the operation effectively succeeded.
4.  **Transaction IDs/Idempotency Keys**: Pass a unique idempotency key (often the `BusEvent.id`) with operations to downstream services that support idempotency.
5.  **`isIdempotent` Flag**: The `EventHandler` interface includes an optional `isIdempotent` flag. While the system *assumes* all handlers are idempotent, setting this flag to `false` can be used for debugging or specific scenarios where a handler genuinely cannot be made idempotent (and thus accepts the risk of side effects from duplicates). **It is strongly recommended to always ensure handlers are idempotent.**

### Multi-Tenant Isolation Contract

The `@bigmuddy/events` module provides built-in support for multi-tenant applications, ensuring strict data isolation between tenants.

**TenantScope:**

Events and handlers are categorized by their `TenantScope`:

*   `TenantScope.SINGLE_TENANT`: An event originating from a specific tenant (`BusEvent.tenantId` is present) and intended to be processed by handlers *only for that tenant*. Most application-specific events fall into this category.
*   `TenantScope.MULTI_TENANT`: An event that is relevant to *multiple tenants* or requires processing across tenant boundaries. The `BusEvent.tenantId` might still be present (indicating the originating tenant) but the handler is designed to operate on data from various tenants.
*   `TenantScope.GLOBAL`: An event that is not associated with any specific tenant (e.g., system-level events, background jobs not tied to user data). `BusEvent.tenantId` will be `undefined` or `null`.

**The Isolation Contract (`isAllowedDelivery`):**

The core of the multi-tenant isolation is the `isAllowedDelivery` function, which enforces the following rules:

1.  **Single-Tenant Event to Single-Tenant Handler**: Allowed *only if* the `event.tenantId` matches the `handler.tenantId`. If `handler.tenantId` is not specified, it is assumed the handler is "global" in nature and can handle any single-tenant event (this should be used with extreme caution and only if the handler truly does not access tenant-specific data without explicit scoping).
2.  **Single-Tenant Event to Multi-Tenant Handler**: Allowed. The handler is expected to manage tenant context internally.
3.  **Single-Tenant Event to Global Handler**: Allowed.
4.  **Multi-Tenant Event to Single-Tenant Handler**: **Not Allowed**. A handler designed for a single tenant should not process an event that spans multiple tenants.
5.  **Multi-Tenant Event to Multi-Tenant Handler**: Allowed.
6.  **Multi-Tenant Event to Global Handler**: Allowed.
7.  **Global Event to Single-Tenant Handler**: **Not Allowed**. A global event has no tenant context, so it cannot be processed by a tenant-specific handler.
8.  **Global Event to Multi-Tenant Handler**: Allowed.
9.  **Global Event to Global Handler**: Allowed.

This contract ensures that tenant-specific data is never accidentally exposed or processed by handlers not designed for that tenant's context. Developers must correctly define the `tenantScope` for both events (via `PublishInput`) and handlers (`RegisterInput`) to leverage this protection.

## Types

This section defines the core data structures used throughout the `@bigmuddy/events` module.

### `BusEvent`

Represents an event that is published to the bus. This is the canonical form of an event, containing essential metadata.

```typescript
interface BusEvent<TData = any, TMetadata = any> {
  /**
   * A unique identifier for this event. Generated by the system upon publication.
   */
  id: string;
  /**
   * The name of the event (e.g., 'user.created', 'order.placed').
   */
  name: EventType;
  /**
   * The actual payload of the event.
   */
  data: TData;
  /**
   * Additional metadata associated with the event, such as correlation IDs, source system, etc.
   */
  metadata: TMetadata;
  /**
   * The identifier of the tenant associated with this event. Undefined for global events.
   */
  tenantId?: string;
  /**
   * The scope of the event regarding multi-tenancy. Defaults to SINGLE_TENANT if tenantId is present, GLOBAL otherwise.
   */
  tenantScope: TenantScope;
  /**
   * Timestamp when the event was published (ISO 8601 string).
   */
  publishedAt: string;
}
```

### `EventHandler`

Represents a registered handler for a specific event type. Handlers encapsulate the logic for processing events.

```typescript
interface EventHandler<TData = any, TMetadata = any> {
  /**
   * A unique name for this handler. Used for registration and identification.
   */
  name: string;
  /**
   * The function that contains the business logic to process the event.
   * It must return a Promise resolving to a DeliveryStatus.
   */
  handle: HandlerFn<TData, TMetadata>;
  /**
   * Indicates if this handler is idempotent. While all handlers are *expected* to be idempotent,
   * this flag can be used for explicit declaration or debugging. Defaults to true.
   */
  isIdempotent?: boolean;
  /**
   * The tenant scope of this handler. Defaults to SINGLE_TENANT.
   */
  tenantScope?: TenantScope;
  /**
   * The specific tenant ID this handler is bound to, if tenantScope is SINGLE_TENANT.
   * This is typically used for handlers that only process events for a single, specific tenant.
   */
  tenantId?: string;
}
```

### `EventDelivery`

The structure of an event as it is delivered to a handler. It includes the original event plus delivery-specific metadata.

```typescript
interface EventDelivery<TData = any, TMetadata = any> {
  /**
   * The original BusEvent being delivered.
   */
  event: BusEvent<TData, TMetadata>;
  /**
   * The current attempt number for delivering this event to this handler.
   * Starts at 1.
   */
  deliveryAttempt: number;
  /**
   * Contextual information available during delivery, such as the handler name.
   */
  context: HandlerContext;
}
```

### `DeliveryStatus`

An enumeration representing the outcome of an event handler's execution.

```typescript
enum DeliveryStatus {
  /**
   * The event was successfully processed by the handler.
   */
  SUCCESS = 'SUCCESS',
  /**
   * The event handler failed to process the event and it should be retried.
   */
  FAILURE = 'FAILURE',
  /**
   * The event handler failed and the event should not be retried.
   * It will be moved to a dead-letter queue if configured.
   */
  DEAD_LETTER = 'DEAD_LETTER',
  /**
   * The event was not delivered to the handler due to multi-tenant isolation rules.
   * This is an internal status and typically means the event is dropped for this specific handler,
   * but not necessarily for others.
   */
  ISOLATION_VIOLATION = 'ISOLATION_VIOLATION',
}
```

### `TenantScope`

An enumeration defining the multi-tenancy scope for events and handlers.

```typescript
enum TenantScope {
  /**
   * The event/handler is specific to a single tenant. `tenantId` is expected.
   */
  SINGLE_TENANT = 'SINGLE_TENANT',
  /**
   * The event/handler deals with data or operations across multiple tenants.
   */
  MULTI_TENANT = 'MULTI_TENANT',
  /**
   * The event/handler is global and not associated with any specific tenant.
   * `tenantId` should be undefined.
   */
  GLOBAL = 'GLOBAL',
}
```

### `EventType`

A string literal type representing the name of an event. This allows for type-safe event naming.

```typescript
type EventType = string; // In practice, often a union of specific string literals: 'user.created' | 'order.placed'
```

### `PublishInput`

The input structure for publishing a new event.

```typescript
interface PublishInput<TData = any, TMetadata = any> {
  /**
   * The name of the event (e.g., 'user.created').
   */
  name: EventType;
  /**
   * The payload of the event.
   */
  data: TData;
  /**
   * Optional metadata to attach to the event.
   */
  metadata?: TMetadata;
  /**
   * The ID of the tenant associated with this event. Required for SINGLE_TENANT scope.
   */
  tenantId?: string;
  /**
   * The tenant scope of this event. Defaults to SINGLE_TENANT if tenantId is present, GLOBAL otherwise.
   */
  tenantScope?: TenantScope;
  /**
   * Optional delay in milliseconds before the event should be delivered.
   */
  delay?: number;
}
```

### `RegisterInput`

The input structure for registering a new event handler.

```typescript
interface RegisterInput<TData = any, TMetadata = any> {
  /**
   * The name of the event this handler subscribes to.
   */
  eventName: EventType;
  /**
   * The handler object containing the name, handle function, and optional flags.
   */
  handler: EventHandler<TData, TMetadata>;
}
```

### `HandlerFn`

The signature for the `handle` method within an `EventHandler`.

```typescript
type HandlerFn<TData = any, TMetadata = any> = (
  delivery: EventDelivery<TData, TMetadata>
) => Promise<DeliveryStatus>;
```

### `HandlerContext`

Contextual information passed to the `HandlerFn` during event delivery.

```typescript
interface HandlerContext {
  /**
   * The name of the handler currently processing the event.
   */
  handlerName: string;
  /**
   * The event name this handler is registered for.
   */
  eventName: EventType;
  /**
   * The scope of the handler regarding multi-tenancy.
   */
  handlerTenantScope: TenantScope;
  /**
   * The specific tenant ID the handler is bound to, if applicable.
   */
  handlerTenantId?: string;
}
```

## API Reference

### `bus` Module

The `bus` module manages the registration and retrieval of event handlers within the application's memory. It acts as an in-memory registry.

#### `getHandler`

Retrieves a specific handler for a given event name and handler name.

```typescript
function getHandler(eventName: EventType, handlerName: string): EventHandler | undefined;
```

*   **Parameters:**
    *   `eventName`: The name of the event.
    *   `handlerName`: The unique name of the handler.
*   **Returns:**
    *   The `EventHandler` object if found, otherwise `undefined`.
*   **Description:**
    Useful for inspecting registered handlers or for internal dispatching logic.

#### `setHandler`

Registers or updates an event handler for a specific event type. This function is typically used internally by `register` and `registerMany`.

```typescript
function setHandler(eventName: EventType, handler: EventHandler): void;
```

*   **Parameters:**
    *   `eventName`: The name of the event the handler subscribes to.
    *   `handler`: The `EventHandler` object to register.
*   **Returns:** `void`
*   **Description:**
    Adds or replaces a handler in the in-memory registry. If a handler with the same `eventName` and `handler.name` already exists, it will be overwritten.

#### `hasHandler`

Checks if a specific handler is registered for a given event name.

```typescript
function hasHandler(eventName: EventType, handlerName: string): boolean;
```

*   **Parameters:**
    *   `eventName`: The name of the event.
    *   `handlerName`: The unique name of the handler.
*   **Returns:**
    *   `true` if the handler is registered, `false` otherwise.
*   **Description:**
    Provides a way to verify handler registration.

#### `listHandlerNames`

Lists the names of all handlers registered for a specific event type.

```typescript
function listHandlerNames(eventName: EventType): string[];
```

*   **Parameters:**
    *   `eventName`: The name of the event.
*   **Returns:**
    *   An array of strings, where each string is the `name` of a registered `EventHandler` for the specified `eventName`. Returns an empty array if no handlers are found.
*   **Description:**
    Useful for introspection and understanding which handlers are subscribed to a particular event.

### `publish` Module

The `publish` module provides the primary interface for emitting events into the system.

#### `publish`

Publishes one or more events to the event bus. This function serializes the event(s) and enqueues them for delivery via the underlying message queue (e.g., PgBoss).

```typescript
function publish(input: PublishInput | PublishInput[]): Promise<void>;
```

*   **Parameters:**
    *   `input`: A single `PublishInput` object or an array of `PublishInput` objects.
*   **Returns:**
    *   A `Promise<void>` that resolves when the event(s) have been successfully enqueued in the message queue. It does *not* wait for the event(s) to be processed by handlers.
*   **Description:**
    This is the entry point for all events into the system. When an event is published:
    1.  A unique `id` is generated for the `BusEvent`.
    2.  `publishedAt` timestamp is recorded.
    3.  `tenantScope` is automatically determined if not provided (defaults to `SINGLE_TENANT` if `tenantId` is present, `GLOBAL` otherwise).
    4.  For each handler registered for the `event.name`, a delivery job is enqueued in the message queue. The `isAllowedDelivery` function is used to determine if a delivery should be enqueued based on the multi-tenant isolation contract.
    5.  The job payload includes the `BusEvent` and information about the target handler.
    If `delay` is specified in `PublishInput`, the event delivery will be scheduled for a future time.

*   **Example:**

    ```typescript
    import { publish, PublishInput, TenantScope } from '@bigmuddy/events';

    interface UserCreatedEventData {
      userId: string;
      email: string;
      name: string;
    }

    async function createUser(userId: string, email: string, name: string, tenantId: string) {
      // ... create user in database ...

      const event: PublishInput<UserCreatedEventData> = {
        name: 'user.created',
        data: { userId, email, name },
        tenantId: tenantId,
        metadata: { source: 'api' },
        delay: 5000 // Deliver after 5 seconds
      };

      await publish(event);
      console.log(`User created event published for user ${userId} in tenant ${tenantId}.`);
    }

    // Publishing multiple events
    async function updateOrder(orderId: string, status: string, tenantId: string) {
        // ... update order in database ...
        const event1: PublishInput = {
            name: 'order.updated',
            data: { orderId, status },
            tenantId,
        };
        const event2: PublishInput = {
            name: 'order.status.changed',
            data: { orderId, newStatus: status },
            tenantId,
            tenantScope: TenantScope.SINGLE_TENANT, // Explicitly declare scope
        };
        await publish([event1, event2]);
        console.log(`Order ${orderId} updated and status changed events published.`);
    }
    ```

### `register` Module

The `register` module is responsible for declaring event handlers to the system.

#### `register`

Registers a single event handler for a specific event type.

```typescript
function register<TData = any, TMetadata = any>(input: RegisterInput<TData, TMetadata>): void;
```

*   **Parameters:**
    *   `input`: A `RegisterInput` object containing the `eventName` and the `EventHandler` to register.
*   **Returns:** `void`
*   **Description:**
    This function adds the handler to the in-memory registry managed by the `bus` module. It also ensures that the handler's `tenantScope` defaults correctly if not provided (to `SINGLE_TENANT` if `handler.tenantId` is present, otherwise `GLOBAL`).
    Handlers must have a unique `name` within the scope of an `eventName`. Registering a handler with the same `eventName` and `handler.name` will overwrite the previous one.
    It is crucial to register all handlers during application startup, *before* starting any event processing workers.

*   **Example:**

    ```typescript
    import { register, HandlerFn, DeliveryStatus, TenantScope } from '@bigmuddy/events';

    interface UserCreatedEventData {
      userId: string;
      email: string;
      name: string;
    }

    const sendWelcomeEmailHandler: HandlerFn<UserCreatedEventData> = async ({ event, deliveryAttempt }) => {
      console.log(`Attempt ${deliveryAttempt}: Sending welcome email to ${event.data.email} for user ${event.data.userId} (tenant: ${event.tenantId}).`);
      try {
        // Simulate sending email
        if (Math.random() < 0.1 && deliveryAttempt < 3) { // Simulate transient failure
            throw new Error('Email service temporarily unavailable');
        }
        // await emailService.sendWelcomeEmail(event.data.email, event.data.name);
        console.log(`Welcome email sent to ${event.data.email}.`);
        return DeliveryStatus.SUCCESS;
      } catch (error) {
        console.error(`Failed to send welcome email for ${event.data.email}: ${error.message}`);
        // Decide whether to retry or dead-letter
        return deliveryAttempt < 5 ? DeliveryStatus.FAILURE : DeliveryStatus.DEAD_LETTER;
      }
    };

    register<UserCreatedEventData>({
      eventName: 'user.created',
      handler: {
        name: 'SendWelcomeEmail',
        handle: sendWelcomeEmailHandler,
        tenantScope: TenantScope.SINGLE_TENANT, // This handler operates on a single tenant's data
      },
    });

    console.log("Handler 'SendWelcomeEmail' registered for 'user.created' event.");
    ```

#### `registerMany`

Registers multiple event handlers simultaneously.

```typescript
function registerMany(inputs: RegisterInput[]): void;
```

*   **Parameters:**
    *   `inputs`: An array of `RegisterInput` objects.
*   **Returns:** `void`
*   **Description:**
    A convenience function for registering a collection of handlers. It internally calls `register` for each input.

*   **Example:**

    ```typescript
    import { registerMany, HandlerFn, DeliveryStatus } from '@bigmuddy/events';

    const auditLogHandler: HandlerFn = async ({ event }) => {
      console.log(`[AUDIT] Event ${event.name} (ID: ${event.id}) processed by AuditLogHandler.`);
      // await auditService.log(event);
      return DeliveryStatus.SUCCESS;
    };

    const analyticsHandler: HandlerFn = async ({ event }) => {
      console.log(`[ANALYTICS] Event ${event.name} (ID: ${event.id}) sent to analytics.`);
      // await analyticsService.track(event);
      return DeliveryStatus.SUCCESS;
    };

    registerMany([
      {
        eventName: 'user.created',
        handler: { name: 'AuditLogHandler', handle: auditLogHandler, tenantScope: 'MULTI_TENANT' },
      },
      {
        eventName: 'user.created',
        handler: { name: 'AnalyticsHandler', handle: analyticsHandler, tenantScope: 'GLOBAL' },
      },
      {
        eventName: 'order.placed',
        handler: { name: 'AuditLogHandler', handle: auditLogHandler, tenantScope: 'MULTI_TENANT' },
      },
    ]);

    console.log("Multiple handlers registered for various events.");
    ```

### `dispatcher` Module

The `dispatcher` module contains functions for processing event deliveries and managing dead letters. These functions are typically used by the underlying message queue worker.

#### `processBatch`

Processes a batch of `EventDelivery` objects. This is the core logic that executes handlers.

```typescript
function processBatch(deliveries: EventDelivery[]): Promise<DeliveryStatus[]>;
```

*   **Parameters:**
    *   `deliveries`: An array of `EventDelivery` objects received from the message queue.
*   **Returns:**
    *   A `Promise<DeliveryStatus[]>` that resolves to an array of `DeliveryStatus` outcomes, corresponding to each input `EventDelivery` in the same order.
*   **Description:**
    For each `EventDelivery` in the batch:
    1.  It retrieves the corresponding `EventHandler` from the in-memory bus.
    2.  It applies the `isAllowedDelivery` multi-tenant isolation check. If the delivery is not allowed, `DeliveryStatus.ISOLATION_VIOLATION` is returned.
    3.  If allowed, it executes the handler's `handle` method.
    4.  It captures the `DeliveryStatus` returned by the handler.
    This function is designed to be called by the message queue's worker process (e.g., PgBoss's job processor). It handles potential errors during handler execution, ensuring a `DeliveryStatus` is always returned for each delivery.

#### `replayDeadLetter`

Attempts to re-enqueue a specific dead-lettered event for a given handler.

```typescript
function replayDeadLetter(eventId: string, handlerName: string): Promise<boolean>;
```

*   **Parameters:**
    *   `eventId`: The ID of the original `BusEvent` that was dead-lettered.
    *   `handlerName`: The name of the handler for which the event was dead-lettered.
*   **Returns:**
    *   A `Promise<boolean>` resolving to `true` if the dead-letter was successfully re-enqueued, `false` otherwise (e.g., if the event/handler combination wasn't found in dead letters or couldn't be re-enqueued).
*   **Description:**
    When an event reaches a `DeliveryStatus.DEAD_LETTER` state, it is typically moved to a separate dead-letter queue by the message queue system. This function allows an administrator or an automated process to manually re-attempt delivery of a specific dead-lettered event to its intended handler. This is useful after an issue causing the dead-letter has been resolved.

### `isolation` Module

The `isolation` module provides the core logic for enforcing multi-tenant isolation rules.

#### `isAllowedDelivery`

Determines whether a given `BusEvent` is allowed to be delivered to a specific `EventHandler` based on their `tenantScope` and `tenantId`.

```typescript
function isAllowedDelivery(event: BusEvent, handler: EventHandler): boolean;
```

*   **Parameters:**
    *   `event`: The `BusEvent` being considered for delivery.
    *   `handler`: The `EventHandler` that is meant to process the event.
*   **Returns:**
    *   `true` if the delivery is permitted by the multi-tenant isolation contract, `false` otherwise.
*   **Description:**
    This function implements the "Multi-Tenant Isolation Contract" described in the Core Concepts section. It is a critical security and data integrity component, preventing tenant-specific data from being processed by unintended handlers. This function is called internally by `publish` (when enqueuing jobs) and `dispatcher.processBatch` (as a final check before handler execution).

*   **Example (internal logic):**

    ```typescript
    import { BusEvent, EventHandler, TenantScope, isAllowedDelivery } from '@bigmuddy/events';

    // Example 1: Single-tenant event to single-tenant handler (matching tenantId) -> ALLOWED
    const event1: BusEvent = { id: 'e1', name: 'user
