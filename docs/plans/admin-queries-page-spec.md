# Design Document: Admin Queries Page

## 1. Introduction

This document outlines the design and implementation plan for a new administrative page, `/admin/queries`, aimed at empowering non-technical team members (e.g., Tracy, Amy, Chase) to independently retrieve critical data from our Sanity CMS and Postgres database. Currently, these stakeholders often rely on engineering resources for ad-hoc data requests, leading to potential delays and diverting developer time from core product work.

The "Admin Queries" page will provide a user-friendly interface with pre-defined, read-only queries, allowing users to select a data source, choose a specific query, provide necessary parameters, and instantly view or export the results. This initiative is expected to significantly reduce operational friction, improve data accessibility, and free up engineering bandwidth.

## 2. Goals

*   **Self-Service Data Access:** Enable non-technical administrators to access frequently requested data without writing code.
*   **Operational Efficiency:** Reduce the number of ad-hoc data requests handled by the engineering team.
*   **Unified Interface:** Provide a single page to query data from both Sanity (GROQ) and Postgres (Prisma).
*   **Security & Safety:** Ensure all executed queries are read-only, parameters are validated, and access is strictly admin-gated and rate-limited.
*   **Data Export:** Facilitate easy export of query results in JSON and CSV formats.
*   **User-Friendly Experience:** Design an intuitive and clear user interface.

## 3. Non-Goals

*   **Ad-hoc Query Writing:** Users will *not* be able to write arbitrary GROQ or Prisma queries. All queries will be pre-defined and curated by engineering.
*   **Data Modification:** The page will exclusively support read-only operations. No `create`, `update`, `delete` functionality will be exposed.
*   **Complex BI/Reporting:** This page is not intended to replace dedicated business intelligence tools or provide complex analytical dashboards. Its purpose is raw data retrieval.
*   **Public Access:** This page and its API endpoint will be strictly for internal administrative use.

## 5. Data Source 1: Sanity (GROQ Queries)

### 5.1. Overview

Integration with Sanity will leverage the official `@sanity/client` library. All queries will be expressed using GROQ (Graph-Relational Object Queries), which is Sanity's powerful query language.

### 5.2. Query Structure

Each Sanity query definition will be an object with the following structure:

```typescript
// types/admin-queries.ts
export type QueryParameterType = 'string' | 'number' | 'boolean' | 'date';

export interface QueryParameter {
  name: string; // e.g., 'authorId', 'slug'
  label: string; // User-facing label, e.g., 'Author ID'
  type: QueryParameterType;
  description?: string; // Hint for the user
  optional?: boolean; // True if the parameter can be omitted
  defaultValue?: any; // Default value if optional and not provided
}

export interface SanityQueryDefinition {
  id: string; // Unique identifier
  name: string; // User-friendly name, e.g., 'List All Articles'
  description: string; // Detailed explanation of what the query does
  category: string; // For grouping in the UI, e.g., 'Articles', 'Staff'
  groq: string; // The actual GROQ query string
  parameters?: QueryParameter[]; // Optional array of parameters
}
```

These definitions will be stored in a dedicated backend file (e.g., `lib/admin-queries/sanity-queries.ts`) for easy management and validation.

3.  **Name:** `listFAQsByBrand`
    *   **Description:** Fetches FAQs associated with a particular brand ID.
    *   **Category:** FAQs
    *   **Parameters:** `[{ name: 'brandId', label: 'Brand ID', type: 'string', description: 'Sanity ID of the brand', optional: false }]`

4.  **Name:** `listActiveStaff`
    *   **Description:** Returns details (ID, name, email, role) for all staff members marked as active.

5.  **Name:** `listUpcomingShowEvents`
    *   **Description:** Lists events scheduled for a future date, including title, date/time, and location.

6.  **Name:** `listAllCategories`
    *   **Description:** Provides a list of all defined content categories with their ID, title, and slug.
    *   **Category:** Content Structure
    *   **GROQ:** `*[_type == "category"]{_id, title, "slug": slug.current}`
7.  **Name:** `listProductsInStock`
    *   **Description:** Displays products that are currently marked as in stock, including their name and price.
    *   **Category:** Products

8.  **Name:** `listRecentComments`
    *   **Description:** Shows the 100 most recent comments, ordered by creation date, including the comment text and author's name.
    *   **Category:** User Generated Content

9.  **Name:** `getArticleBySlug`
    *   **Description:** Retrieves a single article by its URL slug, including its full content.

    *   **Parameters:** `[{ name: 'slug', label: 'Article Slug', type: 'string', description: 'The URL slug of the article (e.g., "my-great-article")', optional: false }]`
    *   **GROQ:** `*[_type == "article" && slug.current == $slug][0]{_id, title, content, publishedAt}`
10. **Name:** `listAllPages`
    *   **Description:** Lists all generic pages in the CMS with their title and slug.
    *   **Category:** Content Structure
    *   **GROQ:** `*[_type == "page"]{_id, title, "slug": slug.current, _updatedAt}`
11. **Name:** `listRedirects`
    *   **Description:** Shows all configured redirects (from path, to path, and type).
    *   **Category:** Site Management
    *   **GROQ:** `*[_type == "redirect"]{_id, from, to, type, _createdAt}`
12. **Name:** `listAssetsWithoutAltText`
    *   **Description:** Identifies image assets that are missing `altText`, which is crucial for accessibility.
    *   **Category:** Asset Management
    *   **GROQ:** `*[_type == "sanity.imageAsset" && !defined(altText)]{_id, originalFilename, url, size}`
13. **Name:** `listUntranslatedContent`
    *   **Description:** Finds articles that do not have a Spanish translation defined. (Assumes a `translations.es` field).
    *   **Category:** Localization

14. **Name:** `listAllAuthors`

    *   **Description:** Retrieves a list of all newsletter subscribers with their email and subscription date.

## 6. Data Source 2: Postgres (Prisma Queries)

### 6.1. Overview

For Postgres, we will use Prisma ORM to interact with the database. This provides type-safety and a robust interface, mitigating the risks associated with raw SQL.

### 6.2. Query Structure

Similar to Sanity, each Postgres query will have a structured definition. To ensure safety and prevent arbitrary Prisma method calls, we will define a `PrismaExecutor` function for each query. This function will encapsulate the specific Prisma call and its arguments.

```typescript
// types/admin-queries.ts (continued)
import { PrismaClient } from '@prisma/client';

export type PrismaExecutor = (
  prisma: PrismaClient,
  params: Record<string, any>
) => Promise<any>;

export interface PostgresQueryDefinition {
  id: string; // Unique identifier
  name: string; // User-friendly name, e.g., 'List Active Subscriptions'
  description: string; // Detailed explanation of what the query does
  category: string; // For grouping in the UI, e.g., 'Subscriptions', 'Users'
  parameters?: QueryParameter[]; // Optional array of parameters
  executor: PrismaExecutor; // Function to execute the Prisma query safely
}
```

These definitions will also reside in a dedicated backend file (e.g., `lib/admin-queries/postgres-queries.ts`).
