# MBT Incident Response Runbook

This document outlines the standard operating procedures for responding to incidents affecting MBT's production systems. The goal is to provide a clear, calm, and prescriptive guide to minimize downtime, reduce impact, and ensure effective communication.

---

## 1. Severity Levels

Incident severity is classified based on impact to customers, revenue, and data. This classification dictates the urgency and resources allocated to resolution.

### SEV-1: Critical - Revenue Stopped / Data Loss Possible

*   **Definition:** The service is completely unavailable, core functionality is broken for all users, or there is an immediate risk of data loss or corruption. This directly impacts revenue generation or the integrity of user data.
*   **Impact:** Widespread, immediate, and severe. Core business operations are halted or severely compromised.
*   **Examples:**
    *   Stripe payment processing is completely down, preventing all new subscriptions or purchases.
    *   The primary database is unresponsive or corrupted, leading to 5xx errors across the application.
    *   The application is returning 5xx errors for all user requests, rendering it unusable.
    *   User data is being actively deleted or corrupted.
*   **Urgency:** **Immediate action required.** All available resources must focus on resolution 24/7 until the incident is mitigated.

### SEV-2: Major - Partial Outage / Customer-Impacting

*   **Definition:** A significant portion of users or a major feature is affected, leading to degraded service. Core functionality is impaired but not entirely broken, or a critical background process is failing.
*   **Impact:** Significant, affecting a subset of users or a key feature, potentially leading to customer dissatisfaction or churn.
*   **Examples:**
    *   Slow page loads or timeouts for a noticeable percentage of users.
    *   A critical integration (e.g., Immich for photo search) is offline, rendering a specific feature unusable.
    *   Certain user flows are failing (e.g., profile updates, specific content uploads).
    *   A major scheduled job (e.g., Buzzsprout sync) is consistently failing, impacting content delivery.
    *   A significant UI component is broken, making a part of the application difficult to use.
*   **Urgency:** **High priority.** Resolution expected during business hours, with potential for after-hours work if the impact is severe or prolonged. Proactive communication to affected users may be necessary.

### SEV-3: Minor - Degraded / Internal-Impacting

*   **Definition:** Minor impact on user experience or significant impact on internal operations. No direct customer impact or revenue loss, but productivity or monitoring is affected.
*   **Impact:** Moderate, affecting internal teams or a small, non-critical aspect of the user experience.
*   **Examples:**
    *   The admin panel is slow or partially broken, affecting internal management tasks.
    *   Internal reporting tools are malfunctioning.
    *   Non-critical background jobs are failing, but without immediate user impact (e.g., analytics processing).
    *   Minor UI glitches that don't prevent core functionality.
    *   A monitoring alert system is not functioning correctly.
*   **Urgency:** **Medium priority.** Address within a few days or schedule for the next available development sprint.

### SEV-4: Cosmetic - Low Impact

*   **Definition:** A minor visual or functional bug with no impact on core functionality, user experience, or internal operations.
*   **Impact:** Low, typically unnoticed by most users or easily workaround.
*   **Examples:**
    *   A typo on a non-critical page.
    *   A broken image on a static page.
    *   Minor UI alignment issues.
    *   A console warning in the browser that doesn't affect functionality.
*   **Urgency:** **Low priority.** Address at convenience, typically batched with other minor improvements or fixes.

## 2. First 5 Minutes: Assess and Secure

The initial moments of an incident are crucial for understanding the situation and preventing further damage.

1.  **Acknowledge the Alert / Report:**
    *   If an alert fires, acknowledge it immediately.
    *   If a user or team member reports an issue, confirm receipt and initiate investigation.

2.  **Confirm Scope and Impact:**
    *   **What is broken?** Try to reproduce the issue yourself.
    *   **Who is affected?** All users, a subset, internal users?
    *   **How widespread is it?** Is it isolated or global?
    *   Check recent monitoring alerts (if any) for clues.
    *   Look at the most recent deployments – could a new deploy be the cause?

3.  **Mute Auto-Deploys (if active):**
    *   **Action:** Immediately pause any automated deployments to production. This prevents new, potentially broken code from being pushed, or existing issues from being overwritten with another bad deploy.
    *   **How to:** On Vercel, navigate to your project settings. You can typically pause or disable automatic deployments triggered by Git pushes. For critical incidents, it's safer to disable them entirely until the incident is resolved.
    *   **Why:** A new deploy, even if seemingly unrelated, can exacerbate an existing issue or introduce new ones, making diagnosis harder.

4.  **Communicate to Key Stakeholders:**
    *   **Action:** Send an immediate, concise notification to Chase and Tracy.
    *   **Method:**
        *   **Primary:** Slack (e.g., `#incidents` channel or direct message).
        *   **Secondary (if no Slack response):** Phone call/text.
    *   **Content:**
        *   **What:** Brief description of the problem (e.g., "Stripe payments failing").
        *   **Severity:** Initial SEV level assessment (e.g., "SEV-1").
        *   **Impact:** Who/what is affected (e.g., "All users unable to subscribe").
        *   **Current Status:** "Investigating," "Muted deploys."
        *   **Next Steps:** "Will provide update in 5-10 mins."
    *   **Example Message:**
        ```
        @Chase @Tracy Incident: Stripe payments failing (SEV-1).
        Impact: All users unable to complete purchases/subscriptions.
        Status: Investigating. Auto-deploys muted.
        Will provide update in 5 minutes.
        ```

## 3. First 15 Minutes: Initial Diagnosis

With the immediate threat contained and initial communication sent, the next step is rapid diagnosis using our primary tools.

1.  **Vercel Dashboard & Logs:**
    *   **Action:** This is your first stop for application-level issues.
    *   **How to:** Navigate to the Vercel project dashboard.
        *   **Deployments Tab:** Check the status of the most recent deployments. Did a deployment fail? Is the current production deployment healthy? If a recent deploy is suspected, consider a rollback to the previous stable version (see "Vercel Deploy Gone Bad" in SEV-1 scenarios).
        *   **Logs Tab:** Filter by severity (Error, Warning), time range (last 5, 15, 30 minutes), and specific endpoints if known.
            *   **Look for:**
                *   `5xx` errors (Server errors).
                *   `4xx` errors (Client errors, but could indicate a backend issue if widespread).
                *   Specific error messages related to database connections, API calls, or unhandled exceptions.
                *   High latency warnings (`cold start`, `duration`).
                *   Errors related to external services (Stripe, Sanity, etc.).
            *   **Pro Tip:** Look for patterns. Is it one specific endpoint? All endpoints? Is it happening consistently or intermittently?

2.  **Postgres Connection State (Neon / Supabase / Vercel Managed):**
    *   **Action:** Database issues are a common root cause for application-wide failures.
    *   **How to:** Access your database provider's console (e.g., Neon dashboard, Supabase dashboard, or Vercel's integrated Postgres console).
        *   **Check Connection Pool:** Is the database maxing out its connection limit? Look for "too many connections" errors in Vercel logs.
        *   **Active Queries:** Are there long-running or blocked queries? Identify queries consuming excessive resources.
        *   **CPU/Memory Usage:** Is the database instance under heavy load?
        *   **Replication Status:** If applicable, ensure read replicas are healthy and in sync.
    *   **Why:** Connection saturation or slow queries can bring the entire application to a crawl or make it unresponsive.

3.  **Stripe Dashboard:**
    *   **Action:** If the incident involves payments, subscriptions, or webhooks, Stripe is critical.
    *   **How to:** Log in to your Stripe dashboard.
        *   **Developers > Webhooks:** Check for any failed webhook attempts. A backlog here can indicate an issue with our webhook endpoint.
        *   **Payments:** Review recent payment attempts. Are payments failing? What are the error codes?
        *   **Balances/Transfers:** Check for any anomalies in funds flow.
        *   **API Logs:** Look for errors in calls made *from* our application to Stripe.
    *   **Why:** Stripe provides detailed insights into payment processing and can quickly pinpoint issues related to financial transactions.

4.  **Cloudflare Status:**
    *   **Action:** Cloudflare is our CDN and DNS provider. Outages here can impact accessibility.
    *   **How to:** Visit [https://www.cloudflarestatus.com/](https://www.cloudflarestatus.com/).
        *   **Check for:** Active incidents or widespread outages.
        *   **Review our settings:** Briefly check DNS records or WAF rules if you suspect they might be interfering.
    *   **Why:** If Cloudflare is experiencing issues, our site might be unreachable or certain assets might not load, regardless of our backend health.

## 4. Common SEV-1 Scenarios + Immediate Response

These are critical, high-impact situations requiring immediate, decisive action.

### Scenario: Stripe Webhook Backlog

*   **Symptoms:**
    *   Stripe Dashboard shows a growing number of failed webhook attempts.
    *   Vercel logs show errors for the `/api/stripe/webhook` endpoint (e.g., 500 errors, timeouts).
    *   User actions dependent on webhook processing (e.g., subscription activations, invoice updates) are delayed or not happening.
*   **Diagnosis:**
    1.  Go to [Stripe Dashboard > Developers > Webhooks](https://dashboard.stripe.com/test/webhooks).
    2.  Observe the "Attempts" column for your primary webhook endpoint. A high number of recent failures or pending retries indicates a problem.
    3.  Cross-reference with Vercel logs for the `/api/stripe/webhook` route to identify the specific error our server is returning.
*   **Immediate Response:**
    1.  **Identify Root Cause:** Is our server crashing on webhook receipt? Is it a database connection issue? An external API call failing?
    2.  **Temporarily Disable Webhook Processing (if severe):** If the backlog is causing resource exhaustion on our server, consider temporarily disabling the webhook endpoint (e.g., by returning a quick `200 OK` from the route without processing, or by using a Cloudflare WAF rule to block specific Stripe IPs, or by removing the webhook endpoint from Stripe). **Caution:** This will delay event processing further but can stabilize the server. Re-enable as soon as possible.
    3.  **Clear Backlog (if possible):** Once the underlying server issue is fixed, Stripe will automatically retry webhooks. If you need to force a retry for specific events, you can do so from the Stripe dashboard for individual events.
    4.  **Scale Resources:** If the issue is simply high load, consider scaling up Vercel serverless functions or database resources.

### Scenario: Database Connection Saturation

*   **Symptoms:**
    *   Vercel logs are flooded with "too many connections," "connection refused," or "database unavailable" errors.
    *   Application endpoints are returning 5xx errors or timing out.
    *   The entire application feels slow or unresponsive.
*   **Diagnosis:**
    1.  Access your Postgres console (Neon/Supabase/Vercel).
    2.  Check the "Connections" or "Active Queries" section. Look for the current connection count nearing or exceeding the maximum limit.
    3.  Identify any long-running or "idle in transaction" queries that are holding connections open.
    4.  Review database CPU/memory usage for spikes.
*   **Immediate Response:**
    1.  **Identify Long-Running Queries:** If specific queries are the culprit, try to identify the application code path that generates them.
    2.  **Kill Problematic Queries (with extreme caution):** If a query is clearly stuck and causing widespread issues, you might need to manually terminate it from the database console. **Only do this if you understand the implications and potential for data inconsistency.**
    3.  **Scale Database:** Immediately increase the connection limit or upgrade the database instance to a higher tier if available and necessary.
    4.  **Restart Application (last resort):** A full application restart might clear stale connections, but it's a temporary fix if the underlying query issue persists.
    5.  **Hotfix:** Prioritize deploying a hotfix to optimize or disable the problematic query/feature.

### Scenario: Sanity Outage

*   **Symptoms:**
    *   Content from Sanity (e.g., blog posts, product descriptions) is missing or appears outdated.
    *   Vercel logs show errors when trying to fetch data from Sanity's API.
    *   The application might display fallback content or error messages where Sanity data should be.
*   **Diagnosis:**
    1.  Go to [https://status.sanity.io/](https://status.sanity.io/).
    2.  Check for any active incidents or maintenance related to their API or CDN.
*   **Immediate Response:**
    1.  **Communicate:** Inform Chase and Tracy about the external dependency outage.
    2.  **Fallback to Cached Data:** If your application has a caching layer for Sanity data, ensure it's serving the last known good data.
    3.  **Graceful Degradation:** Ensure the application handles missing Sanity data gracefully (e.g., displaying "Content unavailable" instead of crashing).
    4.  **Monitor Sanity Status:** Regularly check their status page for updates.
    5.  **Prepare for Re-sync:** Once Sanity is back online, monitor for data consistency and initiate any necessary re-syncs if your cache is stale.

### Scenario: Cloudflare Outage

*   **Symptoms:**
    *   The entire website is unreachable or loads very slowly.
    *   Users report "DNS resolution error" or "connection timed out."
    *   Specific assets (images, CSS, JS) might not load if served via Cloudflare CDN.
*   **Diagnosis:**
    1.  Go to [https://www.cloudflarestatus.com/](https://www.cloudflarestatus.com/).
    2.  Check for any active incidents or widespread outages affecting their DNS, CDN, or proxy services.
*   **Immediate Response:**
    1.  **Communicate:** Inform Chase and Tracy about the external dependency outage.
    2.  **No Direct Fix:** As this is an external outage, there's no immediate fix on our side.
    3.  **Monitor Cloudflare Status:** Keep a close eye on their status page for resolution.
    4.  **Prepare for DNS Changes (extreme case):** In very rare, prolonged outages, you might consider temporarily changing DNS records away from Cloudflare, but this is a complex and risky maneuver and should only be considered with full team agreement.

### Scenario: Vercel Deploy Gone Bad

*   **Symptoms:**
    *   A recent deployment (especially if automated) immediately leads to widespread 5xx errors or broken functionality.
    *   Vercel logs for the new deployment show critical errors during build or runtime.
    *   The application is completely unusable or core features are failing.
*   **Diagnosis:**
    1.  Go to [Vercel Dashboard > Deployments](https://vercel.com/dashboard).
    2.  Identify the most recent deployment. Check its status (failed, errored).
    3.  Click into the deployment to view its build logs and runtime logs. Look for obvious errors or misconfigurations.
    4.  Compare the current deployment with the previous successful one.
*   **Immediate Response:**
    1.  **Revert to Previous Deployment:** This is the fastest way to restore service.
        *   On Vercel, navigate to the `Deployments` tab.
        *   Find the last known **successful** deployment before the incident started.
        *   Click the "..." menu next to it and select "Redeploy to Production" or "Promote to Production."
        *   **Confirm:** This will immediately switch your production alias to point to the stable deployment.
    2.  **Disable Auto-Deploys:** Ensure auto-deploys are still muted to prevent the bad code from being redeployed.
    3.  **Investigate Root Cause:** Once service is restored, analyze the problematic deployment's code changes, environment variables, or build process to understand why it failed. Prepare a hotfix branch if necessary.

## 5. Common SEV-2 Scenarios

These incidents are significant but might allow for a slightly longer diagnosis period or might not require an immediate all-hands-on-deck response outside of business hours unless impact escalates.

### Scenario: Slow Page Load

*   **Symptoms:**
    *   Users report the website feels "sluggish."
    *   Browser developer tools show long network requests or slow asset loading.
    *   Vercel logs show high `duration` times for certain serverless functions.
*   **Diagnosis:**
    1.  **Browser Dev Tools:** Use Chrome DevTools (Network tab) to identify which requests are slow (API calls, asset loading).
    2.  **Vercel Logs:** Filter logs by specific endpoints or function names. Look for functions with consistently high execution times. Check for `cold start` times.
    3.  **Database Performance:** Check Postgres console for slow queries, high CPU usage, or connection contention (as per SEV-1 scenario).
    4.  **External APIs:** Are any third-party API calls taking a long time?
    5.  **CDN/Cache:** Is Cloudflare or Vercel's Edge caching working effectively?
*   **Response:**
    1.  **Identify Bottleneck:** Pinpoint the exact component causing the slowdown (frontend rendering, specific API endpoint, database query, external service).
    2.  **Optimize:**
        *   **Frontend:** Code splitting, lazy loading, image optimization, reducing render-blocking resources.
        *   **Backend:** Optimizing serverless functions, reducing external API calls, improving database queries (indexing, reducing N+1 queries).
        *   **Database:** Optimize slow queries, add indexes.
        *   **Caching:** Implement or improve caching strategies (CDN, in-memory, database query caching).
    3.  **Scale:** If it's purely a capacity issue, consider scaling up Vercel function concurrency or database resources.

### Scenario: Immich Offline (Wizard Photo Search Degraded)

*   **Symptoms:**
    *   The "Wizard Photo Search" feature (or any feature relying on Immich) is not working or returning errors.
    *   Vercel logs show errors when making API calls to the Immich service.
    *   Users report inability to upload or search photos.
*   **Diagnosis:**
    1.  **Check Immich Server:** If Immich is self-hosted, log in to the server where it's running. Check if the Immich service is running.
    2.  **Immich Logs:** Review Immich's own logs for errors.
    3.  **API Connectivity:** From the Vercel environment, try to `curl` the Immich API endpoint to check basic network connectivity.
    4.  **Credentials/Configuration:** Verify API keys or environment variables used to connect to Immich.
*   **Response:**
    1.  **Restart Immich:** Often, a simple restart of the Immich service can resolve transient issues.
    2.  **Troubleshoot Immich Server:** If restart doesn't work, investigate logs on the Immich server for disk space, memory, or specific application errors.
    3.  **Communicate Degradation:** Inform relevant stakeholders (Chase, Tracy, Amy) that the photo search functionality is degraded.
    4.  **Hotfix/Workaround:** If Immich remains offline, consider a temporary hotfix to disable the feature gracefully or provide an explanatory message to users.
