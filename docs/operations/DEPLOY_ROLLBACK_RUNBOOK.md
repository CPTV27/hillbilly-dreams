### 2. Pre-Deploy Checklist

Before merging any code to `main` (and thus triggering a production deploy), the following checklist *must* be completed. This applies to the code on the feature branch that is about to be merged.

1.  **TypeScript Check (Local):**
    *   **Command:** `pnpm exec tsc --noEmit -p apps/web/tsconfig.json`
    *   **Purpose:** Ensures that all TypeScript code compiles without errors. Vercel's build process will fail if there are type errors, but catching them locally *before* pushing saves time and build resources. This check validates type safety and consistency across the `apps/web` project.
    *   **Action:** Resolve any reported TypeScript errors or warnings.

2.  **Prisma Migration Plan Review:**
    *   **Context:** If the PR includes changes to the `prisma/schema.prisma` file, a new database migration will be required.
    *   **Process:**
        1.  **Generate Migration (Local, if not already done):** Run `npx prisma migrate dev --create-only` in your local development environment. This generates the migration files without applying them.
        2.  **Review Generated SQL:** Carefully inspect the SQL generated in the `prisma/migrations` directory. Look for:
            *   Accidental drops of tables or columns.
            *   Unintended data type changes.
            *   Missing `NOT NULL` constraints where expected.
            *   Any operation that could lead to data loss or unexpected behavior.
        3.  **Discuss Breaking Changes:** If the migration involves breaking changes (e.g., column renames, type changes that lose data), ensure these follow the "Expand-Contract Pattern" (see Section 6) and are explicitly documented in the PR description and discussed with the team.
    *   **Action:** Confirm the migration SQL is safe and aligns with the intended schema changes.

3.  **Environment Variables Verification:**
    *   **Context:** If the PR introduces new environment variables or modifies existing ones, they *must* be configured in Vercel.
    *   **Process:**
        1.  Navigate to the [Vercel Project Settings](https://vercel.com/dashboard/<org>/<project>/settings/environment-variables).
        2.  Verify that all new or modified environment variables are correctly set for:
            *   `Production` environment.
            *   `Preview` environment (for future PRs).
            *   `Development` environment (if applicable, for Vercel functions).
        3.  Pay close attention to values, especially for sensitive data (API keys, secrets) and ensure no trailing whitespace (see Section 7).
    *   **Action:** Add/update necessary environment variables in Vercel.

4.  **Breaking Schema Notes (if applicable):**
    *   **Context:** If the PR *must* introduce a breaking schema change (e.g., renaming a critical column, changing a data type that loses precision), ensure the team is aware and the deployment strategy accounts for this.
    *   **Action:** Explicitly note any breaking schema changes in the PR description, link to the expand-contract strategy documentation (if used), and confirm team alignment. **Avoid merging breaking schema changes without a phased deployment plan.**

### 3. Vercel Preview Deploys

Every Pull Request (PR) opened against the `main` branch automatically triggers a Vercel Preview Deployment. This is a critical step for quality assurance before merging.

**Process:**

1.  **PR Opened:** When a new PR is created, Vercel starts a new deployment for that specific branch.
2.  **Access Preview URL:** Once the preview deployment is "Ready," Vercel posts a link to the preview URL directly in the GitHub PR comments.
3.  **Inspect Preview Behavior:**
    *   **Manual Testing:** Thoroughly test all affected features and user flows on the preview URL.
    *   **Visual Regression:** Check for any unintended UI changes or regressions.
    *   **Console Errors:** Open the browser's developer console and look for JavaScript errors or network request failures.
    *   **Network Tab:** Verify API calls are returning expected data and status codes.
    *   **Specific Routes:** Test any new routes or modified pages directly.
    *   **Performance:** While not a full performance test, observe general responsiveness.
    *   **Vercel Logs:** Check the Vercel dashboard for the preview deployment's logs for any server-side errors.
4.  **When NOT to Merge Based on Preview Signals:**
    *   Any CI check failure (lint, tests, TS).
    *   Any unexpected visual regressions or broken UI elements.
    *   Critical functionality is broken or behaves unexpectedly.
    *   New errors appear in the browser console or Vercel deployment logs.
    *   Performance significantly degrades.
    *   Unresolved feedback from code reviewers.

**Action:** Only merge a PR to `main` if the preview deployment is stable, functional, and meets all quality criteria.

### 4. Production Promotion

When a PR is merged into the `main` branch, Vercel initiates a production deployment. This process replaces the previously deployed version of the application with the new one.

**Process:**

1.  **Merge to `main`:** The merged code becomes the new production candidate.
2.  **Vercel Build & Deploy:** Vercel builds the application and deploys it to the production alias (e.g., `mbt-app.com`). This is typically a zero-downtime deployment, meaning traffic is seamlessly switched to the new version once it's ready.
3.  **Vercel Cron & In-Flight Events:**
    *   **Vercel Cron Jobs:** Any Vercel Cron jobs configured for the project will now execute against the newly deployed production code.
    *   **Implications for In-Flight Events:** This is a critical consideration. If your application has long-running background tasks, webhooks, or user sessions that span across deployments, there's a potential for issues:
        *   **Schema Mismatch:** If the new deployment includes a database migration that changes a schema, and a cron job or background task from the *old* deployment is still processing data based on the *old* schema, it could lead to errors or data corruption.
        *   **API Changes:** If API endpoints or internal service contracts change, an in-flight process from the old version might make requests to the new version that are no longer compatible.
        *   **State Inconsistency:** User sessions might encounter issues if their server-side state is incompatible with the new application version.

    *   **Mitigation:**
        *   **Idempotency:** Design cron jobs and background tasks to be idempotent, meaning they can be run multiple times without causing unintended side effects.
        *   **Backward Compatibility:** Strive for backward compatibility in API changes and data structures, especially during phased rollouts.
        *   **Graceful Shutdowns:** While Vercel handles traffic switching, consider how your application handles in-flight work if a process is abruptly terminated (e.g., using message queues for background tasks).
        *   **Monitoring:** Closely monitor logs and error reporting systems immediately after a production deploy, especially for cron jobs.

**Action:** Be aware that Vercel Cron jobs will immediately pick up the new code. Plan database migrations and API changes carefully to minimize impact on in-flight processes.

### 5. Rollback

Vercel provides a straightforward "one-click" rollback mechanism, but it's crucial to understand its limitations, especially concerning external state.

**Procedure:**

1.  **Identify Problem:** A critical bug or issue is discovered in production after a recent deploy.
2.  **Navigate to Vercel Dashboard:** Go to the [Vercel Dashboard](https://vercel.com/dashboard) for the MBT project.
3.  **Select Deployments:** Click on the "Deployments" tab.
4.  **Choose Previous Deployment:** Locate the last *stable* deployment (the one before the problematic one).
5.  **Initiate Rollback:** Click the "Rollback to Production" button (or similar "Promote to Production" button next to an older deployment).
6.  **Confirm:** Confirm the rollback action. Vercel will immediately promote the chosen older deployment to production, switching traffic to it.
7.  **Verify Rollback:** After the rollback is complete, re-verify the live URLs (as per Section 1) to ensure the previous stable version is now live and the issue is resolved.

**What Vercel Rollback DOESN'T Undo:**

A Vercel rollback only reverts the *application code* deployed to the edge network. It **does NOT** undo external state changes:

*   **Database Migrations:** Vercel rollback *cannot* revert database schema changes. If a new deployment included a `prisma migrate deploy` that altered your database, rolling back the application code will *not* revert the database schema. This is the most critical limitation.
*   **Stripe State:** Payments processed, subscriptions created/canceled, or customer data updated in Stripe will remain as they are.
*   **Sent Emails:** Transactional emails (e.g., order confirmations, password resets) or marketing emails sent by the problematic deployment cannot be unsent.
*   **External API Calls:** Any side effects caused by API calls made by the problematic deployment (e.g., creating a user in an external service, posting to a social media API) are permanent.
*   **Cached Data:** While Vercel handles its own caching, external caches (e.g., Redis, CDN caches beyond Vercel's scope) might still hold stale data.

**When NOT to Rollback:**

*   **Database Migration Incompatibility:** If the issue in the new deployment is due to a database migration that introduced breaking changes (e.g., renamed a column that the old code *cannot* read from), rolling back the application code will likely cause the *old* code to crash because it's trying to access a non-existent or incompatible database schema. In such cases, a hotfix (see Section 9) that addresses the code's interaction with the new schema is often safer than a rollback.
*   **Irreversible External State:** If the problematic deployment caused irreversible changes to critical external systems (e.g., mass deletion of data in a third-party service), rolling back the code won't undo those effects, and might even exacerbate them if the old code tries to operate on an inconsistent state.

**Action:** Only rollback if the issue is purely application-code related and does not involve irreversible external state changes or database schema incompatibilities that the old code cannot handle.

### 6. Database Migration Safety

Database migrations are the most sensitive part of any deployment. Incorrect migrations can lead to data loss or application downtime.

**`prisma migrate deploy` vs `prisma migrate dev`:**

*   **`prisma migrate dev`:**
    *   **Purpose:** Used for local development only. It creates new migrations, applies them, and can even reset the database and re-seed it. It's designed for rapid iteration and schema changes during development.
    *   **Usage:** `npx prisma migrate dev --name <migration-name>`
    *   **NEVER use in Production.**

*   **`prisma migrate deploy`:**
    *   **Purpose:** Used for applying *existing* migration files to a production or staging database. It only applies pending migrations and does not generate new ones or reset the database. It's designed for idempotent, non-interactive application of migrations in CI/CD environments.
    *   **Usage:** `npx prisma migrate deploy`
    *   **Execution:** This command is typically run as part of your Vercel build process (e.g., in a `postbuild` script or a separate Vercel Build Command if using a custom build step for migrations). **Important:** Ensure it runs *before* the application starts serving traffic.

**The "Never Roll Back a Migration in Prod" Rule:**

This is a cardinal rule in database management: **Once a database migration has been applied to a production database, you should generally never "roll it back" by reverting the migration.**

*   **Why?**
    *   **Data Loss:** Rolling back a migration that, for example, dropped a column or changed a data type, could lead to irreversible data loss.
    *   **Schema Drift:** Manually reverting SQL or using `prisma migrate reset` in production is extremely dangerous and can lead to an inconsistent state between your application's expected schema and the actual database schema.
    *   **Application Incompatibility:** If your application code (even a rolled-back version) expects a certain schema, and the database has been reverted to an older schema, your application will crash.

*   **Instead of Rolling Back a Migration:**
    *   **Forward Fix:** If a migration caused an issue, create a *new* migration to fix it (e.g., re-add a dropped column, correct a data type). This is safer as it maintains a linear history of schema changes.
    *   **Hotfix:** Deploy an application hotfix that is compatible with the *current* (problematic) database schema (see Section 9).

**The Expand-Contract Pattern for Breaking Schema Changes:**

When a database schema change is "breaking" (e.g., renaming a column, changing a column's type in a way that requires data transformation), a multi-phase deployment strategy is required to ensure zero downtime and prevent data loss.

**Example: Renaming a column `oldColumn` to `newColumn`**

1.  **Phase 1: Expand (Add new, keep old)**
    *   **Database Migration:** Add the `newColumn` to the database schema. **Do NOT remove `oldColumn` yet.**
        *   `ALTER TABLE MyTable ADD COLUMN newColumn TEXT;`
    *   **Application Code:** Update the application to:
        *   **Write to both `oldColumn` and `newColumn`** (if data needs to be duplicated).
        *   **Read from `oldColumn`**.
        *   (Optional) Backfill data from `oldColumn` to `newColumn` in a separate process.
    *   **Deploy Application (Version A):** Deploy the application changes.
    *   **Verify:** Ensure both columns are present and the application continues to function.

2.  **Phase 2: Migrate (Switch to new)**
    *   **Application Code:** Update the application to:
        *   **Write *only* to `newColumn`**.
        *   **Read from `newColumn`**.
        *   (Optional) Remove writing to `oldColumn`.
    *   **Deploy Application (Version B):** Deploy the application changes.
    *   **Verify:** Ensure the application is now fully using `newColumn`. The `oldColumn` is now deprecated but still exists.

3.  **Phase 3: Contract (Remove old)**
    *   **Database Migration:** Remove the `oldColumn` from the database schema.
        *   `ALTER TABLE MyTable DROP COLUMN oldColumn;`
    *   **Deploy Application (Version C - no code changes related to column):** A new application deployment is good practice to ensure no stale code paths accidentally refer to the old column.
    *   **Verify:** Ensure the application continues to function correctly without the `oldColumn`.

**Action:** Always follow the Expand-Contract pattern for breaking schema changes. Review all migration SQL carefully. Never manually revert a production migration.

### 7. Common Deploy Failures + Fixes

This section outlines frequently encountered deployment issues on Vercel and their typical resolutions.

1.  **Sanity Environment Variable Trailing Whitespace:**
    *   **Problem:** Vercel's environment variable parsing can be sensitive. If you copy-paste an environment variable value from a text editor or console, you might accidentally include a trailing whitespace character. While often invisible, this can lead to connection failures (e.g., to Sanity, database, or other APIs) because the API key or URL is slightly incorrect.
    *   **Detection:** Vercel logs will show connection errors, authentication failures, or malformed URLs related to the specific service.
    *   **Fix:**
        1.  Go to [Vercel Project Settings](https://vercel.com/dashboard/<org>/<project>/settings/environment-variables).
        2.  Carefully edit the problematic environment variable.
        3.  Ensure there are absolutely no leading or trailing whitespace characters in the value.
        4.  Save and redeploy.

2.  **ESLint Plugin Missing / Linting Errors during Build:**
    *   **Problem:** Vercel's build process runs linting checks. If a newly added ESLint plugin or configuration is missing from `package.json` or `pnpm-lock.yaml`, or if new code violates existing lint rules, the build will fail.
    *   **Detection:** Vercel build logs will explicitly show ESLint errors like `Module not found: Can't resolve 'eslint-plugin-...'` or a list of linting violations.
    *   **Fix:**
        1.  **Missing Plugin:** Add the missing ESLint plugin as a `devDependency`: `pnpm add -D <eslint-plugin-name>`.
        2.  **Linting Errors:** Fix the reported linting errors in the code.
        3.  Commit changes and push.

3.  **Prisma Client Out of Sync / Missing Generated Client:**
    *   **Problem:** After making changes to `prisma/schema.prisma`, the Prisma Client needs to be re-generated for your application code to recognize the new schema. If `npx prisma generate` is not run locally and the generated client isn't committed (or if Vercel's build command doesn't include it), the build will fail because the application code refers to non-existent types or fields.
    *   **Detection:** Vercel build logs will show TypeScript errors related to Prisma types (e.g., `Property 'myNewField' does not exist on type 'MyModel'`) or runtime errors indicating the Prisma Client is not found.
    *   **Fix:**
        1.  Locally, after any `schema.prisma` changes, run: `npx prisma generate`.
        2.  Ensure the generated client (typically in `node_modules/.prisma/client`) is either committed (if `node_modules` is not gitignored) or, more commonly, ensure your Vercel build command explicitly includes `npx prisma generate` *before* the `next build` step. A common Vercel build command is `pnpm prisma generate && pnpm build`.
        3.  Commit and push.

4.  **Vercel Build Memory Exhaustion:**
    *   **Problem:** For large Next.js applications, especially those with many pages, complex build-time data fetching, or extensive static asset processing, the default Vercel build memory limit (typically 1GB) can be exceeded.
    *   **Detection:** Vercel build logs will show an error message like `Build exceeded memory limit` or similar.
    *   **Fix:**
        1.  **Optimize Build:**
            *   Reduce the number of pages being built at once (e.g., use `getStaticPaths` with `fallback: 'blocking'` or `true` for dynamic routes).
            *   Optimize images (smaller sizes, WebP format).
            *   Lazy load components and dynamic imports.
            *   Ensure `node_modules` is properly cached between builds.
        2.  **Increase Build Memory:**
            *   Go to [Vercel Project Settings](https://vercel.com/dashboard/<org>/<project>/settings).
            *   Under "General" or "Build & Development Settings," locate "Build Memory."
            *   Increase the memory limit (e.g., to 2GB or 4GB, depending on your plan).
        3.  Redeploy.

5.  **Environment Variable Typo:**
    *   **Problem:** A common mistake is a typo in an environment variable name, either in the code (`process.env.MY_VAR`) or in Vercel's configuration (`MY_VAR_NAME`). This leads to the variable being `undefined` at runtime.
    *   **Detection:**
        *   Vercel build logs might show `undefined` values if the variable is used during build time.
        *   Runtime errors in Vercel logs or browser console (if it's a client-side env var) indicating a missing value (e.g., `Cannot read properties of undefined (reading 'length')` when trying to use a missing string).
        *   **Debugging:** Temporarily add `console.log(process.env.MY_VAR_NAME)` in your code (
