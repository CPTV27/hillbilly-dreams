# Stripe Webhook Events Reference

The MBT Commerce module relies heavily on Stripe webhooks to maintain synchronization between Stripe's billing and payment states and the module's internal database. This document provides a comprehensive catalog of all Stripe webhook events handled by the module, detailing their triggers, the module's responses, idempotency considerations, and failure modes.

## Webhook Security and Configuration

Stripe webhooks are a critical component for real-time updates. To ensure the integrity and authenticity of these events, the MBT Commerce module implements stringent security measures and requires specific configuration.

### Webhook Signature Verification

Every webhook event sent by Stripe includes a `Stripe-Signature` header. This signature is crucial for verifying that the event originated from Stripe and has not been tampered with.

**Verification Flow:**

1.  **Receive Webhook Request:** The MBT Commerce webhook endpoint (`/api/commerce/webhook`) receives a `POST` request from Stripe.
2.  **Extract Signature:** The `Stripe-Signature` header is extracted from the incoming request. This header contains a timestamp and one or more signatures.
3.  **Retrieve Secret:** The module retrieves the webhook signing secret from its environment variables (specifically, `COMMERCE_WEBHOOK_SECRET`).
4.  **Construct Event:** Using the Stripe Node.js library (`stripe.webhooks.constructEvent`), the raw request body, the `Stripe-Signature` header, and the `COMMERCE_WEBHOOK_SECRET`, the module attempts to construct a `Stripe.Event` object.
5.  **Validation:**
    *   If the signature is valid, the `constructEvent` method returns a `Stripe.Event` object, and the event processing proceeds.
    *   If the signature is invalid (e.g., due to a tampered payload or an incorrect secret), `constructEvent` throws an error. The module then rejects the request, preventing unauthorized or malicious payloads from being processed.

**Importance:**
Signature verification protects against spoofed events and ensures that only legitimate Stripe events influence the module's state. Without it, an attacker could send fabricated events to grant free access, alter subscription statuses, or disrupt services.

### `COMMERCE_WEBHOOK_SECRET` Environment Variable

This environment variable stores the unique signing secret provided by Stripe for your webhook endpoint.

*   **Purpose:** Used in the webhook signature verification process to authenticate incoming webhook events.
*   **Obtaining the Secret:**
    1.  Navigate to the Stripe Dashboard.
    2.  Go to **Developers > Webhooks**.
    3.  Select your webhook endpoint (or create a new one).
    4.  Click **Reveal** next to "Signing secret."
    5.  Copy the secret (e.g., `whsec_...`) and set it as the value for `COMMERCE_WEBHOOK_SECRET` in your deployment environment.
*   **Security:** This secret should be treated like a password. Do not hardcode it in your codebase or commit it to version control. Ensure it's securely managed as an environment variable.

## Stripe's Webhook Retry Policy

Stripe is designed to be resilient. If the MBT Commerce module's webhook endpoint fails to acknowledge an event (e.g., returns a non-2xx HTTP status code, times out), Stripe will automatically retry sending the event.

*   **Mechanism:** Stripe uses an exponential backoff schedule for retries. This means that the time between retries increases with each subsequent attempt.
*   **Duration:** Stripe will continue to retry delivering the event for up to **3 days**.
*   **Implications for MBT Commerce:**
    *   **Idempotency is Crucial:** Because events can be retried, handlers must be idempotent. Processing the same event multiple times should yield the same result without causing unintended side effects (e.g., duplicate records, double charges, or incorrect state transitions).
    *   **Robust Error Handling:** The module should aim to process events quickly and return a `200 OK` status code as soon as possible, even if subsequent asynchronous tasks are queued. This signals to Stripe that the event was received, minimizing retries.

## Local Testing with Stripe CLI

Developing and debugging webhook handlers locally can be challenging due to network accessibility. The Stripe CLI provides a convenient way to forward webhook events from your Stripe account to your local development environment.

**Command:**

```bash
stripe listen --forward-to localhost:3000/api/commerce/webhook
```

*   **`stripe listen`:** This command initiates a listener that captures webhook events from your Stripe account.
*   **`--forward-to localhost:3000/api/commerce/webhook`:** This flag specifies the local URL where captured events should be forwarded. Replace `3000` with your local server's port and `/api/commerce/webhook` with the exact path of your webhook handler.

**How it works:**
The Stripe CLI establishes a secure tunnel, allowing Stripe to send real or test webhook events directly to your local machine. This enables you to test your event handlers, observe their behavior, and debug issues without deploying your code to a public server.

### `checkout.session.completed`

**When Stripe fires it:**
This event is fired when a customer successfully completes a Stripe Checkout Session. This can be for a one-time payment or a new subscription purchase.

**What the Commerce module does:**
The module inspects the `mode` of the `checkout.session` object to determine the appropriate action.

*   **If `session.mode` is `'subscription'` (Subscription Purchase):**
    *   **Prisma Table Updates:**
        *   **`Subscription`:** A new `Subscription` record is created or an existing one is updated. Fields populated include `customerId` (linked to `User`), `stripeSubscriptionId` (from `session.subscription`), `status` (e.g., `'active'`, `'trialing'`), `currentPeriodStart`, `currentPeriodEnd`, `cancelAtPeriodEnd` (if applicable), `priceId`, `quantity`, and `trialEnd`.
        *   **`Customer`:** If a new Stripe Customer was created, or an existing one was used, the associated `Customer` record (linked to `User`) is confirmed/updated.
    *   **Downstream Side Effects:**
        *   **User Access:** The associated `User` is granted access to the purchased subscription features/tier.
        *   **Email Notification:** A welcome email or subscription confirmation email is sent to the user.
        *   **Onboarding:** Triggers any initial onboarding flows specific to the new subscription.

*   **If `session.mode` is `'payment'` (One-Time Payment):**
    *   **Prisma Table Updates:**
        *   **`Payment`:** A new `Payment` record is created. Fields populated include `customerId`, `stripePaymentIntentId` (from `session.payment_intent`), `amount`, `currency`, and `status` (`'succeeded'`). This `Payment` record is typically linked to an `Order` record created earlier.
        *   **`Order`:** The associated `Order` record's `status` is updated to `'paid'`.
    *   **Downstream Side Effects:**
        *   **Fulfillment:** Triggers the fulfillment process for digital goods, access to specific content, or initiates physical product shipping.
        *   **Email Notification:** An order confirmation and receipt email is sent to the user.

**Idempotency notes:**
The handler checks for the existence of a `Subscription` or `Payment` record using the `checkout_session_id` or the `subscription_id`/`payment_intent_id` contained within the event.
*   If a record already exists with the corresponding Stripe ID, the handler will update it if necessary (e.g., confirm status) but will not create a duplicate.
*   If no record exists, a new one is created.
This ensures that retried events do not result in duplicate subscriptions, payments, or access grants.

**Failure mode if the handler errors:**
*   **Subscription Mode:** The `Subscription` record might not be created or updated, leading to the user not receiving access to the subscribed features. The user might have paid but remains in an unprovisioned state. This requires manual review and reconciliation.
*   **Payment Mode:** The `Payment` record might not be created, or the `Order` status might not be updated to `'paid'`. This prevents fulfillment, leaving the user without their purchased item and the order in an unfulfilled state. Manual intervention is required to mark the order as paid and trigger fulfillment.

### `customer.subscription.created`

**When Stripe fires it:**
This event is fired when a new subscription is created for a customer in Stripe. This can happen through a Checkout Session (as covered by `checkout.session.completed`), directly via the Stripe API, or through the Stripe Dashboard.

**What the Commerce module does:**
*   **Prisma Table Updates:**
    *   **`Subscription`:** A new `Subscription` record is created. Fields populated include `customerId` (linked to `User`), `stripeSubscriptionId` (from the event object), `status` (e.g., `'active'`, `'trialing'`), `currentPeriodStart`, `currentPeriodEnd`, `cancelAtPeriodEnd` (if applicable), `priceId`, `quantity`, and `trialEnd`.
*   **Downstream Side Effects:**
    *   **User Access:** The associated `User` is provisioned with access to the subscribed features/tier.
    *   **Email Notification:** A welcome email or subscription confirmation email is sent to the user.

**Idempotency notes:**
The handler checks for an existing `Subscription` record using the `stripe_subscription_id` from the event.
*   If a record with this ID already exists, it is updated to reflect the latest state (e.g., ensuring `status` is correct).
*   If no record exists, a new `Subscription` record is created.
This prevents duplicate subscription entries in the database if the event is retried.

**Failure mode if the handler errors:**
The `Subscription` record might not be created in the local database. This leads to the user not being provisioned with access, despite having a valid subscription in Stripe. Manual reconciliation is required to create the `Subscription` record and grant access.

### `customer.subscription.updated`

**When Stripe fires it:**
This event is fired whenever a subscription undergoes any change. This includes, but is not limited to:
*   Subscription status changes (e.g., `active`, `past_due`, `unpaid`, `canceled`, `ended`).
*   Plan changes (upgrades/downgrades).
*   Quantity changes.
*   Trial period ending (transition from `trialing` to `active`).
*   `cancel_at_period_end` flag being set or unset.
*   Payment method changes affecting the subscription.

**What the Commerce module does:**
*   **Prisma Table Updates:**
    *   **`Subscription`:** The existing `Subscription` record matching the `stripe_subscription_id` is updated with the latest information from the Stripe event. Key fields updated include: `status`, `currentPeriodStart`, `currentPeriodEnd`, `cancelAtPeriodEnd`, `priceId`, `quantity`, `trialEnd`, and `endedAt` (if applicable).
*   **Downstream Side Effects:**
    *   **User Access Adjustment:**
        *   If `status` changes from `trialing` to `active`: Confirms full access.
        *   If `status` changes to `past_due` or `unpaid`: May trigger a grace period or send dunning emails.
        *   If `status` changes to `canceled` or `ended`: Triggers revocation of access to subscription-gated features.
        *   If `priceId` or `quantity` changes: May adjust user's feature limits or tier access.
    *   **Email Notifications:**
        *   Notification for plan changes (upgrade/downgrade).
        *   Dunning emails for `past_due` or `unpaid` statuses.

**Idempotency notes:**
The handler identifies the existing `Subscription` record using its `stripe_subscription_id` and applies the updates. Since the event provides the complete current state of the subscription, applying it multiple times will simply overwrite with the same data, ensuring idempotency. No new records are created.

**Failure mode if the handler errors:**
The local `Subscription` record may become out of sync with Stripe. This can lead to:
*   Incorrect user access (e.g., user retaining access after cancellation, or losing access prematurely).
*   Incorrect billing logic (e.g., dunning emails not sent for `past_due` subscriptions).
*   Inaccurate reporting and analytics.
Manual intervention is required to reconcile the local database with Stripe's state.

### `customer.subscription.deleted`

**When Stripe fires it:**
This event is fired when a subscription is fully cancelled and ends. This typically occurs at the end of the current billing period if `cancel_at_period_end` was set, or immediately if a subscription is cancelled without prorating.

**What the Commerce module does:**
*   **Prisma Table Updates:**
    *   **`Subscription`:** The existing `Subscription` record matching the `stripe_subscription_id` is updated. Its `status` is set to `'canceled'` or `'ended'`, and the `endedAt` timestamp is populated.
*   **Downstream Side Effects:**
    *   **User Access Revocation:** The associated `User` is immediately or period-end-ly (depending on the subscription's `cancel_at_period_end` status) stripped of access to subscription-gated features.
    *   **Email Notification:** A subscription cancellation confirmation email is sent to the user.

**Idempotency notes:**
The handler identifies the existing `Subscription` record using its `stripe_subscription_id`. If the record's `status` is already `'canceled'` or `'ended'`, no further action is taken. Otherwise, the status and `endedAt` are updated. This ensures that repeated events do not cause issues.

**Failure mode if the handler errors:**
The local `Subscription` record might not be marked as `'canceled'` or `'ended'`. This could lead to:
*   The user retaining access to subscription features indefinitely, despite the subscription being deleted in Stripe.
*   Inaccurate subscription metrics and reporting.
Manual intervention is required to update the `Subscription` status and revoke user access.

### `customer.subscription.trial_will_end`

**When Stripe fires it:**
This event is fired a configurable number of days (default 3 days) before a subscription's trial period is set to end.

**What the Commerce module does:**
*   **Prisma Table Updates:**
    *   No direct `Subscription` record updates, but metadata or a flag on the `Subscription` or `User` record might be updated to indicate that a trial-end notification has been sent.
*   **Downstream Side Effects:**
    *   **Email Notification:** An email is sent to the user, reminding them that their trial is ending soon and that their card will be charged for the first full billing period. It typically includes a call to action to update payment methods if needed or to cancel before the trial ends.

**Idempotency notes:**
The handler should check if a trial-end notification has already been sent for this specific subscription's trial period. This can be achieved by:
*   Storing a `lastTrialEndNotificationSentAt` timestamp on the `Subscription` record.
*   Storing a unique identifier for the notification event (e.g., a hash of subscription ID and trial end date) and checking against a sent-notifications log.
This prevents sending duplicate trial-end emails if the event is retried.

**Failure mode if the handler errors:**
The user will not receive the trial-end notification. This can lead to:
*   Users being surprised by a charge at the end of their trial.
*   Increased churn, as users might cancel due to unexpected charges rather than proactively managing their subscription.
*   Increased support inquiries.
While not critical for system state, it negatively impacts user experience.

### `invoice.created`

**When Stripe fires it:**
This event is fired when Stripe generates a new invoice. This happens for subscription renewals, one-off charges, or when an invoice is manually created.

**What the Commerce module does:**
*   **Prisma Table Updates:**
    *   **`Invoice`:** A new `Invoice` record is created in the local database. Fields populated include `customerId`, `stripeInvoiceId` (unique), `stripeSubscriptionId` (if applicable), `status` (e.g., `'draft'`, `'open'`), `amountDue`, `amountPaid`, `currency`, and `invoicePdf` URL.
*   **Downstream Side Effects:**
    *   **User Portal Display:** The invoice becomes available for viewing in the user's billing portal.
    *   **Email Notification:** (Optional, often handled by `invoice.upcoming` or `invoice.payment_succeeded` for user-facing emails) An internal notification might be sent to accounting for new invoices.

**Idempotency notes:**
The handler checks for an existing `Invoice` record using the `stripe_invoice_id`.
*   If a record already exists, it is updated (e.g., if the invoice status changes from `draft` to `open`).
*   If no record exists, a new `Invoice` record is created.
This prevents duplicate invoice entries.

**Failure mode if the handler errors:**
The `Invoice` record might not be created or updated in the local database. This leads to:
*   Users being unable to view their invoices in the billing portal.
*   Incomplete financial records within the module.
Manual reconciliation or re-triggering the event (if possible) is required.
