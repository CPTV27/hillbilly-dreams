Welcome to the **MBT (Multi-Tenant Boilerplate) Local Development Setup Guide**!

This guide will walk you through setting up your new MacBook to run the MBT platform locally. We'll cover everything from installing essential tools to running the application, managing your local database, and simulating real-world scenarios like Stripe webhooks and multi-tenant access.

Please follow the steps sequentially. If you encounter any issues, refer to the Troubleshooting section at the end.

Before we start, let's get all the necessary tools installed on your system. We'll primarily use Homebrew, the macOS package manager, for installations. If you don't have Homebrew installed, open your Terminal and run:


### 1.2 Node.js (v20 or higher) & pnpm

Node.js is the JavaScript runtime, and pnpm is our preferred package manager for its efficiency with monorepos and disk space. We recommend using a Node Version Manager (NVM) like `volta` or `nvm` to manage Node.js versions easily.

#### Option A (Recommended): Volta

Volta is a fast, reliable, and universal tool for managing your JavaScript tools. It ensures that the correct Node, npm, and pnpm versions are used automatically based on your project's `package.json`.

1.  **Install Volta:**

    ```bash
    curl https://get.volta.sh | bash
    ```

    Follow the on-screen instructions to set up your shell environment variables. You might need to restart your terminal.

2.  **Install Node.js v20 and pnpm via Volta:**

    ```bash
    volta install node@20 pnpm
    ```

    Volta will automatically use Node.js v20 and pnpm for projects that specify these versions.

3.  **Verify installations:**

    ```bash
    node -v
    pnpm -v
    ```

    You should see versions like `v20.x.x` and `8.x.x` respectively.

#### Option B (Alternative): NVM

NVM (Node Version Manager) is another popular choice.

1.  **Install NVM:**

    ```bash
    curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
    ```

    Follow the on-screen instructions to add NVM to your shell's profile. Restart your terminal.

2.  **Install Node.js v20:**

    ```bash
    nvm install 20
    nvm use 20
    nvm alias default 20 # Set v20 as the default
    ```

3.  **Install pnpm globally:**

    ```bash
    npm install -g pnpm
    ```

4.  **Verify installations:**

    ```bash
    node -v
    pnpm -v
    ```

## 3. Local Postgres Database

We highly recommend running a local Postgres database using Docker Compose for development. This provides a consistent and isolated database environment.

### 3.1 Recommended: Docker Compose Setup

1.  **Create `docker-compose.yml`:**
    In the root of your project, create a file named `docker-compose.yml` (if it doesn't already exist) and add the following content. This configuration sets up a PostgreSQL 16 container.

    ```yaml
    version: '3.8'

    services:
      postgres:
        image: postgres:16-alpine # Using a lightweight Alpine image
        restart: always
        environment:
          POSTGRES_DB: mbt_dev # Your database name
          POSTGRES_USER: postgres # Your database user
          POSTGRES_PASSWORD: mysecretpassword # Your database password
        ports:
          - "5432:5432" # Map container port 5432 to host port 5432
        volumes:
          - postgres_data:/var/lib/postgresql/data # Persist data

    volumes:
      postgres_data:
    ```

    **Note:** For a production environment, you would use stronger passwords and potentially more robust volume management. For local development, this is sufficient.

2.  **Start the Postgres container:**

    ```bash
    docker compose up -d postgres
    ```

    This command will download the Postgres image (if not already present), create and start the `postgres` service in detached mode (`-d`).

3.  **Verify the container is running:**

    ```bash
    docker ps
    ```

    You should see an entry for `postgres:16-alpine`.

4.  **Database Connection String:**
    Your local Postgres database connection string will be:

    ```
    postgresql://postgres:mysecretpassword@localhost:5432/mbt_dev
    ```

    You'll use this in your `.env.local` file.

### 3.2 Alternative: Vercel-managed Postgres (Development Branch)

While connecting directly to a remote Vercel-managed Postgres *development* database is possible, it's generally not recommended for day-to-day local development due to potential latency, rate limits, and the risk of affecting other developers or the dev environment.

If you absolutely need to, you would connect to it using the `DATABASE_URL` provided by Vercel. This URL is typically pulled down with `vercel env pull`, as described in the next section.

**Recommendation:** Stick with the local Docker Postgres for the best development experience.

## 4. Environment Variables

Environment variables are crucial for configuring your application. We'll pull these directly from Vercel.

1.  **Link your project to Vercel:**

    If you haven't already, link your local project to the Vercel project:

    ```bash
    vercel link
    ```

    Follow the prompts to select your Vercel scope and the correct project.

2.  **Pull environment variables:**

    This command fetches all environment variables from your Vercel project (typically from the `development` environment) and saves them into a local `.env.local` file.

    ```bash
    vercel env pull .env.local --yes
    ```

    The `--yes` flag automatically confirms the overwrite of any existing `.env.local` file.

3.  **Local Overrides:**

    After pulling, open the newly created `.env.local` file. You will likely need to override *at least* the `DATABASE_URL` to point to your local Docker Postgres instance.

    Modify your `.env.local` to include (or update) the `DATABASE_URL`:

    ```env
    # .env.local

    # ... other variables pulled from Vercel ...

    # Override for local Docker Postgres
    DATABASE_URL="postgresql://postgres:mysecretpassword@localhost:5432/mbt_dev"

    # You might also need to override these if your Sanity project is different for local dev
    # NEXT_PUBLIC_SANITY_PROJECT_ID="your_local_sanity_project_id"
    # NEXT_PUBLIC_SANITY_DATASET="your_local_sanity_dataset"
    # SANITY_API_TOKEN="your_local_sanity_api_token" # Only if Sanity requires authenticated access locally
    ```

    **Important:** The goal is to have as few local overrides as possible. `DATABASE_URL` is the most common one. Ensure `NEXT_PUBLIC_` variables are correctly set for your local Sanity instance if you're running a separate local Sanity project.

1.  **Generate Prisma Client:**

    This command reads your `schema.prisma` file and generates the Prisma Client, which allows your application code to interact with the database in a type-safe manner.

    ```bash
    pnpm exec prisma generate
    ```

2.  **Apply Database Migrations:**

    This command will apply any pending database migrations to your local Postgres database. It will also create the database if it doesn't exist and synchronize your schema.

    ```bash
    pnpm exec prisma migrate dev
    ```

    You will be prompted to name the migration. A descriptive name like `init` or `dev-setup` is usually fine for the first one.

3.  **Optional: Open Prisma Studio:**

    Prisma Studio is a GUI that allows you to view and edit data in your local database. It's incredibly useful for debugging and testing.

    ```bash
    pnpm exec prisma studio
    ```

    This will open a new browser tab, typically at `http://localhost:5555`.


This command typically runs the Next.js development server. Your application will usually be accessible at `http://localhost:3000`.

*   If the project uses a custom Node.js server (e.g., `server.ts` or `server.js`) to handle custom routing, middleware, or integrations *before* Next.js, `pnpm dev` will likely start this custom server, which in turn will proxy requests to the Next.js app.

In either case, `pnpm dev` is the command you want to run.

## 8. Stripe Webhook Forwarding

To test Stripe integrations (e.g., subscription lifecycle, payments), you need to forward webhooks from Stripe to your local development server.

1.  **Get your Stripe Webhook Secret:**
    You'll need a webhook signing secret to verify the authenticity of incoming webhooks. You can usually find this in your Stripe Dashboard under "Developers > Webhooks" or generate a new one for local testing. It typically starts with `whsec_`.

2.  **Start the Stripe CLI listener:**

    Open a **new terminal tab/window** and run:

    ```bash
    stripe listen --forward-to localhost:3000/api/commerce/webhook --events customer.subscription.created,invoice.payment_succeeded,checkout.session.completed
    ```

    *   Replace `localhost:3000/api/commerce/webhook` with the actual endpoint your application expects Stripe webhooks.
    *   The `--events` flag is optional but recommended to only forward the events you care about, reducing noise.

    The Stripe CLI will output a webhook secret (e.g., `whsec_...`). **Copy this secret and add it to your `.env.local`** file as `STRIPE_WEBHOOK_SECRET` (or whatever your project names it). You'll need to restart your `pnpm dev` server for this new environment variable to take effect.

## 9. Test Flows

Let's verify that the core functionalities of the platform are working locally.

1.  **Access the Admin Dashboard:**
    Open your browser and navigate to `http://localhost:3000/admin`.
    *   You should see a login page or be redirected to an authentication flow.
    *   After logging in (if required), you should see the admin dashboard. This confirms your main application and authentication are working.

2.  **Create a Plan (via Admin UI):**
    *   In the `/admin` dashboard, navigate to the section for managing "Plans" or "Products" (the exact path may vary based on your project's implementation, e.g., `/admin/plans` or `/admin/products`).
    *   Create a new plan (e.g., "Pro Plan") with a specific price and billing interval. This often involves associating it with a Stripe Product/Price ID, which you might need to create in your Stripe dashboard first (in test mode).
    *   Ensure the plan is marked as "active" or "published."

3.  **Simulate a Subscription (via Stripe CLI):**
    This simulates a customer successfully subscribing to your newly created plan.

    1.  **Trigger a checkout session completion:**
        First, you need a `checkout.session.completed` event to simulate a new customer.
        ```bash
        stripe trigger checkout.session.completed
        ```
        This will create a dummy `Checkout Session` and send a `checkout.session.completed` webhook to your local server (if your `stripe listen` is running). Your application should process this, potentially creating a new customer in your database.

    2.  **Trigger a subscription creation:**
        Next, simulate a subscription being created for that customer. You'll need the `customer_id` from the previous step's output or from your local database.
        ```bash
        stripe trigger customer.subscription.created --add customer:<CUSTOMER_ID> --override subscription:plan=<YOUR_STRIPE_PRICE_ID>
        ```
        *   Replace `<CUSTOMER_ID>` with the ID of the customer created in the previous step.
        *   Replace `<YOUR_STRIPE_PRICE_ID>` with the ID of the Stripe Price object associated with the "Pro Plan" you created earlier. You can find this in your Stripe dashboard.

        Your local server should receive the `customer.subscription.created` webhook, and your application should update the customer's subscription status in your local database.

    3.  **Verify:**
        *   Check your local database using `pnpm exec prisma studio`. You should see the new customer and their active subscription.
        *   Check the `/admin` dashboard. The customer's subscription should be reflected there.

## 10. Multi-Tenant Local Testing

MBT projects often require testing different tenants. Here are two common ways to do this locally.

### 10.1 Method 1: Modifying `/etc/hosts`

This method allows you to access different tenants using distinct subdomains on `localhost`.

1.  **Open `/etc/hosts`:**
    Open your `hosts` file with elevated privileges:

    ```bash
    sudo nano /etc/hosts
    ```

    You'll be prompted for your macOS password.

2.  **Add tenant entries:**
    Add lines like these at the end of the file:

    ```
    127.0.0.1   tenant1.localhost
    127.0.0.1   tenant2.localhost
    127.0.0.1   dashboard.localhost # For your main app/dashboard
    ```

    Save the file (Ctrl+O, Enter) and exit (Ctrl+X).

3.  **Access tenants:**
    Now you can access your application for different tenants by navigating to:
    *   `http://tenant1.localhost:3000`
    *   `http://tenant2.localhost:3000`
    *   `http://dashboard.localhost:3000` (for your main app/dashboard if it has a specific hostname)

    Your application's multi-tenancy logic (e.g., reading the hostname to determine the tenant) should then route requests accordingly.

### 10.2 Method 2: Dev-Route Hostname Override Middleware

Some projects implement a custom development middleware that allows you to specify the tenant via a URL query parameter, avoiding `/etc/hosts` modifications.

If your project supports this, you might access tenants like:

*   `http://localhost:3000?__tenant=tenant1`
*   `http://localhost:3000?__tenant=tenant2`

Check your project's documentation or codebase (e.g., in a `middleware.ts` or custom server file) to see if this feature exists. If it does, this is often the most convenient method for local testing.

## 11. Troubleshooting

Here are some common issues you might encounter and their solutions.

### 11.1 Sanity Environment Variable Crash (Multiline JSON)

**Problem:** Your Sanity Studio crashes on startup, often with an error related to environment variables, especially if you have a multiline JSON string in your `.env.local` file (e.g., a Firebase private key).

**Cause:** Shells often don't handle multiline environment variables well when passed directly.

**Solution:**
Ensure any multiline JSON environment variables are either:
1.  **Converted to a single line:** Remove all newlines and escape quotes within the JSON string.
    ```env
    FIREBASE_PRIVATE_KEY="{\"type\": \"service_account\", \"project_id\": \"my-project\", ...}"
    ```
2.  **Read from a file:** If your application supports it, store the JSON in a separate file (e.g., `firebase-key.json`) and configure your app to read from that file path.
3.  **Removed from `.env.local` for Sanity:** If Sanity doesn't strictly need that specific variable, remove it from the `.env.local` file that Sanity reads.

### 11.2 Postgres Connection Refused

**Problem:** Your application throws an error like `FATAL: password authentication failed for user "postgres"` or `connect ECONNREFUSED 127.0.0.1:5432`.

**Causes & Solutions:**

*   **Docker container not running:**
    *   **Solution:** Ensure Docker Desktop is running and your Postgres container is active.
        ```bash
        docker ps
        ```
        If not running, start it:
        ```bash
        docker compose up -d postgres
        ```
*   **Incorrect `DATABASE_URL`:**
    *   **Solution:** Double-check your `.env.local` file. The `DATABASE_URL` must exactly match the user, password, host, port, and database name defined in your `docker-compose.yml`.
        ```
        DATABASE_URL="postgresql://postgres:mysecretpassword@localhost:5432/mbt_dev"
        ```
*   **Firewall blocking port 5432:**
    *   **Solution:** Temporarily disable your firewall or ensure it allows connections to `localhost:5432`.
*   **Another process using port 5432:**
    *   **Solution:** Check if another Postgres instance or application is using port 5432.
        ```bash
        sudo lsof -i :5432
        ```
        If a process is listed, you can kill it with `kill -9 <PID>` (replace `<PID>` with the process ID).
