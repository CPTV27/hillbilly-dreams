# How to Add a New Event Handler to the Coordination Layer Event Bus

The Coordination Layer is the backbone of our system's asynchronous communication and side-effect management. It provides a robust, observable, and resilient mechanism for different modules to interact without tight coupling. This guide will walk you through the process of adding a new event handler, from defining the event to deploying and monitoring it.

## Core Principles

Before diving into the steps, it's crucial to understand two fundamental principles that underpin all event handler development within the Coordination Layer:

1.  **Idempotency:** Every handler **must** be idempotent. This means that calling the handler multiple times with the same event payload should produce the same result and have no additional side effects beyond the first successful execution. The event bus guarantees "at least once" delivery, making idempotency a non-negotiable contract for reliable processing.
2.  **Robust Error Handling:** Distinguish clearly between transient (retryable) and fatal (dead-letter) errors. The event bus relies on your handler to throw specific error types to determine its retry strategy.

## 1. Handler vs. Direct Call: When to Use What

One of the first decisions you'll make is whether your logic warrants an asynchronous event handler or a synchronous, direct function call.

**Heuristic:**

*   **Cross-Module Side Effects / Asynchronous Work = Event Handler:**
    *   If the operation involves updating state in a different module, sending notifications, integrating with external services, or any work that doesn't need to block the immediate request/response cycle, an event handler is the correct choice.
    *   It decouples modules, improves responsiveness, and provides built-in retry and observability.
    *   *Example:* A user signs up, and you need to: create a billing account (different module), send a welcome email (notification service), and provision resources (external API).
*   **Same-Module Synchronous Work = Direct Call:**
    *   If the operation is tightly coupled to the current module, completes quickly, and its success is required for the immediate continuation of the current request/transaction, a direct function call is appropriate.
    *   *Example:* A user updates their profile, and you need to update their record in the `User` database table within the same `User` module.

**Code Examples:**

```typescript
// --- DIRECT CALL EXAMPLE ---
// packages/modules/user/src/services/UserService.ts

import { UserRepository } from '../repositories/UserRepository';

export class UserService {
  constructor(private userRepository: UserRepository) {}

  async updateUserName(userId: string, newName: string): Promise<void> {
    // This is synchronous, tightly coupled to the User module,
    // and its success is critical for the immediate API response.
    await this.userRepository.update(userId, { name: newName });
    console.log(`User ${userId} name updated to ${newName}`);
  }
}

// In an API controller:
// const userService = new UserService(new UserRepository());
// await userService.updateUserName(req.userId, req.body.name);
// res.status(200).send("User name updated");
```

```typescript
// --- EVENT HANDLER EXAMPLE ---
// packages/modules/social/src/services/SocialService.ts

import { eventBus } from '@coordination-layer/events';
import { UserSignedUpEvent } from '@coordination-layer/events/types'; // Assuming this event exists

export class SocialService {
  async handleUserSignup(userId: string, username: string): Promise<void> {
    // This action (e.g., publishing a directory entry) is a cross-module side effect.
    // It doesn't need to block the user signup API response.
    // It can fail and be retried independently.
    await eventBus.publish<UserSignedUpEvent>({
      type: 'user.signedUp',
      payload: { userId, username },
      tenantId: 'some-tenant-id', // Crucial for multi-tenancy
      correlationId: 'some-request-id', // For tracing
    });
    console.log(`Published UserSignedUpEvent for user ${userId}`);
  }
}

// In a User module's signup service:
// const socialService = new SocialService();
// await socialService.handleUserSignup(newUser.id, newUser.username);
// The signup API can return success immediately, even if social processing takes time.
```

## 2. Add the Event Type to `packages/modules/events/src/types.ts`

All events flowing through the Coordination Layer event bus must have a defined type. This ensures strict type safety across modules and provides a central registry of all system events.

Open `packages/modules/events/src/types.ts` and add your new event interface.

```typescript
// packages/modules/events/src/types.ts

// ... other event types

/**
 * @event social.publishDirectoryEntry
 * @description Fired when a new user signs up and their profile should be
 *              published to the public directory.
 */
export interface SocialPublishDirectoryEntryEvent {
  type: 'social.publishDirectoryEntry';
  payload: {
    userId: string;
    username: string;
    profileUrl: string;
    // Add any other necessary data for the handler
  };
  tenantId: string; // Always include tenantId for multi-tenancy
  correlationId?: string; // Optional, but highly recommended for tracing
}

// Union type of all possible events
export type CoordinationEvent =
  // ... existing events
  | SocialPublishDirectoryEntryEvent;

// Export specific event types for easier import
export {
  // ... existing event exports
  SocialPublishDirectoryEntryEvent,
};
```

**Key Considerations:**

*   **`type` field:** This string literal is crucial. It uniquely identifies the event and follows the `module.action` naming convention (e.g., `social.publishDirectoryEntry`).
*   **`payload`:** This object contains all the data the handler needs to perform its work. Keep it minimal but complete. Avoid fetching additional data within the handler if it can be included here.
*   **`tenantId`:** **Absolutely critical for multi-tenancy.** Every event *must* include the `tenantId` of the tenant for which the event is being processed. This ensures tenant isolation.
*   **`correlationId`:** Highly recommended for end-to-end tracing across distributed systems. Pass this from the initial request context.
*   **JSDoc:** Add a `description` and `@event` tag for documentation and discoverability.

## 3. Define the Handler Function in Your Module

Now, create the actual function that will process your event. This function resides within the module responsible for handling the event.

Create a new file, typically `packages/modules/<your-module>/src/handlers.ts`, and define your handler.

```typescript
// packages/modules/social/src/handlers.ts

import {
  HandlerContext,
  TransientError, // For retryable errors
  FatalError, // For non-retryable errors
} from '@coordination-layer/events';
import { SocialPublishDirectoryEntryEvent } from '@coordination-layer/events/types';
import { DirectoryService } from './services/DirectoryService'; // Assuming a service exists

/**
 * Handles the 'social.publishDirectoryEntry' event to add/update a user's
 * entry in the public directory.
 *
 * @param event The SocialPublishDirectoryEntryEvent payload.
 * @param context The handler execution context providing utilities.
 */
export async function socialPublishDirectoryEntryHandler(
  event: SocialPublishDirectoryEntryEvent,
  context: HandlerContext
): Promise<void> {
  const { logger, services } = context; // Access logger and other services
  const { userId, username, profileUrl, tenantId } = event.payload;

  logger.info(
    `Processing social.publishDirectoryEntry for user ${userId} in tenant ${tenantId}`,
    { eventType: event.type, userId, tenantId, correlationId: event.correlationId }
  );

  const directoryService = services.get(DirectoryService); // Get service instance from context

  try {
    // --- Idempotency Check ---
    // This is a crucial step for idempotency.
    // Before performing any side effects, check if the action has already been completed.
    // Example: Check if the entry already exists and is up-to-date.
    const existingEntry = await directoryService.getDirectoryEntry(userId, tenantId);

    if (existingEntry && existingEntry.username === username && existingEntry.profileUrl === profileUrl) {
      logger.info(`Directory entry for user ${userId} already up-to-date. Skipping.`, { userId, tenantId });
      return; // Already processed, so we exit gracefully.
    }

    // --- Perform the core logic ---
    await directoryService.publishEntry({ userId, username, profileUrl, tenantId });

    logger.info(`Successfully published directory entry for user ${userId}.`, { userId, tenantId });

  } catch (error: any) {
    logger.error(`Error publishing directory entry for user ${userId}: ${error.message}`, {
      userId, tenantId, error: error.stack, correlationId: event.correlationId
    });

    // --- Error Handling ---
    // Distinguish between transient (retryable) and fatal (dead-letter) errors.
    if (error.message.includes('Network Error') || error.message.includes('Timeout')) {
      // Example: External service is temporarily unavailable.
      throw new TransientError('External directory service temporarily unavailable', error);
    } else if (error.message.includes('Invalid Profile Data')) {
      // Example: The event payload contained fundamentally bad data that won't succeed on retry.
      throw new FatalError('Invalid profile data provided, cannot publish entry', error);
    } else {
      // Default to transient if unsure, allowing for retries.
      throw new TransientError('Unexpected error during directory entry publication', error);
    }
  }
}
```

**Key Elements:**

*   **Handler Signature:** `async function handlerName(event: EventType, context: HandlerContext): Promise<void>`
*   **`HandlerContext`:** This object provides essential utilities:
    *   `context.logger`: A pre-configured logger with event-specific metadata. **Always use this logger.**
    *   `context.services`: A dependency injection container to retrieve service instances. This ensures testability and proper lifecycle management.
    *   `context.tenantId`: The `tenantId` from the event, useful for ensuring tenant isolation in service calls.
*   **Idempotency Implementation:** The example shows checking for an existing state before performing the action. This is the most common pattern.
*   **Error Handling:**
    *   **`TransientError`**: Throw this when an error is temporary (e.g., network issues, temporary service unavailability). The event bus will retry the handler according to its configuration.
    *   **`FatalError`**: Throw this when an error indicates a permanent issue (e.g., malformed data, business logic validation failure that won't pass on retry). The event will be moved to a Dead-Letter Queue (DLQ) immediately without retries.
    *   Always wrap the original error for better debugging.
*   **Logging:** Use `context.logger` with relevant event and tenant IDs for effective debugging and monitoring.
*   **Naming Convention:** Follow `moduleActionHandler` (e.g., `socialPublishDirectoryEntryHandler`).

## 4. Register on App Boot

For the event bus to know about your handler, you must register it during the application's initialization phase. This typically happens in `apps/web/instrumentation.ts` or a dedicated initialization file if your module has complex setup.

```typescript

import { socialPublishDirectoryEntryHandler } from 'packages/modules/social/src/handlers';
import { DirectoryService } from 'packages/modules/social/src/services/DirectoryService'; // For DI
import { SocialPublishDirectoryEntryEvent } from 'packages/modules/events/src/types';

// ... other imports and setup

export async function registerCoordinationHandlers() {
  console.log('Registering Coordination Layer event handlers...');

  // Register your handler
  eventBus.registerHandler<SocialPublishDirectoryEntryEvent>(
    'social.publishDirectoryEntry', // The exact event type string
    socialPublishDirectoryEntryHandler,
    {
      // Configuration options for this handler

      retries: 5,         // See step 6
      timeoutMs: 30000,   // See step 6
      // You can also specify specific services needed for DI:

    }
  );
```

## 5. Set `tenantScope` Appropriately

The `tenantScope` setting is crucial for multi-tenant applications. It dictates which events a handler is allowed to process based on the event's `tenantId` and the handler's configuration.

*   **`'own'` (Default):** The handler will only process events where the event's `tenantId` matches the `tenantId` configured for the *handler instance itself*. This is the most common and safest setting for handlers that operate within a single tenant's context.
    *   *Example:* `socialPublishDirectoryEntryHandler` should only update the directory for the tenant that the event originated from.
*   **`'all'`:** The handler will process events for *any* `tenantId`. This is reserved for system-level handlers that manage cross-tenant concerns, like global metrics, license management, or cross-tenant data synchronization. Use with extreme caution and ensure your handler logic explicitly handles tenant isolation when operating on multiple tenants.
    *   *Example:* A `system.auditLogEvent` handler might log events for all tenants to a central audit system.
*   **`['tenant-id-1', 'tenant-id-2']` (Allowlist):** The handler will only process events where the `tenantId` is one of the explicitly listed IDs. Useful for specialized handlers or testing environments.

**Configuration Example:**

```typescript
eventBus.registerHandler<SocialPublishDirectoryEntryEvent>(
  'social.publishDirectoryEntry',
  socialPublishDirectoryEntryHandler,
  {
    tenantScope: 'own', // Most common and safest default
    // tenantScope: 'all', // Use with extreme caution
    // tenantScope: ['tenant_prod_1', 'tenant_dev_2'], // For specific scenarios
    retries: 5,
    timeoutMs: 30000,
    dependencies: [DirectoryService],
  }
);
```

## 6. Set `retries` + `timeoutMs` Based on Operation Cost

These settings control the resilience and behavior of your handler in the face of transient failures.

*   **`retries`:**
    *   **Purpose:** The maximum number of times the event bus will attempt to re-execute the handler if it throws a `TransientError`.
    *   **Guidance:**
        *   For quick, idempotent operations with external dependencies that might have brief outages (e.g., API calls, database writes), 3-5 retries are often sufficient.
        *   For operations involving longer-running external processes or less reliable services, you might go higher (e.g., 10-15).
        *   Too many retries can clog the queue; too few might lead to lost events.
        *   Consider the exponential backoff strategy that the event bus typically employs.
    *   *Default:* If not specified, a sensible default (e.g., 3-5) will be used by the event bus.
*   **`timeoutMs`:**
    *   **Purpose:** The maximum amount of time (in milliseconds) the event bus will wait for your handler function to complete before considering it timed out and triggering a retry (if retries are available).
    *   **Guidance:**
        *   Set this value based on the *expected* maximum duration of your handler's operation, plus a reasonable buffer.
        *   If your handler makes external API calls, ensure `timeoutMs` is greater than the sum of all individual API call timeouts + processing time.
        *   Too short a timeout leads to unnecessary retries. Too long delays detection of hung processes.
    *   *Default:* If not specified, a default (e.g., 30,000ms = 30 seconds) might be applied.

**Configuration Example:**

```typescript
eventBus.registerHandler<SocialPublishDirectoryEntryEvent>(
  'social.publishDirectoryEntry',
  socialPublishDirectoryEntryHandler,
  {
    tenantScope: 'own',
    retries: 5,         // Allow up to 5 retries for transient issues
    timeoutMs: 30000,   // Give the handler 30 seconds to complete
    dependencies: [DirectoryService],
  }
);
```

## 9. Production Rollout

When deploying your new handler to production, follow your standard deployment process.

*   **Handler Enabled by Default:** Once deployed, the handler will be active and listening for events.
*   **Database Toggle Mechanism:** A key feature of the Coordination Layer is the ability to enable or disable handlers dynamically *without requiring a new code deployment*.
    *   There's typically a `CoordinationHandler` table in the database that stores the configuration for each registered handler (including its `enabled` status).
    *   To disable a handler (e.g., if it's causing issues), an administrator can update its `enabled` flag to `false` in the database. The event bus will pick up this change and stop processing new events for that handler.
    *   To re-enable it, set `enabled` back to `true`.
    *   This provides a critical safety mechanism for rapid incident response.

## Additional Considerations

### Idempotency Contract

Reiterating this critical point: **every handler must be idempotent.** The event bus guarantees "at least once" delivery, meaning your handler might be invoked multiple times for the same event due to transient failures, network issues, or infrastructure restarts.

**Strategies for Idempotency:**

1.  **Unique Identifiers (Event ID / Transaction ID):**
    *   Use a unique identifier from the event payload (e.g., `userId`, `orderId`, or a generated `eventId` if the event represents a unique transaction) as a key to check for prior processing.
    *   *Example:* If processing an `order.paid` event, check if `orderId` has already been marked as `paid`.
2.  **State Checks:**
    *   Before performing an action, check the current state of the system. If the desired state has already been achieved, simply return.
    *   *Example:* If a `social.publishDirectoryEntry` event is received, check if the directory entry for that `userId` already exists and is up-to-date. If so, do nothing.
3.  **Transactional Writes with Unique Constraints:**
    *   When writing to a database, leverage unique constraints. If a record with a specific unique key already exists, the database will prevent a duplicate insertion, making the operation idempotent. Handle the `duplicate key` error gracefully.
4.  **Conditional Updates:**
    *   Update records only if certain conditions are met, ensuring you don't overwrite newer data with older, reprocessed event data.
    *   *Example:* `UPDATE users SET last_login = :new_time WHERE last_login < :new_time`.

**Bad Example (Non-Idempotent):**

```typescript
// NOT IDEMPOTENT: This would send multiple emails if retried
async function sendWelcomeEmailHandler(event: UserSignedUpEvent, context: HandlerContext) {
  await emailService.sendEmail(event.payload.email, 'Welcome!');
}
```

**Good Example (Idempotent):**

```typescript
// IDEMPOTENT: Checks if email was already sent
async function sendWelcomeEmailHandler(event: UserSignedUpEvent, context: HandlerContext) {
  const emailSentRecord = await emailTrackingRepo.getByUserIdAndEventType(event.payload.userId, event.type);
  if (emailSentRecord) {
    context.logger.info(`Welcome email already sent to user ${event.payload.userId}. Skipping.`);
    return;
  }
  await emailService.sendEmail(event.payload.email, 'Welcome!');
  await emailTrackingRepo.create({ userId: event.payload.userId, eventType: event.type, timestamp: new Date() });
  context.logger.info(`Welcome email sent to user ${event.payload.userId}.`);
}
```

### Error Handling: Transient vs. Fatal

The distinction between transient and fatal errors is critical for the event bus's retry mechanism.

*   **`TransientError` (Retryable):**
    *   Indicates a temporary problem that might resolve itself upon retry.
    *   Examples: Network timeouts, database connection issues, external API rate limits, temporary service unavailability.
    *   When thrown, the event bus will mark the event as `FAILED` and schedule it for a retry after a backoff period, up to the configured `retries` limit.
    *   **Always wrap the original error:** `throw new TransientError('Service unavailable', originalError);`

*   **`FatalError` (Dead-Letter Immediately):**
    *   Indicates a permanent problem that will not be resolved by retrying.
    *   Examples: Invalid event payload (e.g., missing required fields, data type mismatch), business logic validation failure, unrecoverable configuration error.
    *   When thrown, the event bus will immediately move the event to the Dead-Letter Queue (DLQ) without further retries. This prevents endlessly retrying an event that's guaranteed to fail and frees up processing capacity.
    *   **Always wrap the original error:** `throw new FatalError('Invalid payload structure', originalError);`

**Custom Error Classes:**

It's good practice to define custom error classes for `TransientError` and `FatalError` (or use provided ones from `@coordination-layer/events`) to make the intent explicit and allow the event bus to reliably differentiate them.

```typescript
// packages/modules/events/src/errors.ts (example definitions)

export class CoordinationError extends Error {
  constructor(message: string, public originalError?: Error) {
    super(message);
    this.name = this.constructor.name;
    Error.captureStackTrace(this, this.constructor);
  }
}

export class TransientError extends CoordinationError {}
export class FatalError extends CoordinationError {}
```
