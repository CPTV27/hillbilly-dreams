## Purpose

The `social` module serves as the central nervous system for all social media publishing and interaction within the MBT ecosystem. As a critical component of Phase C Block 4, it works in close conjunction with the `broadcast` module, providing specialized functionalities for public-facing social channels. Its primary purpose is to enable seamless, controlled, and brand-compliant cross-platform social media publishing, scheduling, and performance monitoring for multiple tenants and their respective brands. This module abstracts away the complexities of various social media APIs, offering a unified interface while rigorously enforcing brand voice and preventing cross-contamination between distinct brand identities. It is designed to handle both direct API integrations and specialized workflows involving third-party services like Postiz for platforms with restrictive or device-dependent publishing requirements.

## Features

This module is designed to encapsulate and manage the following core capabilities:

*   **Cross-Platform Social Publishing**: Facilitates publishing content to major social media platforms including Instagram, TikTok, Facebook, and X (formerly Twitter). It handles platform-specific requirements and API nuances.
*   **Per-Brand Voice Routing & Contamination Guards**: Implements a robust system to ensure that content published to a specific social account adheres strictly to its designated brand voice profile. It actively prevents "cross-brand contamination" by hard-failing any publishing attempt that violates these rules.
*   **Postiz Bridge (Mac mini)**: Integrates with the Postiz service, leveraging a dedicated Mac mini setup, to handle publishing scenarios for platforms or content types that require specific device-based interactions or advanced media processing not available via standard cloud APIs (e.g., certain Instagram or TikTok video workflows).
*   **Platform-Native Fallbacks**: Where direct API publishing is limited or fails, the module intelligently leverages platform-native features or alternative pathways (e.g., Postiz) to ensure content delivery where possible, maintaining robust operation.
*   **Scheduling**: Provides comprehensive scheduling capabilities, allowing content to be queued for future publication at precise dates and times across selected platforms.
*   **Performance Reporting**: Gathers and aggregates key performance metrics (e.g., reach, engagement, impressions) from published posts across integrated social platforms, providing insights for analysis and optimization.
*   **Multi-Tenant Support**: Built from the ground up to support multiple independent tenants and their respective brands, ensuring strict data isolation and operational segregation.

## Public API

The `social` module exposes a set of well-defined functions to interact with its core capabilities. All API calls require `tenantId` and `brandId` for multi-tenancy and brand isolation.

### `schedulePost(tenantId, brandId, content, platforms, publishAt, voiceProfileId)`

Schedules a social media post for future publication across specified platforms. The content will be queued and published automatically at the designated time, subject to voice routing and platform-specific validations.

*   **`tenantId` (string, required)**: Unique identifier for the tenant making the request.
*   **`brandId` (string, required)**: Unique identifier for the brand within the `tenantId` context.
*   **`content` (object, required)**: An object representing the content to be published. This typically includes:
    *   `text` (string): The caption or main text for the post.
    *   `mediaAssets` (array of objects): An array of media asset references (e.g., URLs, IDs from a Media Asset Management system). Each object might specify `type` (`image`, `video`) and `url` or `assetId`.
    *   `hashtags` (array of strings, optional): Recommended hashtags to include.
    *   `mentions` (array of strings, optional): User handles to mention.
    *   `metadata` (object, optional): Any additional platform-specific metadata (e.g., location for Instagram, video title for TikTok).
*   **`platforms` (array of strings, required)**: An array of target social media platforms (e.g., `['instagram', 'tiktok', 'facebook', 'x']`).
*   **`publishAt` (Date or string, required)**: The precise UTC timestamp when the post should be published.
*   **`voiceProfileId` (string, required)**: The identifier for the brand voice profile associated with this content. This is crucial for the voice routing mechanism.

*   **Returns**: `Promise<string>` – A promise that resolves with a unique `jobId` for the scheduled post. This `jobId` can be used to track the status of the publication.
*   **Throws**:
    *   `ValidationError`: If required parameters are missing or invalid.
    *   `VoiceMismatchError`: If the `voiceProfileId` does not match the configured voice for any target platform/brand account.
    *   `PlatformIntegrationError`: If there's an issue with the underlying social platform integration (e.g., invalid credentials, API rate limits).
    *   `PostizServiceError`: If the Postiz bridge is required but unavailable or fails.

### `publishImmediate(tenantId, brandId, content, platforms, voiceProfileId)`

Publishes a social media post immediately across the specified platforms. This function bypasses the scheduling queue and attempts to publish the content as quickly as possible.

*   **`tenantId` (string, required)**: Unique identifier for the tenant.
*   **`brandId` (string, required)**: Unique identifier for the brand.
*   **`content` (object, required)**: Same structure as `content` in `schedulePost`.
*   **`platforms` (array of strings, required)**: An array of target social media platforms.
*   **`voiceProfileId` (string, required)**: The identifier for the brand voice profile.

*   **Returns**: `Promise<string>` – A promise that resolves with a unique `postId` or `jobId` for the immediate publication attempt.
*   **Throws**: Same error types as `schedulePost`.

### `syncWithPostiz(tenantId, brandId, postizContentId)`

Initiates a synchronization process with the Postiz service. This function is typically used for content that has been prepared or processed by Postiz (e.g., complex video edits on a Mac mini) and now needs to be pushed to social platforms under the module's management. It bridges the Postiz workflow with the MBT social publishing pipeline.

*   **`tenantId` (string, required)**: Unique identifier for the tenant.
*   **`brandId` (string, required)**: Unique identifier for the brand.
*   **`postizContentId` (string, required)**: The unique identifier provided by the Postiz service for the content ready for publishing. This ID allows the module to retrieve the final content package from Postiz.

*   **Returns**: `Promise<string>` – A promise that resolves with a unique `jobId` for the social publishing process initiated by Postiz content.
*   **Throws**:
    *   `ValidationError`: If `postizContentId` is missing or invalid.
    *   `PostizServiceError`: If there's an issue communicating with the Postiz API or retrieving content.
    *   `PlatformIntegrationError`: If publishing to social platforms fails after retrieving content from Postiz.

### `getPerformance(tenantId, brandId, platform, startDate, endDate)`

Retrieves aggregated performance metrics for social media posts published by a specific brand on a given platform within a defined date range. This function pulls data from the respective social platform APIs.

*   **`tenantId` (string, required)**: Unique identifier for the tenant.
*   **`brandId` (string, required)**: Unique identifier for the brand.
*   **`platform` (string, required)**: The target social media platform (e.g., `'instagram'`, `'facebook'`).
*   **`startDate` (Date or string, required)**: The start date (inclusive) for the performance data collection.
*   **`endDate` (Date or string, required)**: The end date (inclusive) for the performance data collection.

*   **Returns**: `Promise<object>` – A promise that resolves with an object containing aggregated performance data, which may include metrics like:
    *   `totalPosts` (number)
    *   `totalReach` (number)
    *   `totalImpressions` (number)
    *   `totalEngagements` (number)
    *   `averageEngagementRate` (number)
    *   `topPosts` (array of objects, e.g., post ID, URL, key metrics)
*   **Throws**:
    *   `ValidationError`: If parameters are missing or invalid.
    *   `PlatformIntegrationError`: If there's an issue retrieving data from the social platform API.
    *   `DataUnavailableError`: If no performance data is found for the specified criteria.

### `republishClip(tenantId, brandId, originalPostId, newPlatforms, newPublishAt)`

Facilitates the republishing or repurposing of previously successful social media content (e.g., a video clip or image post) to new platforms or at a different scheduled time. This can be used to extend the life of evergreen content or target new audiences. The original content and its associated voice profile are re-validated.

*   **`tenantId` (string, required)**: Unique identifier for the tenant.
*   **`brandId` (string, required)**: Unique identifier for the brand.
*   **`originalPostId` (string, required)**: The ID of the previously published post to be re-published. This ID must refer to content managed by this module.
*   **`newPlatforms` (array of strings, required)**: An array of new target social media platforms.
*   **`newPublishAt` (Date or string, optional)**: The new precise UTC timestamp for publication. If omitted, it defaults to immediate publication (similar to `publishImmediate`).

*   **Returns**: `Promise<string>` – A promise that resolves with a new `jobId` for the re-published content.
*   **Throws**:
    *   `ValidationError`: If parameters are missing or invalid, or `originalPostId` is not found.
    *   `VoiceMismatchError`: If the original post's voice profile is not compatible with the `brandId` and `newPlatforms`.
    *   `PlatformIntegrationError`: If there's an issue with integration or publishing to `newPlatforms`.

## Voice Routing & Cross-Brand Contamination Guards

A cornerstone feature of the `social` module is its robust voice routing and contamination guard system. Each brand within a tenant is associated with one or more predefined "voice profiles." These profiles encapsulate stylistic guidelines, tone, language nuances, and even specific content types permissible for that brand.

**Mechanism:**

1.  **Voice Profile Association**: When a social account (e.g., `instagram-account-for-brand-X`) is integrated into the MBT system, it is explicitly configured with a specific `voiceProfileId` (e.g., `corporate_professional`, `youthful_edgy`, `magazine_artsy`).
2.  **Publishing Request**: Every publishing request (via `schedulePost`, `publishImmediate`, `republishClip`) *must* include a `voiceProfileId` as part of its parameters. This `voiceProfileId` represents the intended voice of the content being published.
3.  **Dispatcher Validation**: Before any content is dispatched to a target social platform, the module's internal dispatcher performs a critical validation step:
    *   It checks if the `voiceProfileId` provided in the publishing request *matches* the `voiceProfileId` configured for the specific social account(s) targeted by the `brandId` and `platforms`.
    *   If a mismatch is detected (e.g., content tagged with `magazine_artsy` voice is attempted to be published to a social account configured for `records_gritty` voice), the dispatcher will **hard-fail** the request.
4.  **Hard-Fail Policy**: A hard-fail means the publishing operation is immediately aborted, and a `VoiceMismatchError` is thrown. This prevents any content from being published to an unintended audience or under an incorrect brand identity.

**Example**:
If a user attempts to publish a post with `voiceProfileId: 'magazine_artsy'` to an Instagram account associated with `brandId: 'mbt-records'` which is configured to only accept `voiceProfileId: 'records_gritty'`, the `social` module will reject the request outright. This mechanism is crucial for maintaining brand integrity and preventing embarrassing or damaging "cross-brand contamination" where, for instance, a refined "Magazine" voice accidentally appears on a raw "Records" social feed.

This system relies on accurate initial configuration of social accounts and their associated voice profiles within the MBT platform.

## Postiz Bridge

The `social` module includes a dedicated bridge to the Postiz service, which is instrumental for handling complex social media publishing requirements, particularly for platforms like Instagram and TikTok that may have stringent API limitations or require specific media processing workflows.

**Role of Postiz:**
Postiz often acts as an intermediary, especially for content that requires:
*   **Device-based Uploads**: Some platform APIs (historically Instagram, certain TikTok features) prefer or require uploads from a physical device, which a dedicated Postiz Mac mini can simulate.
*   **Advanced Media Processing**: Postiz can perform intricate video encoding, watermarking, or graphic overlays that are difficult or inefficient to do purely server-side.
*   **Live/Interactive Content**: For certain live streaming or interactive features, Postiz might provide a more robust or compliant interface.
