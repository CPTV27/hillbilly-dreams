# MBT Coordination Layer Event Bus (`packages/modules/events/`) - Operations Runbook

This document serves as a comprehensive operations runbook for the MBT Coordination Layer Event Bus module. It complements the development-focused README by detailing operational concerns, common failure modes, recovery procedures, and best practices for managing the event bus in a production environment.

The event bus leverages `PgBoss` for reliable, transactional job processing using PostgreSQL. Understanding `PgBoss` concepts (jobs, states, retries, dead-lettering) is crucial for effective operation of this module.

## 2. Common Failures + Recovery

This section outlines typical failure scenarios encountered with the event bus and provides actionable recovery steps.

### 2.1 Handler Timeouts

Handlers are functions that process events. If a handler takes too long to execute, PgBoss will mark the job as timed out.

*   **Symptoms:**
    *   `PgBoss: Job timed out` errors in application logs.
    *   Jobs remaining in the `active` state in `pgboss.job` for longer than their configured `job_timeout`.
    *   Slow processing rates, increasing `active` job counts without corresponding `completed` jobs.
    *   High CPU/memory utilization on worker instances.
*   **Detection:**
    *   Monitoring: Alert on `pgboss.job` metrics for `active` jobs count exceeding a threshold, or a high rate of `timed out` jobs.
    *   Logs: Filter application logs for "Job timed out" messages.
    *   PgBoss Admin UI/API: Observe jobs stuck in `active` state.
*   **Recovery:**
    1.  **Identify the Slow Handler:** Examine logs for the specific event type (`job.name`) associated with timeouts. Correlate with recent deployments or external service dependencies.
    2.  **Immediate Action (Temporary):**
        *   **Increase `job_timeout`:** If the timeout is consistently hit due to temporary upstream slowness or a known increase in processing time, you can temporarily increase the `job_timeout` for the affected event type in the handler registration. This buys time but doesn't solve the root cause.
        *   **Scale Workers:** Increase the number of event worker instances to distribute the load, but this might exacerbate upstream issues if the slowness is due to a shared dependency.
    3.  **Root Cause Analysis & Fix:**
        *   **Optimize Handler Logic:** Identify inefficient database queries, complex calculations, or synchronous blocking calls.
        *   **External Service Calls:** If external API calls are involved, check their performance. Implement retries with backoff for transient errors, and consider circuit breakers for sustained outages. Ensure these calls are asynchronous where possible.
        *   **Resource Contention:** Check database performance, message queue performance (if the handler pushes to another queue), or other shared resources.
    4.  **Clear Stuck Jobs:**
        *   **Manually Fail/Retry:** For jobs that are stuck in `active` state indefinitely after a fix, you may need to manually update their state in `pgboss.job` to `failed` (if they should not be retried) or `retry` (to allow PgBoss to re-pick them up).
        *   **SQL (use with extreme caution, ensure you target specific job IDs):**
            ```sql
            -- To mark a timed-out job as failed (moves to dead-letter if retries exhausted)
            UPDATE pgboss.job
            SET state = 'failed', completed_on = NOW(), output = '{"error": "Manually failed after timeout investigation"}'
            WHERE id = 'your-job-id' AND state = 'active';

            -- To retry a specific timed-out job
            UPDATE pgboss.job
            SET state = 'retry', retrycount = retrycount + 1, started_on = NULL, output = NULL
            WHERE id = 'your-job-id' AND state = 'active';
            ```

### 2.2 Handler Exceptions

Unhandled errors within a handler function.

*   **Symptoms:**
    *   Jobs moving to `failed` state after exhausting `maxRetries`.
    *   Application logs showing stack traces and error messages from handler functions.
    *   Error monitoring systems (Sentry, Datadog, etc.) reporting exceptions.
*   **Detection:**
    *   Monitoring: Alert on `pgboss.job` metrics for `failed` jobs count.
    *   Logs: Filter application logs for "Error in handler", "Unhandled exception", or specific error messages.
    *   Error Monitoring: Dashboards showing recent exceptions.
    *   PgBoss Admin UI/API: Jobs in `failed` state, with `output` containing error details.
*   **Recovery:**
    1.  **Analyze Stack Trace:** Identify the exact line of code and the type of error.
    2.  **Distinguish Transient vs. Permanent:**
        *   **Transient Errors:** (e.g., network issues, temporary database unavailability, external API rate limits). These should ideally be handled by retries with backoff. If `maxRetries` is too low for a frequently transient issue, increase it.
        *   **Permanent Errors:** (e.g., invalid input data, logic bugs, schema mismatches). These will always fail and require a code fix.
    3.  **Code Fix & Re-deployment:** For permanent errors, develop and deploy a fix.
    4.  **Replay Dead-Lettered Events:** After the fix is deployed, use the replay playbook (Section 3) to re-process the events that failed due to this bug.

### 2.3 PgBoss Schema Drift

The PgBoss schema in the database does not match the version expected by the deployed application code.

*   **Symptoms:**
    *   `PgBoss` failing to start, often with SQL errors related to missing tables/columns or invalid queries.
    *   Application logs indicating `schema_version` mismatch or migration failures.
    *   `PgBoss` operations (enqueue, dequeue) failing with database errors.
*   **Detection:**
    *   Application startup logs: `PgBoss` typically logs its schema version and any migration attempts. Look for "schema_version mismatch" or SQL errors immediately after application restart.
