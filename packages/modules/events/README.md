# `packages/modules/events` — MBT Coordination Layer Event Bus

## 1. Purpose

The `events` module serves as the **MBT Coordination Layer Event Bus**, providing a robust, multi-tenant, and observable mechanism for cross-module communication within our Next.js / Prisma monorepo. Its primary function is to enable modules to publish events when significant changes occur (e.g., a new directory entry is created), and for other modules to react to these events reliably (e.g., publish to social media, update a magazine outline, notify stakeholders). This decouples modules, improving maintainability and scalability by reducing direct dependencies.

We chose **PgBoss** as the foundation for this event bus due to its powerful features and alignment with our existing infrastructure. Alternative solutions were considered:
- **`LISTEN`/`NOTIFY` (Postgres):** While simple, it lacks crucial features like durability, at-least-once delivery guarantees, retries, dead-letter queues, and built-in multi-tenancy. Implementing these would require significant custom development, defeating the purpose of a ready-made solution. It's primarily for real-time, in-process notifications, not robust inter-service communication.
- **Inngest / Vercel Queues:** These are external, managed services. While they offer many desirable features, they introduce vendor lock-in, potential latency, and additional costs. More importantly, we aim to leverage our existing Postgres database, which is already a core component of our infrastructure, providing cost efficiency and direct control over our data and queueing mechanism. PgBoss allows us to keep our event queueing within our existing data plane, simplifying deployment and operational overhead while providing a feature set comparable to external queueing services.

PgBoss leverages our existing Postgres database for durable job storage, offering robust features such as at-least-once delivery, configurable retry policies with exponential backoff, dead-letter queues, concurrency control, and scheduling. This makes it an ideal choice for building a reliable and scalable event bus that is fully integrated into our existing data ecosystem, without introducing new external dependencies.

## 3. Data Model

The event bus integrates with our existing Postgres database, using a combination of dedicated Prisma models for audit and handler management, and PgBoss's internal tables for job queueing.

### Prisma Models

These models provide a high-level, human-readable view and management layer for the event bus, offering auditability and configuration for handlers.

-   **`Event`**
    -   `id`: `String` (`@id`, `@default(cuid())`) - Unique identifier for the published event.
    -   `name`: `String` - The name of the event (e.g., `directory.entry.created`).
    -   `tenantId`: `String` - The ID of the tenant that published the event.
    -   `payload`: `Json` - The full JSON payload of the event.
    -   `publishedAt`: `DateTime` (`@default(now())`) - Timestamp when the event was published.
    -   `jobId`: `String` (`@unique`) - The corresponding PgBoss job ID for this event.
    -   `status`: `Enum` (`PUBLISHED`, `FAILED`, `DLQ`) - Current status of the event in its lifecycle.
    -   `error`: `Json?` - Stores error details if the event processing failed and landed in DLQ.
    -   **Purpose:** Serves as an immutable audit log of all events published to the bus. This allows us to trace event flow, debug issues, and reconstruct state if necessary.

-   **`EventHandler`**
    -   `id`: `String` (`@id`, `@default(cuid())`) - Unique identifier for the handler registration.
    -   `name`: `String` (`@unique`) - A unique, human-readable name for the handler (e.g., `directory.entry.created.social-publisher`).
    -   `eventName`: `String` - The event name this handler subscribes to.
    -   `tenantScope`: `Enum` (`OWN`, `ALL`, `ALLOWLIST`) - Defines the multi-tenant isolation mode for this handler.
    -   `allowlistTenants`: `Json?` - An array of tenant IDs if `tenantScope` is `ALLOWLIST`.
    -   `isActive`: `Boolean` (`@default(true)`) - Whether this handler is currently active and should process events.
    -   `lastProcessedJobId`: `String?` - A cursor pointing to the last successfully processed job
